## 1. The platform in one paragraph

UBB is a multi-tenant usage-based-billing backend (Django + Celery + Postgres + Redis, Python SDK) aimed at AI products. A tenant meters its end-customers' usage as immutable events with integer-micros money and a single per-tenant currency (ubb-platform/apps/metering/usage/services/usage_service.py:139-147), prices each event at the ingest choke point through a versioned two-card rate-card engine (cost cards = COGS, price cards = what you charge, plus markup fallback; ubb-platform/apps/metering/pricing/services/pricing_service.py:83-183), and gets margin analytics per customer/dimension/business (ubb-platform/apps/subscriptions/api/margin_endpoints.py:21-237). Money sits in prepaid wallets with credit-grant lots, Stripe Checkout/auto top-ups, budgets, a pre-check spend gate, per-run hard stops, and hourly self-healing reconciliation (ubb-platform/apps/billing/handlers.py:13-100; ubb-platform/config/settings.py:147-254). Postpaid tenants instead get month-end usage invoices and seat-based subscriptions pushed to the tenant's own Stripe Connect account with AR tracking (ubb-platform/apps/billing/invoicing/services/postpaid_service.py:101-453; ubb-platform/apps/subscriptions/orchestration/service.py:147-206). End customers see balances, usage, and invoices through a JWT-authenticated widget API (ubb-platform/api/v1/me_endpoints.py:14-465). UBB itself bills tenants a platform fee on UBB's own Stripe account (ubb-platform/apps/billing/tenant_billing/services.py:13-199). Customers can be individuals, businesses, or seats; pooled businesses pay for their seats while control and attribution stay per-seat (ubb-platform/apps/platform/customers/models.py:46-52).

## 3. Capability catalog by journey

### 3.1 Cost attribution & pricing

Tenant API mounted at `/api/v1/metering/` (NinjaAPI, Bearer API key + `ProductAccess("metering")`; ubb-platform/config/urls.py:27); margin analytics at `/api/v1/margin/` (also gated on the metering product; ubb-platform/config/urls.py:32). SDK: `MeteringClient` (ubb-sdk/ubb/metering.py) + the orchestrating `UBBClient` facade (ubb-sdk/ubb/client.py).

#### 3.1.1 Record a single usage event

- **What:** Ingest one immutable usage event for a customer: raw `units` and/or named `usage_metrics`, optional caller-supplied provider cost and billed price, attribution tags, optional run association, optional backdated `effective_at`. The engine prices whatever the caller does not supply. Idempotent on (tenant, customer, idempotency_key) — replay returns the original event with zero new rows, and a replay wins even if the backfill window has since closed (ubb-platform/apps/metering/usage/services/usage_service.py:122-128).
- **How:** POST `/api/v1/metering/usage` (ubb-platform/api/v1/metering_endpoints.py:42-103) → `UsageService.record_usage` (ubb-platform/apps/metering/usage/services/usage_service.py:113-221). SDK: `MeteringClient.record_usage(customer_id, request_id, idempotency_key, *, provider_cost_micros, billed_cost_micros, units, provider, event_type, currency, tags, product_id, metadata, run_id, usage_metrics, recorded_at)` (ubb-sdk/ubb/metering.py:100-148; `recorded_at` serializes to wire field `effective_at`; naive datetimes raise `ValueError` client-side, ubb-sdk/ubb/metering.py:20-32). The `UBBClient.record_usage` facade exists but REQUIRES `provider_cost_micros` and omits `usage_metrics`/`recorded_at` (ubb-sdk/ubb/client.py:156-188).
- **Options:** `customer_id` (UUID, 404 if not tenant's); `request_id` 1–500 chars in the schema BUT the DB column is varchar(255) (ubb-platform/api/v1/schemas.py:27 vs ubb-platform/apps/metering/usage/models.py:17); `idempotency_key` 1–500; `provider_cost_micros`/`billed_cost_micros` each Optional int 0..999_999_999_999 (ubb-platform/api/v1/schemas.py:30-31); `units` Optional int ≥ 0; `usage_metrics` Optional dict[str,int], values ≥ 0 (ubb-platform/api/v1/schemas.py:32-44); `currency` Optional 3-char, must equal tenant `default_currency` case-insensitively else 422 (ubb-platform/apps/metering/usage/services/usage_service.py:139-147); `tags` dict[str,str]; `event_type`/`provider`/`product_id` max 100 chars; `metadata` free dict; `run_id` UUID; `effective_at` ISO datetime. Errors: 429 + `hard_stop: true` (run killed in a separate transaction), 409 `run_not_active`, 422 `pricing_error` / `effective_at_*` / `billing_period_closed` / validation (ubb-platform/api/v1/metering_endpoints.py:72-100). Response: `event_id, provider_cost_micros, billed_cost_micros, units, run_id, run_total_cost_micros, hard_stop, usage_metrics, pricing_provenance, uncosted_metrics, service_id, agent_id` (ubb-platform/api/v1/schemas.py:69-83; ubb-platform/apps/metering/usage/services/usage_service.py:97-110).
- **Anchors:** ubb-platform/api/v1/metering_endpoints.py:42-103; ubb-platform/apps/metering/usage/services/usage_service.py:113-221; ubb-platform/api/v1/schemas.py:25-53; ubb-sdk/ubb/metering.py:100-148

#### 3.1.2 Batch usage ingestion

- **What:** Ingest 1..100 INDEPENDENT events in one HTTP call. Each item runs as its own atomic commit (deliberately not one mega-transaction); always HTTP 200 with positionally aligned `results[]` plus `succeeded`/`failed` counts. A whole-batch network retry is safe (per-item idempotency returns original event ids); a duplicate idempotency_key WITHIN one batch resolves to the first item's event id. A hard-stop on one item kills the run and the batch continues (later items on that run get `run_not_active`); if the kill itself fails the run stays ACTIVE and a loud log is the only signal (ubb-platform/api/v1/metering_endpoints.py:160-166).
- **How:** POST `/api/v1/metering/usage/batch` (ubb-platform/api/v1/metering_endpoints.py:184-205; per-item mapping at 106-181). SDK: `MeteringClient.record_batch(events) -> BatchResult(results: list[BatchItemResult(ok, error, detail, event_id, data)], succeeded, failed)` (ubb-sdk/ubb/metering.py:150-184; ubb-sdk/ubb/types.py:35-50). Per-event dict takes the same keys as `record_usage` plus `customer_id`; per-event `recorded_at` supported.
- **Options:** `events` list min 1 max 100 (ubb-platform/api/v1/schemas.py:56-57; 0 or >100 → 422). Per-item error codes mirror the single endpoint byte-for-byte plus `ok: false`; 404s become `{error: 'not_found'}`, generic ValueError becomes `{error: 'validation_error'}` (ubb-platform/api/v1/metering_endpoints.py:106-181).
- **Anchors:** ubb-platform/api/v1/metering_endpoints.py:106-205; ubb-platform/api/v1/schemas.py:56-66; ubb-sdk/ubb/metering.py:150-184

#### 3.1.3 Caller timestamps & bounded backfill (`effective_at`)

- **What:** Caller may state when usage economically HAPPENED, distinct from arrival (`created_at`). Validated at the `record_usage` choke point with typed 422 codes: `effective_at_naive` (no tz), `effective_at_in_future` (>5 min skew; ubb-platform/apps/metering/usage/services/usage_service.py:19), `effective_at_too_old` (older than `tenant.backfill_window_days`, default 34, model-clamped 0–60; 0 = no backfill), `billing_period_closed` (the billing OWNER's postpaid invoice for the effective UTC month is frozen). Pricing uses `as_of=effective_at` — events are priced on the card versions valid at the EFFECTIVE time and advance the EFFECTIVE month's tier ladder (ubb-platform/apps/metering/usage/services/usage_service.py:160-167). A backfill into a PRIOR UTC month writes a `BackfillDirtyPeriod` marker in the same transaction; an hourly task re-snapshots that month's margin and acks the marker (ubb-platform/apps/metering/usage/models.py:75-100; ubb-platform/apps/metering/queries.py:432-466).
- **How:** `effective_at` field on POST `/usage` and per-item in `/usage/batch`; SDK kwarg `recorded_at` (tz-aware datetime or ISO string). Validation: `validate_effective_at` (ubb-platform/apps/metering/usage/services/usage_service.py:31-74); closed-period check via `apps.billing.queries.is_usage_period_closed` keyed on `Customer.resolve_billing_owner()`.
- **Options:** Omitted = server now. The tenant knob `backfill_window_days` lives on the Tenant model (ubb-platform/apps/platform/tenants/models.py:60-64) but is NOT settable via any public API endpoint (auditor grep of `api/` found no writer — DB/ops only).

#### 3.1.7 Pricing math — exactly four models

- **What:** EXACTLY these compute paths exist (ubb-platform/apps/metering/pricing/models.py): (1) **per_unit**: `(units*rate_per_unit_micros + unit_quantity//2)//unit_quantity + fixed_micros` — half-up rounding per event (`compute`, models.py:170-180). (2) **flat**: `fixed_micros` per event (models.py:177-178). (3) **graduated** (PRICE cards only): marginal cumulative tiered — cumulative `T(q)` sums, over each band the ladder has entered, the band's `flat_micros` (charged once when first entered) plus half-up-rounded `((upper-lower)*rate + uq//2)//uq`; each event is priced `T(prior+units) - T(prior)` so marginals telescope exactly to `T(total)` (`compute_cumulative`/`compute_marginal`, models.py:182-227); per-band event decomposition in `_event_bands` (ubb-platform/apps/metering/pricing/services/pricing_service.py:16-55). (4) **package** (PRICE cards only): `T(0)=0; T(q>0)=ceil(q/unit_quantity)*rate_per_unit_micros + fixed_micros` once per period (models.py:213-217). `compute()` RAISES on tiered models — tiered cards cannot be priced without period context (models.py:171-176), and tiered cost cards are rejected both at validation (`validate_tiers`, models.py:76-78) and defensively at price time (pricing_service.py:120-125). Tier state: `PricingPeriodCounter` — per (tenant, customer, lineage_id, UTC-calendar-month) unit ladder advanced under SELECT FOR UPDATE inside the event's transaction; keyed on lineage_id so version bumps keep mid-period marginal continuity and customer overrides/dimensional variants get separate ladders (ubb-platform/apps/metering/pricing/models.py:230-258; ubb-platform/apps/metering/pricing/services/tier_counter_service.py:34-66; metrics iterated `sorted()` for deadlock-free lock order, pricing_service.py:142-144).
- **How:** Engine version '2.1.0' (ubb-platform/apps/metering/pricing/services/pricing_service.py:9). Graduated tier shape: `{up_to: int>0|None (last only), rate_per_unit_micros: int>=0, unit_quantity: int>0 default 1_000_000, flat_micros: int>=0 default 0}`; strictly increasing `up_to`; max 20 tiers (`MAX_GRADUATED_TIERS`, models.py:51,59-122). package must have `tiers==[]` and reuses the card's scalar fields.
- **Options:** Free-tier emulation: graduated tier 1 with rate 0. Per-block pricing: per_unit with `unit_quantity` = block size (half-up) or package (ceil).
- **Anchors:** ubb-platform/apps/metering/pricing/models.py:39-227,230-258; ubb-platform/apps/metering/pricing/services/pricing_service.py:9,16-55,120-172; ubb-platform/apps/metering/pricing/services/tier_counter_service.py:16-66

#### 3.1.8 Card resolution: exact keys, dimensional matching, customer override

- **What:** For each `usage_metrics` key, `_resolve_card` filters active-at-`as_of` cards by EXACT equality on (tenant, card_type, provider, event_type, metric_name, currency) — empty provider on a card matches only events with empty provider; there is NO wildcard. Then dimensions: `card.dimensions` must subset-match event tags with string equality (`str(tags.get(k)) == str(v)`). Owner precedence: customer-specific cards first, then tenant-wide; within an owner, most dimensions wins, ties broken by newest `valid_from` (ubb-platform/apps/metering/pricing/services/pricing_service.py:60-80). #### 3.2.3 Auto top-up (saved card) including SCA

- **What:** When a usage drawdown (or a grant-expiry debit) drops the balance below a per-customer trigger threshold, the platform emits `billing.balance_low`; the Stripe connector creates a `TopUpAttempt` and charges the saved default payment method off-session. - **Options:** Attempt statuses: pending, succeeded, failed, expired, requires_action, superseded; triggers manual/auto_topup/widget. #### 3.2.4 Top-up receipts

- **What:** Every succeeded top-up gets a paid (out-of-band) Stripe receipt invoice on the tenant's connected account plus a local Invoice row; missing receipts self-heal hourly. End customers can list them in the widget.
- **How:** Webhook dispatches `ReceiptService.create_topup_receipt` on commit (`Invoice.create` → `InvoiceItem.create` → finalize → pay `paid_out_of_band`, all with `receipt-*-{attempt.id}` idempotency keys); the local Invoice is only written if ALL Stripe calls succeed. Beat `reconcile_missing_receipts` (hourly :30) retries succeeded attempts with no Invoice. Widget list: GET `/api/v1/me/invoices`.

#### 3.2.5 Credit grants (expiring/promo lots)

- **What:** The tenant layers expiring or promo credit lots on the wallet. Lots are consumed soonest-expiry-first (promo before paid on ties); promo is never withdrawable; due lots lazily expire before any money path reads the balance; expiring-soon warnings and expiry events are emitted; grants can be voided; usage refunds re-fund the exact lots that funded the charge; Stripe refunds/disputes claw lots back.
- **How:** POST `/api/v1/billing/customers/{customer_id}/grants {kind paid|promo, amount_micros, expires_at XOR expires_in_days (omit both = non-expiring), idempotency_key, description}` → GRANT txn keyed `grant:{idempotency_key}` + `CreditGrant` in one savepoint, on the resolved BILLING OWNER's wallet; if the balance was negative the credit first self-allocates as `overage_recoup`. GET `.../grants` lists (status filter, cursor). POST `.../grants/{grant_id}/void` debits remaining, clamped to never go below zero, keyed `grant_void:{grant_id}`. `GrantLedger.allocate` consumes lots in order `expires_at ASC NULLS LAST`, promo-before-paid, `created_at`; `GrantLedger.refund` re-funds usage allocations (capped by `refunded_micros`, granted-remaining, and a G1 overdraft budget; expired/voided lots land as base). Clawback cascades: void the source top-up lot first, then consume other lots until `sum(active remaining) <= max(balance, 0)`. Paid top-ups become expiring lots only if `CustomerBillingProfile.topup_grant_expiry_days` is set (NULL = never expires).
- **Options:** kinds: paid, promo; statuses: active, depleted, expired, voided; sources: checkout, auto_topup, api, other. Allocation types: usage, withdrawal, clawback, overage_recoup. Beat `expire_credit_grants` (hourly :10): pass 1 expires due lots exactly-once via `expiry:{grant_id}` keys (clamped, never negative), pass 2 one-shot `CreditGrantExpiring` warnings 7 days ahead (winning-update on `warning_sent_at`). Events: `billing.credit_grant_expiring`, `billing.credit_grant_expired`.
- **Anchors:** ubb-platform/apps/billing/wallets/models.py:89-204; ubb-platform/apps/billing/wallets/grants.py:27-371; ubb-platform/api/v1/billing_endpoints.py:407-560; ubb-platform/apps/billing/wallets/tasks.py:174-256; ubb-platform/api/v1/schemas.py:179-199

#### 3.2.6 Direct debit / credit endpoints

- **What:** The tenant moves arbitrary money: POST `/debit` subtracts (consumes grant lots in usage order, can drive the balance negative — no floor check, no overage/suspend events), POST `/credit` adds non-expiring base money (ADJUSTMENT, deliberately no grant lot).
- **How:** POST `/api/v1/billing/debit` and `/credit` with `{customer_id (EXTERNAL id), amount_micros, reference/source, optional idempotency_key}`; both lock the wallet, replay-safe when a key is supplied; debit writes type DEBIT and calls `GrantLedger.allocate`; credit writes ADJUSTMENT.
- **Options:** `amount_micros` gt 0 le 999_999_999_999 (no cent-alignment requirement, unlike top-ups/withdrawals/grants); `idempotency_key` optional (None = not replay-safe).
- **Anchors:** ubb-platform/api/v1/billing_endpoints.py:54-134; ubb-platform/api/v1/schemas.py:159-171

#### 3.2.7 Withdrawals

- **What:** The tenant withdraws base + paid money from a wallet; promo credit is excluded from availability; the actual payout is the TENANT's job — the platform only writes the WITHDRAWAL ledger row and emits `billing.withdrawal_requested`.
- **How:** POST `/api/v1/billing/customers/{customer_id}/withdraw {amount_micros gt 0 cent-aligned, idempotency_key required, description}`; under wallet lock: lazy-expires due lots, rejects 400 if `balance − active promo remaining < amount`, debits, allocates against non-promo lots (`allocation_type=withdrawal`), emits WithdrawalRequested.
- **Options:** Idempotent replay returns the original transaction; no Stripe payout integration on this path.
- **Anchors:** ubb-platform/api/v1/billing_endpoints.py:201-252; ubb-platform/apps/billing/wallets/grants.py:332-339 (promo_remaining)

#### 3.2.8 Usage refund (lot-aware)

- **What:** The tenant refunds a specific usage event's billed cost back to the wallet; the slices originally funded by still-live grant lots are restored into those lots (promo refunds stay promo, never withdrawable); expired/voided-lot shares and base shares land as base credit; a `refund.requested` event lets metering record the Refund.
- **How:** POST `/api/v1/billing/customers/{customer_id}/refund {usage_event_id, reason, idempotency_key}`; looks up cost via `get_usage_event_cost` (tenant-scoped, 404 if unknown); credits the full cost as a REFUND txn; finds the original USAGE_DEDUCTION by the pinned `WalletTransaction.usage_event_id` and calls `GrantLedger.refund` (`refunded_micros` caps prevent re-funding the same micro twice — a second refund of the same event under a new key lands entirely as base); emits RefundRequested.
- **Options:** Replay via `idempotency_key` returns the existing refund. Refund is whole-event-cost only (no partial amount parameter).
- **Anchors:** ubb-platform/api/v1/billing_endpoints.py:268-335; ubb-platform/apps/billing/wallets/grants.py:264-330

#### 3.2.9 Spend gate: POST /pre-check

- **What:** Pre-flight authorization before doing work for a customer: blocks on suspended/closed status (seat OR its pooled billing owner), per-seat fixed-window rate limit, wallet affordability vs the min-balance floor (skipped for postpaid tenants), and the per-seat budget cap. Optionally starts a Run snapshotting cost-ceiling limits.
- **How:** POST `/api/v1/billing/pre-check {customer_id, start_run, run_metadata, external_run_id}` → `RiskService.check`: (1) status gate → reasons `insufficient_funds` (suspended) / `account_closed`; (2) rate limit via Redis cache key `ratelimit:{customer.id}:rpm` vs `RiskConfig.max_requests_per_minute` — Redis failure silently SKIPS rate limiting (fail-open, hardcoded); (3) balance read from the billing OWNER's wallet (Postgres, not Redis), blocked when `balance < -min_balance` and `tenant.billing_mode != postpaid`; (4) `BudgetService.check` on the SEAT. `start_run=True` creates a Run with `cost_limit_micros` + `hard_stop_balance_micros` copied from `BillingTenantConfig` and returns `run_id` + limits. Requires the `billing` product.
- **Options:** Reasons returned: insufficient_funds, account_closed, rate_limit_exceeded, budget_exceeded, budget_unavailable. Budget fail mode on Redis/Postgres-read failure: allow (default) unless `BudgetConfig.fail_closed` or `RiskConfig.gate_fail_closed`. NOTE: `gate_fail_closed` only affects the budget branch; rate limiting and the balance read have no fail-closed option.
- **Anchors:** ubb-platform/api/v1/billing_endpoints.py:255-265; ubb-platform/apps/billing/gating/services/risk_service.py:8-75; ubb-platform/apps/billing/gating/services/budget_service.py:67-88; ubb-platform/apps/billing/gating/models.py:5-15

#### 3.2.10 Budgets (per-customer / tenant-default monthly caps)

- **What:** Calendar-month spend caps with threshold alert webhooks and optional hard enforcement at the gate. Customer-level config overrides a tenant-wide default. The live spend counter lives in Redis (rebuilt from the Postgres ledger hourly and on miss).
- **How:** PUT/GET `/api/v1/billing/budget` (tenant default) and PUT/GET `/api/v1/billing/customers/{customer_id}/budget {cap_micros>=0 (<=0 = no cap), enforce_mode, hard_stop_pct 1..1000, alert_levels list of pct, fail_closed}`; GET `.../budget/status` returns `{period YYYY-MM, spend_micros, cap_micros, pct, enforce_mode}`. The gate blocks when `enforce_mode==enforcing` and `spend >= cap*hard_stop_pct/100`. The post-drawdown hook `record_usage_spend` INCRBYs Redis key `budget:{customer_id}:{YYYY-MM}` (62-day TTL) and emits `budget.threshold_reached` once per (customer, period, level) — dedup via OutboxEvent existence check; backdated events whose effective month is a PRIOR month skip the live counter (hourly rebuild is the source of truth).
- **Options:** `enforce_mode`: advisory (default, alerts only) | enforcing; default `alert_levels` [50, 80, 100, 110]; `fail_closed` default False; period is fixed 'month' (the model field exists but no other value is wired). Budget basis is EFFECTIVE-month; documented bypass: an enforcing cap is evadable by backdating into the prior month, bounded by `Tenant.backfill_window_days` (0–60, default 34; 0 = airtight).
- **Anchors:** ubb-platform/apps/billing/gating/models.py:18-46; ubb-platform/api/v1/billing_endpoints.py:691-758; ubb-platform/apps/billing/gating/services/budget_service.py:13-153; ubb-platform/apps/billing/handlers.py:104-129; ubb-platform/apps/platform/tenants/models.py:64

#### 3.2.11 Drawdown, overage event, balance-floor suspension

- **What:** Usage recording NEVER blocks on funds; the wallet debit happens asynchronously in the outbox handler. Crossing zero emits a one-shot `billing.balance_overage` early warning; falling below the min-balance floor flips the billing owner to suspended and emits `billing.customer_suspended` (which the gate then enforces).
- **How:** Outbox handler `handle_usage_recorded_billing`: prepaid/meter_only tenants only, debit keyed `usage_deduction:{usage_event_id}` on the PINNED billing owner's wallet under lock, lazy grant expiry first; winning branch only: `GrantLedger.allocate`, BalanceOverage when old ≥ 0 and new < 0, suspend when `new_balance < -min_balance` and owner active, BalanceLow when below the auto-top-up trigger. `min_balance` resolution: `CustomerBillingProfile.min_balance_micros` (per-customer, ORM-only) → `BillingTenantConfig.min_balance_micros` (default 0). A dispute-loss deduction can also suspend (ubb-platform/apps/billing/connectors/stripe/webhooks.py:325-341), as can `invoice.payment_failed` (status-only).
- **Options:** `min_balance` semantics: allowed overdraft depth below zero (threshold negated at both gate and suspension). Suspension reasons: min_balance_exceeded (drawdown), dispute deduction, Stripe invoice payment failure. NO unsuspend path exists anywhere in non-test code — recovery is a manual DB/admin status edit.
- **Anchors:** ubb-platform/apps/billing/handlers.py:13-100; ubb-platform/apps/billing/queries.py:28-40; ubb-platform/apps/billing/connectors/stripe/webhooks.py:201-231,325-341; ubb-platform/apps/platform/events/schemas.py:118-152,225-233

#### 3.2.12 Runs with hard-stop (AI-session cost ceiling)

Two auditors described this capability independently (metering view and billing view); merged here with all anchors.

- **What:** A Run groups usage events into one logical workflow execution with snapshot-at-creation limits: a per-run cost ceiling and an estimated-wallet-balance floor. Creation happens ONLY via billing pre-check with `start_run=true`: `RiskService.check` gates on suspension/closure (seat OR billing owner), per-seat fixed-window rate limit, wallet affordability on the billing owner (skipped for postpaid tenants), and the per-seat budget cap; if allowed it creates the Run snapshotting wallet balance + `BillingTenantConfig.run_cost_limit_micros` / `hard_stop_balance_micros` (ubb-platform/apps/billing/gating/services/risk_service.py:8-75). Each `record_usage` with `run_id` atomically (SELECT FOR UPDATE) accumulates billed cost and event_count and enforces two hard stops BEFORE writing: (1) cost ceiling `new_total > cost_limit_micros` → reason `cost_limit_exceeded`; (2) wallet floor `balance_snapshot − new_total < hard_stop_balance_micros` → `balance_floor_exceeded` (ubb-platform/apps/platform/runs/services.py:54-106). On `HardStopExceeded` the endpoint kills the run in a separate transaction and returns 429 `{error: hard_stop_exceeded, reason, run_id, total_cost_micros, hard_stop: true}` (ubb-platform/api/v1/metering_endpoints.py:72-83); subsequent events on the run get 409 `run_not_active`. Close: POST `/runs/{run_id}/close` → status 'completed', returns `total_cost_micros` + `event_count`; idempotent terminal-state no-op (ubb-platform/apps/platform/runs/services.py:129-141). Abandoned runs are auto-closed by beat `close_abandoned_runs` every 15 min (ubb-platform/config/settings.py:204-207). The batch endpoint mirrors per-item semantics (a kill failure leaves the run active, loud log).
- **How:** POST `/api/v1/billing/pre-check {customer_id, start_run, run_metadata, external_run_id}` (ubb-platform/api/v1/billing_endpoints.py:255-265; requires the `billing` product); POST `/api/v1/metering/runs/{run_id}/close` (ubb-platform/api/v1/metering_endpoints.py:260-273). SDK: `UBBClient.pre_check` / `start_run` (ubb-sdk/ubb/client.py:111-154), `record_usage(run_id=...)` raises `UBBHardStopError` / `UBBRunNotActiveError` (ubb-sdk/ubb/metering.py:186-232; these are NEVER retried, ubb-sdk/ubb/retry.py:23-35), `close_run` (ubb-sdk/ubb/metering.py:240-243).
- **Options:** Run statuses: active/completed/failed/killed (ubb-platform/apps/platform/runs/models.py:6-11). Both limits nullable — no config means no ceiling/floor; limits are snapshots, so changing tenant config never affects in-flight runs. `external_run_id` free-text correlation; `metadata` dict; `kill_reason` recorded in `run.metadata`. Run accumulation counts BILLED cost, not provider cost (ubb-platform/apps/platform/runs/services.py:81). The balance floor uses the run-start snapshot minus run-local cost (an ESTIMATE; concurrent spend elsewhere is invisible to it, ubb-platform/apps/platform/runs/models.py:37-41).

#### 3.2.13 Dispute + Stripe-refund clawback

- **What:** A lost card dispute or a Stripe-side refund of a top-up automatically claws the money back out of the wallet (exactly-once per dispute/refund id), voids the originating top-up's grant lot first, cascades consumption across other lots to keep grant accounting consistent, and may suspend the customer.
- **How:** `charge.dispute.created` → error-log flag only (manual review). `charge.dispute.closed` status=lost → DISPUTE_DEDUCTION keyed `dispute:{dispute_id}` + `GrantLedger.clawback` (source = the checkout/auto_topup-born lot referencing the attempt) + suspend if below floor. `charge.refunded` → one STRIPE_REFUND txn PER refund object keyed `stripe_refund:{refund_id}` (partial-refund safe) + the same clawback cascade; no suspension check on this path. All matched to `TopUpAttempt` by `stripe_charge_id` with connected-account + livemode guards.
- **Options:** Dispute won/withdrawn = no action. Clawback can drive the balance negative (it mirrors money Stripe already took).
- **Anchors:** ubb-platform/apps/billing/connectors/stripe/webhooks.py:234-415; ubb-platform/apps/billing/wallets/grants.py:222-262

#### 3.3.4 Plan fee editing + price versioning (PATCH)

- **What:** A tenant edits access/seat fees on an existing plan. Stripe Prices are immutable, so a fee change on a PROVISIONED axis bumps `plan.pricing_version` once per call and mints a NEW Stripe Price on the same Product; the plan repoints so NEW subscribes get the new fee. Existing subscribers are GRANDFATHERED on their old price by default; `migrate_existing=true` repoints every active subscription item to the new price via `SubscriptionItem.modify` with `proration_behavior='none'` (no mid-cycle proration credit/charge). Unprovisioned axes just update the fee field (lazy provision reads it later); a fee dropped to 0 also just updates the field (subscribe skips zero axes). Same-fee replay is a no-op.

#### 3.3.5 Subscribe (one call provisions everything)

- **What:** One POST subscribes a customer to a plan. The single call: (1) resolves the BILLING OWNER (pooled seat → its business; money always lands on the owner's wallet/customer), (2) creates a Stripe Customer on the connected account if the owner lacks one (idempotency key `cust-create-{owner.id}`), (3) lazily provisions the plan's Stripe Product+Price per non-zero axis (ids saved immediately after each create so a partial provision resumes safely), (4) creates the Stripe Subscription with one licensed item per non-zero axis (access qty=1, seat qty=seats), `collection_method=charge_automatically`, `billing_cycle_anchor` = 00:00 UTC on the 1st of NEXT month, `proration_behavior=create_prorations` (so the partial first month is prorated), optional `automatic_tax`, idempotency key `sub-create-{owner.id}-{plan.id}`, (5) mirrors it locally into `StripeSubscription` + `CustomerSubscriptionItem` rows (axis classification: current plan price ids, then local price-id history, then a qty>1 heuristic).
- **How:** POST `/api/v1/platform/customers/{external_id}/subscribe {plan_key, seats>=0}` → `SubscriptionOrchestrator.subscribe` (ubb-platform/apps/subscriptions/orchestration/service.py:147-206; `_persist_mirror` 511-574). Returns `{subscription_id, amount_micros, quantity}`. Errors: 404 customer/plan, 422 not-charge-ready/Stripe-fatal (e.g. tax misconfig). SDK: `UBBClient.subscribe_customer(external_id, plan_key, seats=0)` (ubb-sdk/ubb/client.py:247-259). Subscribing requires a non-empty tenant `default_currency` (StripeFatalError otherwise, service.py:172-177); prices are minted in the tenant currency (service.py:136).
- **Options:** `seats >= 0` (`SubscribeIn`, ubb-platform/api/v1/schemas.py:429-431); the seats item is only created when `per_seat_micros > 0`. `automatic_tax` rider when `tenant.automatic_tax_enabled` (service.py:188-190). No choice of anchor, collection method, trial, or coupon.
- **Anchors:** ubb-platform/apps/subscriptions/orchestration/service.py:147-206,488-509,511-574; ubb-platform/api/v1/platform_endpoints.py:247-279

#### 3.3.6 Seat management: explicit set_seats + roster auto-sync

#### 3.3.9 Postpaid month-end usage invoicing (period close + two-phase push)

- **What:** For tenants with `billing_mode='postpaid'`: on the 1st at 00:05 UTC a beat task closes the PRIOR calendar month for every customer with any usage, re-keying seats onto their business. Per billing owner, a two-phase push: Phase 1 (DB, select_for_update) claims/creates the `CustomerUsageInvoice` (unique on customer+period_start), FREEZES the line-aggregation snapshot at first claim (retries replay the frozen lines; live-total divergence is alert-only, `postpaid.snapshot_divergence`), skips zero-usage / no-stripe-customer / not-charge-ready rows, reserves the sub-cent residual carry (take-and-zero `PostpaidResidualLedger`, pinned on `carry_in_micros` exactly once), flips to 'pushing'. Phase 2 (Stripe, no txn held): cents-floor each line carrying the residual forward; resume-not-recreate — retrieve the recorded pointer, else metadata lookup (`Invoice.list` + `usage_invoice_id` metadata match, skipping void), else `Invoice.create` as `auto_advance=False` DRAFT; the pointer is persisted BEFORE any item create (claim-stolen guard raises if the row was reclaimed); pin only MISSING InvoiceItems (`invoice=<id>`, metadata `usage_invoice_id`+`line_index`); `finalize_invoice(auto_advance=True)`. An already-finalized invoice → adopt with zero writes. Phase 3 (DB) records line items, deposits the residual back to the ledger, status='pushed', outbox `UsageInvoicePushed` event. All-sub-cent periods create NO invoice; the residual stays banked.
- **How:** Beat: `close_postpaid_usage_periods` crontab(minute=5, hour=0, day_of_month=1) (ubb-platform/config/settings.py:220-226); task ubb-platform/apps/billing/invoicing/tasks.py:74-93 (`_prior_month` 64-71; seat→parent targeting 86-88). Service: `PostpaidUsageService.push_customer_period` (ubb-platform/apps/billing/invoicing/services/postpaid_service.py:101-269) + `_push_to_stripe` (271-453). Idempotency keys `usage-invoice-{rec.id}{gen}`, `usage-item-{rec.id}{gen}-{i}`, `usage-finalize-{rec.id}{gen}` (368,440,449).
- **Options:** Line grouping (`PostpaidUsageConfig.usage_line_item_group_by`): '' = single total line; `tag:<key>` = group by a usage-event tag (NULL/''/missing collapse to '(other)'); `product_id` = group by product (ubb-platform/apps/metering/queries.py:398-429). BUSINESS owners IGNORE group_by: always one line per seat external_id (postpaid_service.py:27-40). Config: GET/PUT `/api/v1/billing/postpaid-config` (ubb-platform/api/v1/billing_endpoints.py:800-823; None-sentinel partial PUT). #### 3.3.10 Consolidated invoicing (opt-in: usage rides the subscription renewal)

- **What:** Tenant-level opt-in (`PostpaidUsageConfig.consolidate_with_subscription`, default False): at close, usage lines are pinned onto the billing owner's subscription-RENEWAL DRAFT invoice so the customer gets ONE Stripe invoice for subscription + usage. #### 3.5.2 API key lifecycle (self-serve)

- **What:** A tenant lists, mints, rotates, and revokes its own API keys using an existing key. The raw key is shown exactly once at mint/rotate; revocation is instant (per-request DB lookup, no caching).

#### 3.5.5 Outgoing webhooks: delivery, signing, retries, event catalog

- **What:** Tenants receive JSON POSTs `{event_type, event_id, tenant_id, timestamp, livemode, data}` signed with two HMAC schemes, retried on failure with bounded backoff, dead-lettered after 5 retries.
- **How:** Outbox pattern: `write_event` inside the domain transaction + on_commit Celery dispatch (ubb-platform/apps/platform/events/outbox.py:15-37) + an every-minute `sweep_outbox` catching lost dispatches/stuck rows (ubb-platform/apps/platform/events/tasks.py:84-132; beat ubb-platform/config/settings.py:180-183). Delivery handler `handle_webhook_delivery` registered per event type (ubb-platform/apps/platform/events/apps.py:40-46); per-handler idempotency via `HandlerCheckpoint` (ubb-platform/apps/platform/events/dispatch.py:59-77). Signing: `X-UBB-Signature` = legacy HMAC-SHA256 over body only (ubb-platform/apps/platform/events/webhooks.py:71-84); `X-UBB-Signature-V2` = `t=<ts>,v1=<HMAC-SHA256 over f"{ts}.{body}">` bounding replay (87-102,150-160); `X-UBB-Event-Type` header also sent. Timeout 10s (webhooks.py:16); non-2xx logged as a failed attempt; timeouts/network errors re-raise for retry (189-221). Retry: `OutboxEvent.retry_count` vs `max_retries=5` (ubb-platform/apps/platform/events/models.py:23-24), backoff 30s/2m/10m/30m/2h (tasks.py:20-26); exhausted → status='failed' + `logger.critical 'outbox.dead_letter'` (tasks.py:29-43,70-72); failed events never auto-deleted (tasks.py:136-137). `WebhookDeliveryAttempt` rows log each POST (webhook_models.py:28-46), cleaned up after 30d success / 90d failure (ubb-platform/apps/platform/events/tasks_webhook_cleanup.py:12-35).
- **Options — the deliverable event catalog (exactly 25; ubb-platform/apps/platform/events/apps.py:13-39; payloads ubb-platform/apps/platform/events/schemas.py):** `usage.recorded`, `usage.refunded`, `refund.requested`, `referral.created`, `referral.reward_earned`, `referral.expired`, `referral.payout_due`, `billing.withdrawal_requested`, `billing.balance_low`, `billing.balance_critical`, `billing.topup_requested`, `billing.customer_suspended`, `margin.customer_unprofitable`, `margin.provider_cost_spike`, `budget.threshold_reached`, `usage.invoice_pushed`, `usage.invoice_push_failed_permanent`, `auto_topup.requires_action`, `billing.balance_overage`, `billing.credit_grant_expiring`, `billing.credit_grant_expired`, `sandbox.reset_completed`, `tenant.api_key_created`, `tenant.api_key_rotated`, `tenant.api_key_revoked`. `customer.deleted` exists as an internal schema (schemas.py:89-93) but is NOT in the deliverable list.
- **Anchors:** ubb-platform/apps/platform/events/apps.py:13-46; ubb-platform/apps/platform/events/webhooks.py:71-223; ubb-platform/apps/platform/events/tasks.py:20-132; ubb-platform/apps/platform/events/outbox.py:15-37; ubb-platform/apps/platform/events/schemas.py:14-288; ubb-platform/apps/platform/events/tasks_webhook_cleanup.py:12-35

## 4. Four worked end-to-end journeys

### 4.1 From zero to same-day COGS and margin visibility

**Bootstrap (operator-side).** There is no public tenant-signup API; the Tenant row + first `ubb_live_` key come from `manage.py seed_dev_data` or Django admin (ubb-platform/apps/platform/tenants/management/commands/seed_dev_data.py:58-92; no Tenant-creating endpoint exists under `api/`). Then PATCH `/api/v1/tenant/config` to set products + billing_mode (+ `default_currency` BEFORE any money exists — it locks at 409 afterwards; ubb-platform/api/v1/tenant_endpoints.py:397-407). Register webhook endpoints (POST `/api/v1/webhooks/config/configs`) and wire `ubb.webhooks.verify_webhook` in the receiver.

**Same-day COGS.** (1) Create a customer via POST `/api/v1/platform/customers` (SDK `UBBClient.create_customer`, ubb-sdk/ubb/client.py:196). (2) Start firing `MeteringClient.record_usage` with `provider='openai'`, an event_type, `usage_metrics={'gpt4_input_tokens': N}` and tags `{'service':'chatbot','agent':'support'}` — no rate cards yet, so provider_cost = 0 and the response's `uncosted_metrics` tells you exactly which metrics lack cost cards (ubb-platform/api/v1/metering_endpoints.py:101-102). (3) Create cost cards per metric (`create_rate_card` card_type='cost', per_unit with `unit_quantity=1_000_000` for per-million-token rates). (4) Set a tenant markup (`set_markup`) so billed = cost + markup with zero price cards. (5) Same day, read GET `/analytics/usage` for totals + by_provider/by_customer and GET `/margin/{customer_id}` for live margin. (6) Once every metric is covered, flip PATCH `/api/v1/tenant/config require_cost_card_coverage=true` so future uncovered metrics are rejected 422 instead of silently costing 0 (ubb-platform/api/v1/tenant_endpoints.py:408-413).

**Tiered price book.** (1) Bulk-create the cost layer with `bulk_create_rate_cards` (1..100 cards, all-or-nothing; ubb-platform/api/v1/metering_endpoints.py:607-655). (2) Create graduated PRICE cards — e.g. tiers `[{up_to:1_000_000, rate_per_unit_micros:30, flat_micros:0},{up_to:10_000_000, rate_per_unit_micros:20},{up_to:null, rate_per_unit_micros:10}]` (max 20 tiers; last `up_to` must be null; requires the `billing` product) and/or a package card (block pricing via `unit_quantity` + per-block rate + one-time period `fixed_micros`, `tiers=[]`). (3) Add per-customer override cards (`customer_id=...`) and dimensional variants (`dimensions={'model':'gpt-4'}`) — most-specific card wins, customer beats tenant (ubb-platform/apps/metering/pricing/services/pricing_service.py:73-79). (4) Record usage: each event's `tier_breakdown` provenance shows prior_units, the bands it crossed, and cumulative before/after — marginals telescope so the month's event sum equals T(total) exactly. (5) Change prices mid-month via `update_rate_card`: new version, same lineage_id; the period ladder keys on lineage so the new price applies marginally from the current ladder position with no re-rating (ubb-platform/apps/metering/pricing/models.py:230-244). (6) Audit with `get_rate_card_history(lineage_id)` and `list_rate_cards(as_of=...)`.

**Backfill recovery (pipeline outage yesterday).** (1) Replay the gap with `record_batch` in chunks of ≤100, each event carrying its original `recorded_at` and a stable idempotency_key — naive timestamps are rejected client-side, and a crashed replay is rerun wholesale because per-item idempotency returns original event ids (ubb-sdk/ubb/metering.py:150-184). (2) Events land priced as-of their effective time on the card versions then valid, and advance the EFFECTIVE month's tier ladder (ubb-platform/apps/metering/usage/services/usage_service.py:160-167). (3) Per-item 422s are typed: `effective_at_too_old` (outside the tenant's `backfill_window_days`, default 34) and `billing_period_closed` (that month's postpaid invoice already froze — those events are unrecoverable through this surface). (4) Backfills into a prior month write `BackfillDirtyPeriod` markers; the hourly resnapshot task refreshes that month's margin snapshot automatically (ubb-platform/apps/metering/usage/models.py:75-100). (5) Verify with `usage_timeseries(granularity='hour')` over the gap window and GET `/analytics/usage` totals.

**Margin review loop.** (1) Monthly snapshots populate `CustomerEconomics`; set the alerting bar via PUT `/margin/threshold {min_margin_pct, consecutive_periods, provider_cost_spike_pct}`. (2) Record manual recurring revenue for non-Stripe customers via `set_customer_revenue` and pick usage-revenue recognition via `set_revenue_mode` ('billed' vs 'metered_only'). (3) Weekly: `get_unprofitable_customers()`, drill in with `get_customer_margin` + `get_margin_trend(periods=12)`, find which dimension is bleeding via `get_margin_by_dimension(tag_key='agent')`. (4) For pooled accounts, read the consolidated business via `get_business_margin(external_id)` with per-seat breakdown.

### 4.2 Prepaid wallet + bounded AI-agent spend

**Wallet from zero.** (1) PATCH `/api/v1/tenant/config billing_mode=prepaid` + products incl. `billing` (ubb-platform/apps/platform/tenants/models.py:106 requires the billing product). (2) Connect Stripe (SDK `start_connect_onboarding`, ubb-sdk/ubb/client.py:450) until `charges_enabled`. (3) POST `/api/v1/platform/customers` with a `stripe_customer_id` (ubb-platform/api/v1/platform_endpoints.py:37). (4) POST `/api/v1/billing/customers/{id}/top-up` → `checkout_url` → end customer pays → `checkout.session.completed` credits the wallet exactly-once (TOP_UP keyed `topup:{session.id}`) + PAID grant lot + receipt invoice (ubb-platform/apps/billing/connectors/stripe/webhooks.py:24-146; ubb-platform/apps/billing/connectors/stripe/receipts.py:20). (5) Optionally PUT `.../auto-top-up {is_enabled, trigger, amount}` so future drawdowns below the trigger auto-charge the saved card (ubb-platform/api/v1/billing_endpoints.py:137). (6) Record usage (POST `/api/v1/metering/usage`) — the outbox handler debits the wallet asynchronously, consuming grant lots expiry-first (ubb-platform/apps/billing/handlers.py:49-78). (7) Watch GET `.../balance` (incl. promo/expiring) and GET `.../transactions`; tenant webhooks receive `billing.balance_low` / `billing.balance_overage` / `billing.customer_suspended` as the balance falls.

**AI-agent run with a cost ceiling.** (1) The operator sets `BillingTenantConfig.run_cost_limit_micros` and `hard_stop_balance_micros` (ORM/admin only — no API; ubb-platform/apps/billing/tenant_billing/models.py:131-132). (2) `client = UBBClient(billing=True); res = client.start_run(customer_id, metadata={...}, external_run_id='job-42')` — the gate checks suspension/closure (seat + pooled owner), per-seat rate limit, owner balance vs the min_balance floor, per-seat budget cap; returns `run_id` + the snapshotted limits (ubb-platform/apps/billing/gating/services/risk_service.py:8-75). (3) Each LLM/tool call → `record_usage(..., run_id=run_id)`: the run row is locked, billed cost accumulates, and the FIRST event that would breach the cost ceiling or wallet floor is REJECTED with `UBBHardStopError` while the run is auto-killed (no event written; ubb-platform/apps/platform/runs/services.py:54-106; ubb-platform/api/v1/metering_endpoints.py:72-89). (4) Later calls raise `UBBRunNotActiveError` (409) — neither error is retried by the SDK (ubb-sdk/ubb/retry.py:29-30); the agent loop treats 429 as stop. (5) On normal completion `close_run(run_id)` returns the final `total_cost_micros` + `event_count`; abandoned runs are auto-closed every 15 min (ubb-platform/config/settings.py:204-207). Per-seat monthly ceilings additionally via PUT `.../budget {cap_micros, enforce_mode: enforcing, hard_stop_pct}`, which makes the NEXT pre-check deny with `budget_exceeded`; alert webhooks `budget.threshold_reached` at 50/80/100/110% by default.

**Dispute / refund recovery.** (a) Tenant-initiated usage refund — POST `/api/v1/billing/customers/{id}/refund {usage_event_id, idempotency_key}` credits the full billed cost; lot slices that funded the original debit are restored into still-live lots (promo stays promo, never withdrawable), the rest lands as base; `refund.requested` event recorded (ubb-platform/api/v1/billing_endpoints.py:268-335). (b) Stripe-side refund of a top-up — `charge.refunded` writes one STRIPE_REFUND per refund object keyed `stripe_refund:{refund_id}` and claws back grant lots (source top-up lot voided first, cascade until the grant-sum invariant holds) (ubb-platform/apps/billing/connectors/stripe/webhooks.py:344-415). (c) Card dispute — `charge.dispute.created` only flags for manual review (234-260); if the dispute closes LOST, DISPUTE_DEDUCTION keyed `dispute:{dispute_id}` debits the wallet, claws back lots, and suspends the customer if the balance falls below −min_balance (263-341); recovery from suspension requires a manual status edit (no unsuspend endpoint); the hourly Stripe sweep also audits succeeded top-up charges for amount/refund drift (ubb-platform/apps/billing/connectors/stripe/tasks.py:176-208).

### 4.3 Subscriptions + month-end postpaid close (operator view)

**Connect and first subscription.** (1) `UBBClient.start_connect_onboarding(return_url)` → POST `/api/v1/connect/start` → redirect the merchant to `authorize_url` (ubb-platform/api/v1/connect_endpoints.py:14-20). (2) Stripe redirects to GET `/api/v1/connect/callback?code&state`; `complete_oauth` stores the acct id + `charges_enabled` (ubb-platform/apps/billing/connectors/stripe/connect.py:58-83); the browser lands on `return_url?connected=true`. (3) Poll `get_connect_status()` until `onboarded=true` (lazy `charges_enabled` refresh, connect_endpoints.py:23-42; the `account.updated` webhook keeps it current thereafter). (4) `create_plan('pro', 'Pro', access_fee_micros=..., per_seat_micros=..., interval='month')` → POST `/platform/plans` (no Stripe objects yet). (5) `create_customer` for the business (`billing_topology='pooled'`) and its seats. (6) `subscribe_customer('acme', 'pro', seats=5)` → POST `/platform/customers/acme/subscribe`: one call ensures the Stripe Customer (`cust-create-{owner.id}`), lazily provisions Product+Price per non-zero axis, creates the subscription anchored to 00:00 UTC on the 1st of next month with create_prorations (partial month prorated), and mirrors locally (ubb-platform/apps/subscriptions/orchestration/service.py:147-206). (7) Ongoing: `customer.subscription.*` webhooks on `/api/v1/subscriptions/webhooks/stripe` keep the mirror fresh; `invoice.finalized/paid` on `/api/v1/webhooks/stripe` create/settle `SubscriptionInvoice` rows with hosted URLs; adding/removing/suspending seats auto-pushes the live seat quantity on commit (ubb-platform/apps/subscriptions/orchestration/seats.py:61-101); `set_seats` is the explicit override; cancel/pause/resume manage the lifecycle.

**Month-end close.** (0) Prereqs: tenant `billing_mode='postpaid'` (PATCH `/tenant/config`), charge-ready, customers have a `stripe_customer_id` (or the subscribe path created one for the owner); optionally PUT `/billing/postpaid-config {usage_line_item_group_by: 'tag:model', consolidate_with_subscription: true}`. (1) 1st of month 00:05 UTC: `close_postpaid_usage_periods` walks postpaid tenants, collects customers with prior-month usage, re-keys seats to their business, calls `push_customer_period` per owner (ubb-platform/apps/billing/invoicing/tasks.py:74-93). (2) Per owner: claim the row (unique customer+period_start), freeze the line snapshot, reserve the residual carry, two-phase push — draft `Invoice.create` (or resolve the <45-min-old subscription renewal draft when consolidation is on), pin items with `line_index` metadata, finalize (standalone only; the renewal auto-finalizes on Stripe's clock) — then record line items, deposit the new sub-cent residual, emit `UsageInvoicePushed` (ubb-platform/apps/billing/invoicing/services/postpaid_service.py:101-269). Sub-cent-only periods bill nothing and bank the residue. (3) Self-healing: hourly :45 `reconcile_postpaid_usage` reclaims stale 'pushing' rows, retries pending/failed (bounded 8 attempts/20h → failed_permanent + outbox alert), and revives recoverable skips (tasks.py:96-143). (4) Operator monitoring: GET `/billing/tenant/usage-invoices?period=2026-05` shows status/skip_reason/push_attempts/last_attempt_error per customer; a failed_permanent row is fixed at the root cause, then `manage.py repush_usage_invoice <row_id>` (resume the same invoice) or `--rebill-void` after voiding the Stripe invoice (fresh re-aggregation, rotated idempotency keys; refused for consolidated rows). (5) Collection: `invoice.finalized` → open, `invoice.paid` → paid (+paid_at), `invoice.payment_failed` stamps the timestamp, refreshes the rotated hosted URL and suspends the customer (the seat roster shrinks); the hourly :15 poller with a 4-day lookback repairs anything webhooks missed along the same transition table.

### 4.4 Evaluate safely, rotate keys, give end customers a window, watch your own bill

**Sandbox evaluation loop.** (1) With the live `ubb_live_` key, POST `/api/v1/tenant/sandbox` → `sandbox_tenant_id` + raw `ubb_test_` key (config shape copied from live, Stripe linkage NOT copied; ubb-platform/api/v1/tenant_endpoints.py:288-305; ubb-platform/apps/platform/tenants/services/sandbox_service.py:36-48). (2) Point the SDK/API at the same base URL with the `ubb_test_` key — every tenant-scoped feature (usage, wallets, rate cards, plans, webhook config, key lifecycle) works against the isolated sibling; Stripe calls use `STRIPE_TEST_SECRET_KEY` (refuse-live) and inbound test webhooks hit `/api/v1/webhooks/stripe/test`; outgoing webhook payloads carry `livemode: false`. (3) When state is dirty, POST `/api/v1/sandbox/reset {keep_config: true}` with the SANDBOX key → 202; the sandbox 401s during the async wipe, config models + Tenant + test keys survive, `sandbox.reset_completed` fires on completion (ubb-platform/api/v1/sandbox_endpoints.py:17-32; ubb-platform/apps/platform/tenants/tasks.py:63-150). (4) `keep_config=false` for a full config wipe. (5) Promote to live by REPLAYING the same config calls against the live key — there is no copy/promote endpoint. (6) GET `/api/v1/tenant/sandbox` (live key) any time for status; each POST `/tenant/sandbox` mints a fresh test key (= test-key rotation).

**API key rotation / incident response.** (1) GET `/api/v1/tenant/api-keys` to find the key id by prefix/label (`last_used_at` may lag up to 5 min — ubb-platform/api/v1/tenant_endpoints.py:175-177). (2) Routine rotation: POST `/api-keys/{id}/rotate` — successor minted and old key deactivated atomically; deploy the new raw key from this one response; the old key 401s on its next request (214-247). (3) Compromise: mint a replacement first (POST `/api-keys`), deploy it, then DELETE the compromised key — deleting the last active key is refused with 409 `last_active_key` (264-276). (4) Audit trail: `tenant.api_key_created/rotated/revoked` webhook events fire for each mutation (ubb-platform/apps/platform/events/apps.py:36-38).

**End-customer billing widget.** (1) The tenant backend holds `widget_secret` + `tenant_id` and mints a short-lived token: `UBBClient(widget_secret=..., tenant_id=...).create_widget_token('cust-uuid', expires_in=900)` — pure local JWT, no HTTP (ubb-sdk/ubb/client.py:91-107); ship it to the browser. (2) The widget calls `/api/v1/me/*` with `Authorization: Bearer <jwt>`; `WidgetJWTAuth` resolves the customer + tenant (ubb-platform/core/widget_auth.py:63-81). (3) What the customer sees: current wallet balance + promo/expiring grant summary (`/me/balance`); individual grant lots with expiries (`/me/grants`, first 100); wallet transaction history (`/me/transactions`, paginated); month-to-date usage broken down per event_type with billed cost (`/me/usage-summary` — always the SEAT's own usage; a business token sees the seat-aggregated rollup); their postpaid usage invoices (`/me/usage-invoices`) and subscription invoices (`/me/subscription-invoices`) with payment_status/status, period, and Stripe `hosted_invoice_url` + `invoice_pdf` links to pay or download — BUT only if they ARE the billing owner: a pooled seat gets empty invoice lists by design (sibling-spend privacy, ubb-platform/api/v1/me_endpoints.py:334-338,422-425); top-up receipts (`/me/invoices`). (4) Self-serve funding: POST `/me/top-up` (cent-aligned amount + redirect URLs) returns a Stripe Checkout URL on a connected tenant, or 202 `topup_requested` (outbox event to the tenant) when Stripe is not connected. Not available in the widget: cancel/pause subscription, payment-method management, plan changes — those are tenant-API-only.

**Monthly platform-fee observation.** (1) During the month each recorded usage event increments the tenant's open `TenantBillingPeriod` (ubb-platform/apps/billing/tenant_billing/services.py:46-73); the hourly reconcile corrects drift. (2) 1st 00:00 UTC the period closes and the fee is computed (`ProductFeeConfig` flat/percentage else legacy percentage, floored to cents; sandbox = 0). (3) 1st 01:00 UTC a Stripe invoice is created on UBB's own account against the tenant's `stripe_customer_id` (charge_automatically, usd). (4) The tenant watches it all via GET `/api/v1/tenant/billing-periods` (status open→closed→invoicing→invoiced, totals, fee) and GET `/api/v1/tenant/invoices` (stripe_invoice_id, amount, status) (ubb-platform/api/v1/tenant_endpoints.py:47-123).

## 5. What a customer cannot do

Every limit the four auditors found, consolidated and deduplicated. These are statements about HEAD of `tl-changes-05-06-26`, each with its anchor.

### 5.1 Pricing & metering

- **Pricing models that DO NOT exist:** volume tiering (all units at the reached tier's rate), stair-step/step-function, percent-of-revenue price cards, minimum commits, included-units allowances as a first-class field, time-of-day/peak pricing, and per-event tiering (tiers are strictly per-calendar-month cumulative). Only per_unit, flat, graduated, package exist (`PRICING_MODEL_CHOICES`, ubb-platform/apps/metering/pricing/models.py:40-45).
- Tiered (graduated/package) COST cards are forbidden — validation rejects them (ubb-platform/apps/metering/pricing/models.py:76-78) and the cost loop raises PricingError on a hand-crafted row (ubb-platform/apps/metering/pricing/services/pricing_service.py:120-125). COGS math is per_unit/flat only.
- Graduated: max 20 tiers (`MAX_GRADUATED_TIERS`, ubb-platform/apps/metering/pricing/models.py:51); last tier must be unbounded (`up_to=None`); `up_to` strictly increasing; tier rates/flats must be non-negative ints.
- Tier periods are fixed UTC calendar months (`month_bounds`, ubb-platform/apps/metering/pricing/services/tier_counter_service.py:16-31) — no weekly/quarterly/anniversary tier reset, no custom billing-period alignment.
- Rate-card matching is EXACT on provider/event_type/metric_name (empty string matches only empty string) — no wildcards, no regex, no "any provider" card (ubb-platform/apps/metering/pricing/services/pricing_service.py:69-72). `product_id` is stored on cards but is NOT a matching key (absent from `_resolve_card` filters).
- Partial price-coverage quirk: if at least one metric matches a price card, unmatched metrics contribute 0 billed — the markup fallback applies only when NO metric matched any price card (ubb-platform/apps/metering/pricing/services/pricing_service.py:141-179).
- Creating a duplicate ACTIVE rate card (same tenant/customer/type/provider/event_type/metric/dimensions/currency) is not caught at the endpoint — the partial unique constraint raises an unhandled IntegrityError (500, not 422) (ubb-platform/api/v1/metering_endpoints.py:588-604; ubb-platform/apps/metering/pricing/models.py:152-163).
- Ingestion caps: batch ≤100 events/call (422 above); caller cost fields capped at 999_999_999_999 micros (~$1M) per event; tags ≤50 keys, key regex `^[a-z][a-z0-9_]{1,63}$`, values ≤256 chars; `usage_metrics` values must be ≥0 — no negative/correction events; corrections only via the billing refund endpoint, which is wallet-side (ubb-platform/api/v1/schemas.py:30-31; ubb-platform/apps/metering/usage/services/usage_service.py:16,77-94).
- Schema/DB mismatch: `request_id` accepts up to 500 chars in the API schema (ubb-platform/api/v1/schemas.py:27) but the column is varchar(255) (ubb-platform/apps/metering/usage/models.py:17) — a 256–500 char request_id fails at the DB layer.
- UsageEvents are immutable: no update, no delete; model `save()`/`delete()` raise (ubb-platform/apps/metering/usage/models.py:66-72). No void/adjust endpoint on the metering surface.
- No GET `/usage/{event_id}`; the usage list omits tags, usage_metrics and pricing_provenance — full provenance is only available on the ingest response (ubb-platform/api/v1/metering_endpoints.py:237-254).
- Backfill bounds: future skew 5 min; window default 34 days, hard max 60 (model clean), and `backfill_window_days` has NO public API writer — DB/ops change only. Months already claimed by a postpaid invoice are permanently closed to backfill (`billing_period_closed`) (ubb-platform/apps/platform/tenants/models.py:60-64,110-113; ubb-platform/apps/metering/usage/services/usage_service.py:31-74).
- Analytics: dimensions ≤6 per call; timeseries hourly window ≤92 days but ONLY enforced when both start_date and end_date are passed (an open-ended hourly query is unbounded, ubb-platform/api/v1/metering_endpoints.py:492); breakdowns are unpaginated full aggregates (no top-N parameter); inconsistent end_date semantics — `/analytics/usage` end_date is INCLUSIVE while `/analytics/usage/timeseries` end_date is EXCLUSIVE (ubb-platform/api/v1/metering_endpoints.py:351-354 vs ubb-platform/apps/metering/queries.py:257-259).
- Margin: trend periods clamp 1..36 monthly snapshots; `/margin/unprofitable` reads persisted monthly snapshots (not live); default windows are month-to-date; no SDK wrapper for `/margin/summary`, the `/margin` list, or `/margin/threshold`; the SDK `CustomerMargin` dataclass silently drops API fields (`revenue_mode`, `usage_revenue_micros`, `total_revenue_micros`, `event_count`) because of its field filter (ubb-sdk/ubb/metering.py:262-264; ubb-sdk/ubb/types.py:133-140).
- No streaming/websocket ingestion, no CSV/S3 bulk import, no event search by metadata/request_id, no export endpoint — retrieval is the cursor-paginated per-customer list plus aggregates (ubb-platform/api/v1/metering_endpoints.py:208-254,338-497).

### 5.2 Money, wallets & spend control

- Recording usage is NEVER blocked by funds or suspension — `record_usage` has no status/balance check (ubb-platform/apps/metering/usage/services/usage_service.py:113-186); enforcement is only at `/pre-check` (advisory unless the integrator calls it), run hard-stops, and after-the-fact suspension. A customer who skips pre-check can overdraw without bound until suspension fires asynchronously.
- No unsuspend/reactivate API exists anywhere (auditor grep for `status="active"`/reactivate across non-test code matches only referral programs); a suspended customer stays gated until someone edits the row manually — topping up does NOT clear suspension (`apply_topup_credit` never touches `customer.status`, ubb-platform/apps/billing/topups/services.py:61-109).
- POST `/debit` has no balance floor and emits NO overage/suspend/balance-low events (ubb-platform/api/v1/billing_endpoints.py:54-91) — external debits can silently drive a wallet negative; `idempotency_key` is optional on debit/credit (None = retries double-apply).
- Hosted checkout and receipt invoices hardcode currency 'usd' regardless of the tenant's `default_currency` (ubb-platform/apps/billing/connectors/stripe/stripe_api.py:59; ubb-platform/apps/billing/connectors/stripe/receipts.py:64) even though wallets/grants carry the tenant currency — a non-USD tenant's top-up checkout would charge USD. Only 18 two-decimal currencies are supported tenant-wide; zero-decimal (jpy/krw) rejected (ubb-platform/apps/platform/tenants/models.py:12-20); one currency per tenant, no FX (ubb-platform/apps/metering/usage/services/usage_service.py:135-147).
- Validation bounds: top-up/withdraw/grant/auto-top-up amounts must be cent-aligned (% 10_000 == 0); debit/credit/grant amounts capped at 999_999_999_999 micros (~$1M) per call; grant `expires_in_days` ≤ 3650; idempotency keys ≤ 400–500 chars; list limits clamped to 100 (ubb-platform/api/v1/schemas.py:108-199).
- Usage refunds are whole-event only (no partial amount); a second refund of the same event under a new idempotency_key is allowed and lands entirely as base credit (`refunded_micros` caps only the LOT re-fund, not the wallet credit) (ubb-platform/api/v1/billing_endpoints.py:268-335).
- Expired/voided grant lots are never re-inflated by refunds (their share lands as base, ubb-platform/apps/billing/wallets/grants.py:281-284); grant expiry/void debits are clamped to never push the balance below zero, so an overdrawn wallet can "keep" expired credit it already spent (grants.py:82-83; ubb-platform/api/v1/billing_endpoints.py:535).
- Promo credit is not withdrawable (withdraw availability = balance − active promo remaining, ubb-platform/api/v1/billing_endpoints.py:225) but IS spendable by `/debit` and usage; `/credit` never creates a lot (base money only).
- Per-customer `min_balance`, `topup_grant_expiry_days`, `RiskConfig` rate limits/`gate_fail_closed`, and `BillingTenantConfig` run_cost_limit/hard_stop_balance/min_balance have NO API or admin surface on this branch — ORM-only (auditor grep of `api/` shows zero references; ubb-platform/apps/billing/wallets/admin.py registers only Wallet + WalletTransaction, omitting CustomerBillingProfile).
- Run balance floor uses the pre-check balance SNAPSHOT minus run-local cost — concurrent spend from other runs/seats is invisible to it (ubb-platform/apps/platform/runs/models.py:37-41; ubb-platform/apps/platform/runs/services.py:82); a batch kill failure leaves the run active with only a log (ubb-platform/api/v1/metering_endpoints.py:160-166).
- Runs: no public endpoints to create (only billing pre-check with the `billing` product), kill, list, or GET a run — the only reads are the pre-check response and close_run's summary; run cost ceiling/wallet floor have no public API writer (auditor grep of `api/` matches tests only). Run accumulation counts BILLED cost, not provider cost (ubb-platform/apps/platform/runs/services.py:81).
- Hard-stop is per-run only — there is no per-customer or per-tenant real-time spend ceiling on this surface (budget caps are enforced at pre-check time, default advisory).
- Budget caps: `cap_micros <= 0` disables the cap; enforcement is calendar-month UTC only (no custom periods despite the model's `period` field); the live counter skips backdated prior-month events — enforcing caps are evadable within `Tenant.backfill_window_days` (documented bypass, ubb-platform/apps/billing/gating/services/budget_service.py:136-142); threshold alerts dedup via OutboxEvent query, not a constraint (a rare duplicate is possible under concurrency, budget_service.py:100-111).
- `reconcile_wallet_balances` is detect-only (logs drift, never repairs) (ubb-platform/apps/billing/wallets/tasks.py:10-96); drawdown self-repair covers prepaid tenants only and a 7-day arrival window (tasks.py:126-127); the Stripe top-up sweep covers 4 days and only PIs carrying `metadata.topup_attempt_id` (ubb-platform/apps/billing/connectors/stripe/tasks.py:127-153).
- Rate limiting is per-seat fixed-window in Redis and silently disabled when Redis is down; there is no fail-closed option for it or for the wallet-balance gate (ubb-platform/apps/billing/gating/services/risk_service.py:22-33).
- Top-up, debit, credit, withdraw, refund and auto-top-up act on the ADDRESSED customer's own wallet — they do not resolve the pooled billing owner (only grants, pre-check, and the usage drawdown/repair do); a pooled seat addressed directly gets its own (normally empty) wallet (ubb-platform/api/v1/billing_endpoints.py:54-335 vs :437).
- Withdrawals never move real money — no Stripe payout integration; the platform writes the ledger row and emits `billing.withdrawal_requested`; the payout is the tenant's job (ubb-platform/api/v1/billing_endpoints.py:201-252).

### 5.3 Subscriptions & postpaid

- Trials and coupons DO NOT EXIST on this surface — an explicit non-goal; Stripe's own trial_period_days/Coupons on the connected account are the levers, UBB does not wrap or sync them (ubb-platform/apps/subscriptions/orchestration/service.py:248-252; ubb-platform/api/v1/platform_endpoints.py docstrings 147-148,216-217,229-230,241-242).
- Plan `interval` is unvalidated: `PlanIn` accepts any string (ubb-platform/api/v1/schemas.py:399-405); the model column is max_length=5 (ubb-platform/apps/subscriptions/models.py:90), so a >5-char value (e.g. 'quarter') fails at the DB layer while 'week'/'day' would flow through to Stripe; `_INTERVAL_MONTHS` only maps month/year and the period fallback silently treats unknown intervals as 1 month (ubb-platform/apps/subscriptions/orchestration/service.py:52,74-80).
- Single currency per tenant, no multi-currency/FX; only 2-decimal currencies; currency permanently locked once money exists; the provisioned Price currency is NOT stored locally so it can never be cross-checked later (ubb-platform/api/v1/tenant_endpoints.py:339-407; ubb-platform/apps/subscriptions/orchestration/service.py:165-177).
- Billing cycle anchor is hard-coded to 00:00 UTC on the 1st of next month with create_prorations — no custom anchors, no anniversary billing, no proration_behavior choice at subscribe (ubb-platform/apps/subscriptions/orchestration/service.py:65-71,200-201).
- Lifecycle verbs only target "the latest non-canceled subscription of the billing owner" — no subscription-id addressing, no multi-subscription management, no plan upgrade/downgrade verb (ubb-platform/apps/subscriptions/orchestration/service.py:475-486). `cancel(at_period_end=false)` is irreversible here: `resume()` only clears a PENDING cancel/pause. The orchestrator's exhaustive verb list is ensure_plan_provisioned/subscribe/set_seats/cancel/pause/resume/update_plan_prices — nothing else.
- Stripe-side changes that do NOT sync back: only `customer.subscription.created/updated/deleted` are handled (ubb-platform/apps/subscriptions/api/endpoints.py:145-149) — dashboard edits to Products/Prices never update `TenantBillingPlan` fees; `handle_subscription_updated` refreshes the `StripeSubscription` mirror aggregates but NOT the `CustomerSubscriptionItem` rows (ubb-platform/apps/subscriptions/api/webhooks.py:54-81); payment-method changes, Stripe-side coupons/trials, and pause/cancel have no local model beyond the pause_collection/cancel_at_period_end flags. POST `/subscriptions/sync` repairs subscription mirrors only — not plans, not items' axis history beyond what `_sum_items` derives.
- No periodic seat-quantity reconciler: a swallowed `seat.qty_push_failed` only self-corrects on the NEXT roster change (explicit note, ubb-platform/apps/subscriptions/orchestration/seats.py:12-15). Seat counts: `seats >= 0` with no upper bound (ubb-platform/api/v1/schemas.py:431,435).
- Postpaid close is strictly monthly, prior-calendar-month, UTC — no mid-month, threshold-triggered, or custom billing periods (ubb-platform/apps/billing/invoicing/tasks.py:64-93); push retry is bounded to 8 attempts/20h (inside Stripe's 24h idempotency window), then a permanent park requiring operator action (ubb-platform/config/settings.py:264-267).
- The line snapshot freezes at first claim: usage events that land in the period AFTER the claim are permanently excluded from that invoice (divergence is alert-only, `postpaid.snapshot_divergence` — ubb-platform/apps/billing/invoicing/services/postpaid_service.py:139-160); group_by changes mid-retry do not re-aggregate.
- `--rebill-void` is refused for consolidated rows (the target is the subscription renewal, not UBB's invoice — ubb-platform/apps/billing/invoicing/management/commands/repush_usage_invoice.py:48-64); a terminal row's pinned carry_in is parked, never auto-returned (postpaid_service.py:69-72).
- The usage-invoice consolidation window is ~45 minutes after the cycle anchor (`CONSOLIDATION_MAX_DRAFT_AGE_SECONDS`, postpaid_service.py:17); a paused or non-UBB-managed subscription is never consolidated (ubb-platform/apps/subscriptions/ports.py:23-49).
- The AR payment_status transition table is strict: paid and void are final — a Stripe-reported regression (e.g. paid→open) is logged and never applied (ubb-platform/apps/billing/connectors/stripe/invoice_routing.py:59-69; ubb-platform/apps/billing/invoicing/tasks.py:226-238). GET `/subscriptions/customers/{id}/invoices` lists PAID invoices only (ubb-platform/apps/subscriptions/api/endpoints.py:89-93); open/uncollectible subscription invoices are visible to tenants only via the AR fields, and to end customers via `/me/subscription-invoices`.
- `invoice.payment_failed` unconditionally SUSPENDS the matched customer (ubb-platform/apps/billing/connectors/stripe/webhooks.py:218-231) — a tenant cannot opt a customer out of suspension-on-failed-invoice on this surface; reactivation is not exposed here.
- No plan list/GET/DELETE endpoints (platform_endpoints.py has only POST `/plans` and PATCH `/plans/{key}`); PATCH cannot change interval/name/usage_mode.
- `plan.usage_mode` ('invoice_item'|'none') is dead config at HEAD: written and echoed but read by no billing code path (auditor grep: refs limited to ubb-platform/api/v1/platform_endpoints.py:118,134 + schemas + migration); whether usage is invoiced is governed solely by `tenant.billing_mode='postpaid'`.

### 5.4 Hierarchy & end-customer widget

- The account hierarchy is exactly one level: seats under a business; a seat cannot be a parent; business/seat types and topology are fixed at create (no PATCH endpoint to convert account_type or re-parent) (ubb-platform/api/v1/platform_endpoints.py:37-81).
- `/me` API: a pooled seat sees EMPTY usage-invoice and subscription-invoice lists (owner gate, ubb-platform/api/v1/me_endpoints.py:334-338,422-425) and an empty own-wallet grants list; `/me/grants` is hard-capped at 100 lots with no pagination (166-191); usage-summary is fixed to current-UTC-month-to-date — no historical periods (382-413). No widget endpoints for cancel/pause, payment methods, or plan info. Widget JWTs are HS256 with a single per-tenant secret; no aud/scope claims (ubb-platform/core/widget_auth.py:10-18).
- No HTTP endpoint mints widget tokens — tenants must run the SDK (or equivalent `jwt.encode`) server-side with `widget_secret` + `tenant_id` (ubb-sdk/ubb/client.py:91-107).

### 5.5 Platform, config & operability

- No self-serve tenant signup: no API endpoint creates a live Tenant — bootstrap is seed_dev_data/Django admin only (auditor grep: no `Tenant.objects.create` outside tests/seed; ubb-platform/config/urls.py has no signup route).
- Tenant config PATCH cannot change: name, min_balance_micros, hard_stop_balance_micros, run_cost_limit_micros, platform_fee_percentage, backfill_window_days, branding_config, metadata — none are in `TenantConfigIn` (ubb-platform/api/v1/schemas.py:389-396 vs ubb-platform/apps/platform/tenants/models.py:30-64).
- 'metering' can never be removed from products; prepaid/postpaid billing_mode without the 'billing' product is rejected (ubb-platform/apps/platform/tenants/models.py:99-109).
- API keys: cannot revoke the last active key (409); no key scopes/permissions, no expiry, no IP allowlists; the raw key is irretrievable after the mint/rotate response; `last_used_at` lags up to 5 min (ubb-platform/api/v1/tenant_endpoints.py:250-282; ubb-platform/config/settings.py:208-211).
- Sandbox: exactly one per tenant (`uq_one_sandbox_per_parent`), no nested sandboxes, no sandbox delete endpoint; reset is async (202) with the sandbox 401ing mid-wipe and left inactive on partial failure; no config-promotion/copy-to-live endpoint (ubb-platform/apps/platform/tenants/models.py:78-83; ubb-platform/apps/platform/tenants/tasks.py:126-131).
- Sandbox Stripe: requires `STRIPE_TEST_SECRET_KEY` (sk_test_) configured server-side or every sandbox Stripe call fails loudly; a sandbox can never be platform-fee invoiced (ubb-platform/apps/billing/stripe/services/stripe_service.py:47-54,157-160).
- Webhook configs: no PATCH/update (delete + recreate to change url/secret/filters), no test-delivery/ping endpoint, no tenant-facing delivery-attempt/log API, no secret-rotation primitive (single secret per config; the SDK verifier accepts multiple `v1=` sigs but the sender only ever signs with one secret) (ubb-platform/apps/platform/events/api/webhook_endpoints.py:27-79; ubb-platform/apps/platform/events/webhooks.py:147-150).
- Webhook secrets are caller-supplied (min 32 chars) and stored in PLAINTEXT in `ubb_tenant_webhook_config.secret` (ubb-platform/apps/platform/events/webhook_models.py:13).
- Webhook URLs: https only, public IPs only — no http, no localhost/private ranges even for development (ubb-platform/core/url_validation.py:24-46).
- Webhook delivery: at-least-once with NO per-config checkpoint — the `HandlerCheckpoint` is per event+handler, so if one config's POST fails, the retry re-POSTs to ALL matching configs including ones that already succeeded; receivers must dedupe on `event_id` (ubb-platform/apps/platform/events/dispatch.py:59-77; ubb-platform/apps/platform/events/webhooks.py:105-132).
- Webhook retries are bounded: 5 retries over ~2h43m (30s/2m/10m/30m/2h) then dead-letter status='failed' with a CRITICAL log; no tenant-facing redelivery API — recovery is operator/ORM-level (ubb-platform/apps/platform/events/models.py:24; ubb-platform/apps/platform/events/tasks.py:20-26,70-72).
- The event catalog is fixed at the 25 registered types; `customer.deleted` is internal-only (not deliverable); no custom/tenant-defined event types (ubb-platform/apps/platform/events/apps.py:13-39; ubb-platform/apps/platform/events/schemas.py:89-93).
- List endpoints (billing-periods, invoices) cap limit at 100 per page (ubb-platform/api/v1/tenant_endpoints.py:50,90).
- Platform-fee config (`ProductFeeConfig` / `BillingTenantConfig`) has no tenant-facing API — tenants can SEE periods/invoices but not their fee schedule terms; platform-fee invoices are hardcoded usd regardless of tenant `default_currency` (ubb-platform/apps/billing/stripe/services/stripe_service.py:195).
- OpenAPI docs UIs and specs are publicly reachable (no auth/docs_decorator) at every mount's `/docs` + `/openapi.json` — an information exposure to consider before production (ninja defaults, ubb-platform/.venv/Lib/site-packages/ninja/main.py:58-61; no overrides in the repo).
- The platform_api endpoints (customers/plans/subscribe/seats/lifecycle) carry no product gate — ApiKeyAuth only (ubb-platform/api/v1/platform_endpoints.py:34) — while the subscriptions read API requires the `subscriptions` product entitlement (ubb-platform/apps/subscriptions/api/endpoints.py:34).
- Health checks are fixed to database + redis; no Celery or Stripe checks (ubb-platform/api/v1/endpoints.py:16-38).

### 5.6 SDK gaps & footguns

- `UBBClient.withdraw` delegates to `/debit` (not `/withdraw`): bypasses the insufficient-balance check, promo-exclusion, and the `billing.withdrawal_requested` event, and can overdraw (ubb-sdk/ubb/client.py:501-514, its docstring says so). `UBBClient.refund_usage` delegates to `/credit` (not `/refund`): credits the caller-supplied amount as a plain ADJUSTMENT, bypassing lot-aware re-funding and `refund.requested` (client.py:516-530). `BillingClient` exposes neither real endpoint.
- `UBBClient.record_usage` (facade) cannot send `usage_metrics` or `recorded_at` and makes `provider_cost_micros` mandatory — engine-priced or backfilled ingestion must use `client.metering.record_usage` directly (ubb-sdk/ubb/client.py:156-188).
- SDK debit/credit never send an `idempotency_key` (not replay-safe) (ubb-sdk/ubb/billing.py:80-236).
- The SDK `CustomerMargin` dataclass silently drops API fields (`revenue_mode`, `usage_revenue_micros`, `total_revenue_micros`, `event_count`) (ubb-sdk/ubb/metering.py:262-264; ubb-sdk/ubb/types.py:133-140).
- No SDK wrapper for: sandbox reset (POST `/api/v1/sandbox/reset`), webhook-config CRUD, tenant billing-periods/invoices, `/margin/summary`, the `/margin` list, or `/margin/threshold` — raw HTTP needed (auditor greps of ubb-sdk found no matches).
- `SubscriptionsClient` is read/sync-only (`sync`, `get_subscription`, `get_invoices` — ubb-sdk/ubb/subscriptions.py:76-93); ALL write verbs live on `UBBClient` and ride the metering product client.
