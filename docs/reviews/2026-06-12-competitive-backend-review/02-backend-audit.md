## 1. APIs & backend logic

**Relevance: critical. Verdict: adequate.**

*(No dedicated source audit existed for this category; the following anchors were gathered and verified directly from the repo for this document.)*

### What exists

- **Single + batch ingestion endpoints** with typed request schemas: `POST /api/v1/metering/usage` (ubb-platform/api/v1/metering_endpoints.py:42-103) and `POST /usage/batch` accepting 1–100 items (`UsageBatchRequest` with `min_length=1, max_length=100`, ubb-platform/api/v1/schemas.py:56-57; endpoint at metering_endpoints.py:184-205).

  > **In plain terms.** UBB gives customers two front doors to report usage: one event at a time, or up to 100 events in a single call. Both doors are strictly typed — the request must match a fixed shape or it bounces immediately, so malformed data never reaches the money logic. This matters because usage ingestion is the firehose that everything else (pricing, invoices, wallets) depends on; a sloppy front door corrupts the books downstream. Think of it as a parcel depot that refuses to accept a box unless the label is filled in correctly. The 100-event batch ceiling matches Metronome and Lago and is below Orb's 500 — competitive, not class-leading. The honest caveat to hold in mind: the batch door is convenience only, not a high-throughput pipe — see the gap analysis on why batching here doesn't actually buy you scale.

- **Idempotency is DB-enforced and replay-first:** `record_usage` checks `(tenant, customer, idempotency_key)` before any validation and returns the original event on replay — explicitly *before* effective_at validation so a whole-batch retry succeeds even if the backfill window has since aged past the timestamp (ubb-platform/apps/metering/usage/services/usage_service.py:122-128). A raced duplicate insert is caught via IntegrityError and resolved back to the existing event (usage_service.py:187-197). The dedup window is **unbounded** (a permanent unique constraint, not a TTL).

  > **In plain terms.** This is a genuine brag-worthy strength. "Idempotency" means if a customer's code sends the same usage event twice (network hiccup, retry, a whole batch replayed), UBB bills it once and returns the original result — never double-charges. UBB enforces this in the database itself (the strongest possible place), and checks for a duplicate *before* any other validation, so even a retry that arrives after the backfill window has closed still succeeds. Crucially, UBB's dedup memory is permanent, whereas every competitor forgets after a fixed window (Metronome 34 days, Stripe ~24h). Analogy: a doorman who never forgets a face, versus one who forgets after a month. The business value: customers can retry safely forever without fear of duplicate revenue charges. The only catch — this superpower is currently invisible to buyers because it isn't documented publicly.

- **Batch semantics are deliberate and documented in code:** each item runs as an independent atomic commit ("deliberately NOT one mega-transaction, which would hold Run/counter locks for the whole batch"), always HTTP 200 with positionally aligned per-item results and `succeeded`/`failed` counts; per-item error bodies mirror the single endpoint byte-for-byte (metering_endpoints.py:106-205).

  > **In plain terms.** When a customer sends 100 events at once, UBB processes each one independently: if item 37 is bad, items 1-36 and 38-100 still succeed, and the response tells you exactly which slot failed and why. This is a better contract than Lago, which throws away the *entire* batch if any single event is malformed. The reasoning is also deliberately documented in the code (the engineers chose not to wrap the whole batch in one giant lock, which would have stalled the system). Analogy: a checkout where one declined card doesn't cancel everyone else's groceries. Business value: a customer's bulk upload doesn't get nuked by one stray record, which means fewer support tickets and less lost usage data. The caveat: this clean per-item contract is about *correctness*, not *speed* — the items still run one after another.

- **Typed error envelopes** rather than bare 500s: `hard_stop_exceeded` (429, with run-kill side effect), `run_not_active` (409), `pricing_error` (422), typed `EffectiveAtError` codes (422), generic validation (422) (metering_endpoints.py:72-100).

  > **In plain terms.** When something goes wrong, UBB returns a precise, machine-readable error code (e.g. "run_not_active", "pricing_error") with the correct HTTP status, rather than a generic "something broke" 500. This matters because UBB's customers are developers integrating it into their own AI products; precise errors let their code react automatically — back off, retry, alert a human, or surface a clear message — instead of guessing. Analogy: a check-engine system that names the actual fault versus one warning light that could mean anything. Business value: faster, cheaper integrations and far fewer "why did my call fail?" support escalations. One of these errors even has a real-world side effect — "hard_stop_exceeded" actively kills a runaway spend run — which shows the error layer is wired into the money-protection logic, not just cosmetic.

- **Backfill window is first-class and tenant-configurable:** `Tenant.backfill_window_days` defaults to **34 days**, clean-validated to 0–60 with the 60-day cap reasoned against the reconcile horizon (ubb-platform/apps/platform/tenants/models.py:60-64,110-113); past-dated `effective_at` outside the window raises `effective_at_too_old` (usage_service.py:37,59-61).

  > **In plain terms.** "Backfill" means accepting usage that's dated in the past — say, events that were delayed and arrive late. UBB lets each customer set how far back it will accept (default 34 days, adjustable 0-60), and rejects anything older with a clear typed error. This matters because late-arriving usage is a fact of life in metering, but accepting *unlimited* backdating would let stale data silently rewrite already-closed bills. The window is a deliberate guardrail. Analogy: an expense system that accepts last month's receipts but not last year's. Business value: UBB's 34-day default exactly matches Metronome and is more flexible than Stripe's fixed 35 days, so it's competitive out of the box and tunable per customer — a selling point for customers with messy or batched data pipelines.

- **Money mutation endpoints carry body-level idempotency keys:** withdraw, debit, credit, grant all require/accept `idempotency_key` (schemas.py:140-199), backed by the wallet partial-unique constraint (see §2).

  > **In plain terms.** Every operation that *moves money* — withdraw, debit, credit, grant credit — requires the caller to attach a unique key, and the database enforces that the same key can never run twice. This is the same anti-double-charge guarantee as for usage events, applied to the wallet/ledger. It matters enormously because these are the operations where a duplicate is a real financial error: charging a customer's balance twice, or granting free credit twice. The database constraint is the actual guarantee — not hopeful application code. Analogy: a bank transfer reference number that the bank refuses to process a second time. Business value: customers and UBB itself are protected from the most expensive class of bug in billing. The caveat (covered in a gap): this is enforced per-endpoint via the request body, not via a single uniform header convention, which is a minor consistency wrinkle versus Orb/Metronome.

- **Input validation at the edge:** cost fields bounded `ge=0, le=999_999_999_999`, `usage_metrics` values validated non-negative, currency pinned to the tenant's single currency at a documented choke point (schemas.py:30-44; usage_service.py:135-147), cursor pagination with clamped limits (metering_endpoints.py:208-230).

  > **In plain terms.** UBB checks the basic sanity of incoming data right at the door: cost amounts must be non-negative and under a ceiling, usage counts can't be negative, and the currency is forced to match the customer's single configured currency at one well-defined checkpoint. This matters because billing math is unforgiving — a negative cost or a stray currency could silently produce a wrong invoice or a refund-shaped charge. Doing it "at the edge" means bad data is rejected before it can pollute the ledger. Analogy: a cashier who verifies the price tag is a real positive number before ringing it up. Business value: structural protection against a whole category of "garbage in, wrong bill out" errors, plus a deliberate decision to avoid currency-mixing complexity (no FX), which keeps the money model simple and auditable.

- **Pricing runs inside the ingest transaction** with versioned rate cards priced `as_of=effective_at` and tier ladders advanced under a row lock inside a savepoint so raced duplicates roll back the ladder advance (usage_service.py:156-186).

  > **In plain terms.** When an event arrives, UBB prices it on the spot — looking up the right rate card *as it existed at the time the usage happened*, and advancing volume tiers under a lock so that two simultaneous events can't both claim the same tier discount. Pricing and recording happen as one all-or-nothing operation: if anything fails, the whole thing rolls back cleanly. The strength here is correctness — prices are historically accurate (a rate change last week doesn't retroactively rewrite older events) and race-proof. Analogy: a till that always applies the price that was on the shelf at the moment you picked the item up. The trade-off, and it's the central one in this section: doing all this pricing work synchronously *inside* every single ingest call is what caps UBB's throughput — accuracy bought at the cost of an asynchronous high-volume path (see the gap analysis).


### Competitor evidence

- **Metronome:** `/ingest` accepts 1–100 events per request; `transaction_id` is the idempotency key with a **34-day dedup window**; backdating up to 34 days "will immediately impact live customer spend" (https://docs.metronome.com/api-reference/usage/ingest-events). A separate `Idempotency-Key` header on POSTs is cached ≥24h, replays the original result, and 409s on same-key-different-params — "even if a request returns an HTTP 500 error" (https://docs.metronome.com/developer-resources/use-api/idempotency/). Invoice finalization has a 24h default grace period for late events (https://docs.metronome.com/guides/get-started/core-concepts/how-invoicing-works).
- **Orb:** 500 events per batch by default; per-event `idempotency_key` deduplicated within the account grace period; event `properties` allow primitives only (https://docs.withorb.com/events-and-metrics/event-ingestion; https://docs.withorb.com/quickstart/ingest). General-API `Idempotency-Key` header valid 48h with an `Idempotent-Replayed: true` response header and 409 on conflicting reuse (https://docs.withorb.com/api-reference/idempotency). Backfills are "always audit-safe" — originals archived, never overwritten or deleted (https://docs.withorb.com/guides/events-and-metrics/reporting-errors).
- **Lago:** `POST /events/batch` accepts up to 100 events, rejects the whole batch if any item fails structural validation, and processes asynchronously (https://getlago.com/docs/api-reference/events/batch; whole-batch rejection per https://getlago.com/docs/guide/events/ingesting-usage). Idempotency keyed on `transaction_id` across REST/Kafka/S3 channels — with the caveat that the ClickHouse pipeline *overwrites* on identical `transaction_id + timestamp` (https://getlago.com/docs/guide/events/ingesting-usage).
- **Stripe Billing Meters:** idempotency via the `identifier` field (auto-generated if omitted); timestamps accepted within the past 35 calendar days and max 5 minutes future (https://docs.stripe.com/billing/subscriptions/usage-based/recording-usage-api).
- **OpenMeter:** ingestion in CloudEvents format (https://github.com/openmeterio/openmeter).

**Where UBB stands:** batch size (100) matches Metronome and Lago, below Orb's 500. The 34-day default backfill window matches Metronome's exactly and is tenant-tunable (0–60 days), versus Stripe's documented 35-day window (no configurability documented). The unbounded DB-constraint dedup is *stronger* than every competitor's TTL'd window. UBB's per-item-isolated batch (partial success, positionally aligned results) is a better failure contract than Lago's all-or-nothing batch. What every competitor has that UBB lacks: an asynchronous, decoupled ingest pipeline (Lago processes events async even in OSS; Metronome/Orb ingest at 100k–250k+ events/s — see §10) and a general-purpose `Idempotency-Key` *header* convention covering all POST mutations uniformly rather than per-endpoint body fields.

### Gaps

| Gap | Severity |
|---|---|
| Ingestion is synchronous-priced-in-transaction only: every event costs ~9–13 DB round trips before the HTTP response (see §10), and the batch endpoint runs items sequentially in one request — batch makes latency 100× worse, not throughput better (metering_endpoints.py:201-202). No async accept-then-price path exists. | major |
| `RecordUsageRequest.metadata` is an unbounded dict (schemas.py:29) and `usage_metrics` has no key-count bound (schemas.py:32) — see §2 for the storage consequence. | moderate |
| No general `Idempotency-Key` header convention for non-ingest POSTs; idempotency is per-endpoint body fields, and read-modify endpoints without a body key (e.g. config PUTs) rely on last-write-wins. | minor |


**Gap analysis — plain-English, verified against code at HEAD.**

**Synchronous-priced ingestion, no async path — major.** **Verified: Confirmed.** I read `metering_endpoints.py:201-202` (the batch endpoint is a plain Python list-comprehension running items one after another, in-request) and `usage_service.py:113-221` (every event runs full pricing, tier-ladder locking, the event insert, and an outbox insert inside one synchronous transaction before the HTTP response returns). There is no "accept now, price later" route anywhere in the metering app. **What's wrong:** UBB does all its expensive bookkeeping — roughly 9-13 database round trips per event — *before* it tells the caller "got it." Batching 100 events into one call doesn't help, because they still run sequentially: you get one slow response instead of 100 fast ones. Throughput is unchanged; latency gets worse. **Why we should care:** This is the single biggest competitiveness ceiling on the product. UBB's reasoned capacity is ~100-300 events/second per instance; Orb advertises its *standard* (non-streaming) API as "tens of millions of events/day" and headline 250,000/sec, Metronome 100,000/sec, and Lago processes events asynchronously even in its free open-source build (03, Orb https://www.withorb.com/pricing; Metronome https://docs.metronome.com/api-reference/usage/ingest-events; Lago https://getlago.com/docs/api-reference/events/batch). Any AI customer with real token volume will hit UBB's wall and a competitor's headline number side by side. **Options:** Build an accept-then-price pipeline: persist the raw event durably and return 202/200 immediately, then price in background workers keyed on the *same* idempotency keys so exactly-once still holds. This is exactly the architecture Lago and OpenMeter use (Kafka/ClickHouse-fed async engines, 03 https://github.com/getlago/lago/wiki/Using-Clickhouse-to-scale-an-events-engine). UBB doesn't need a new datastore on day one — a durable raw-event table plus the existing outbox/worker pattern would do. **Codebase effect:** This is *structural*, not additive, and touches the hottest path. Today callers get the priced result (balance, cost, hard-stop) back in the same response; an async path means that result arrives later, which changes the public API contract and forces decisions about how budget hard-stops fire when pricing is deferred. The idempotency guarantee must be preserved across the new boundary, and the tier-ladder locking would move into workers. Treat it as a major project with careful migration, not a quick win.

**Unbounded metadata / usage_metrics — moderate.** **Verified: Confirmed.** I read `schemas.py:25-44`: `metadata` is `dict = Field(default_factory=dict)` with no size cap, and `usage_metrics` is `Optional[dict[str, int]]` with only a non-negative-value validator (lines 35-44) and no key-count limit — in pointed contrast to `tags`, which `usage_service.py:77-94` caps hard at 50 keys, regex-checked keys, and 256-char values. **What's wrong:** Two fields on the busiest table in the system accept arbitrarily large blobs. A misbehaving or malicious SDK could attach megabytes of JSON to every event. **Why we should care:** This is where §1 and §2 meet: every oversized blob lands in `ubb_usage_event`, the table that never gets pruned and grows fastest (see the §2 partitioning gap). Bloated rows inflate storage cost, slow the ingest insert, and degrade the analytics queries that already run on the live database. It's a slow-burn cost-and-performance leak, not an outage — hence moderate. Orb, for comparison, simply forbids nested data: event properties "allow only primitives… arrays and objects are not permitted" (03 https://docs.withorb.com/quickstart/ingest), which sidesteps the whole problem. **Options:** Mirror the discipline UBB already wrote for `tags` — cap the serialized `metadata` size (e.g. ~16 KB) and bound the `usage_metrics` key count in the pydantic schema. Best practice (and Orb's choice) is to also constrain value types. The code to copy already exists three functions away. **Codebase effect:** Purely *additive* and low-risk for new data — a few validator lines at the edge, rejecting oversized payloads before they're stored. The one caution: any historical rows already storing large blobs are unaffected, and tightening the limit could reject payloads a sloppy existing integration is currently sending, so it warrants a heads-up to customers and a sensible (generous) initial limit.

**No general Idempotency-Key header — minor.** **Verified: Confirmed.** I read `schemas.py:140-199`: withdraw/debit/credit/grant each carry their *own* `idempotency_key` body field, and there is no shared `Idempotency-Key` HTTP header convention spanning all POST mutations. Read-modify config endpoints with no body key fall back to last-write-wins. **What's wrong:** UBB's exactly-once protection is real and strong, but it's applied unevenly — per-endpoint, in the request body, on the endpoints the engineers remembered to protect — rather than as one uniform header rule covering every mutating call. **Why we should care:** Low actual risk (the money-moving endpoints, which matter most, *are* covered), but it's a credibility and consistency issue for technical buyers who evaluate APIs. Both Orb and Metronome offer a standard `Idempotency-Key` header on all POSTs (Orb: 48h validity with an `Idempotent-Replayed: true` response header; Metronome: cached ≥24h, replays even after a 500 — 03 https://docs.withorb.com/api-reference/idempotency, https://docs.metronome.com/developer-resources/use-api/idempotency/). An evaluator comparing API references will see a convention UBB lacks. **Options:** Add a cross-cutting `Idempotency-Key` header handler (middleware or a django-ninja dependency) that any POST can opt into, replaying the stored original response on a repeat and 409-ing on same-key-different-payload — exactly the Orb/Metronome behaviour. The existing body-key endpoints can keep working underneath. **Codebase effect:** Additive — a new middleware layer plus a small response-cache store; it doesn't disturb the existing per-endpoint constraints, which remain the durable backstop. The main design question is where to persist the header-keyed responses and for how long, mirroring the competitors' 24-48h windows.

### Actions

1. Build the async ingest path (accept + persist raw, price in workers under the same idempotency keys) before any high-volume tenant — this is the same action as §10's throughput gap.
2. Bound `metadata` and `usage_metrics` in the pydantic schema (see §2 action).
3. Document the idempotency contract (unbounded window, replay-wins semantics) in the public API docs — it is genuinely stronger than Metronome/Orb's and currently invisible to evaluators.

*(A future-timestamp cap was initially listed as a gap here; fact-checking found it already exists — `_FUTURE_SKEW = 5 minutes` at usage_service.py:19 with `effective_at_in_future` raised at :54-58, matching Stripe's documented posture.)*

## 2. Database & storage

**Relevance: critical. Verdict: adequate.**

### What exists

- **Postgres-first config:** `DATABASES` via dj_database_url with `CONN_MAX_AGE=600` and `CONN_HEALTH_CHECKS=True` (ubb-platform/config/settings.py:81-87); psycopg[binary]>=3.2 + Django 6.0 (ubb-platform/requirements.txt:3-4); `USE_TZ=True`/UTC (settings.py:98-100); postgres:16 in docker-compose.yml:2-3 and CI services (.github/workflows/ci.yml:38-39).

  > **In plain terms.** UBB is built on PostgreSQL, the industry-standard heavy-duty database for systems that handle money, configured sensibly (reused connections, health checks, everything in UTC time) and pinned to a modern version that's the same in development, testing, and the Docker setup. This matters because a billing ledger needs a database with real transactional guarantees — when you move money, it either fully happens or fully doesn't, with no half-states. Analogy: choosing a bank vault with a proper audited locking mechanism rather than a filing cabinet. Business value: the foundation is the right, boring, trustworthy choice — no exotic bets. The serious caveat sits in a later bullet and a gap: the *configuration* silently allows falling back to a toy file-based database (SQLite) if mis-set, which undercuts this otherwise-solid foundation.

- **Money is BigInteger micros everywhere it is an amount:** `Wallet.balance_micros`, `WalletTransaction.amount_micros` + `balance_after_micros` (ubb-platform/apps/billing/wallets/models.py:26,46-47), CreditGrant granted/remaining/expired/voided micros (wallets/models.py:124-127), `UsageEvent.provider_cost_micros`/`billed_cost_micros` (ubb-platform/apps/metering/usage/models.py:30-31), invoice totals/residuals (ubb-platform/apps/billing/invoicing/models.py:67,92,96). DecimalField appears only for rates/percentages, never amounts.

  > **In plain terms.** Every monetary amount in UBB is stored as a whole number of "micros" (millionths of a currency unit) — never as a decimal or floating-point number. This is a genuine strength and exactly what serious billing systems do. Floating-point math infamously produces errors like 0.1 + 0.2 = 0.30000000004; storing money as integers eliminates that entire class of rounding bug. UBB applies this consistently — wallet balances, transactions, invoice totals, costs — and reserves decimals only for rates and percentages, which is correct. Analogy: counting in exact pennies rather than approximate dollars-and-cents. Business value: the money is always penny-exact and auditable; you will never have to explain to a customer why their balance is off by a fraction of a cent due to accumulated rounding. This is table stakes done right.

- **Constraints genuinely encode the money invariants:** partial-unique `uq_wallet_txn_idempotency` on (wallet, idempotency_key) WHERE key IS NOT NULL — *this constraint is the exactly-once guarantee for credits/debits* (wallets/models.py:60-64); `uq_usage_event_idempotency_v2` (usage/models.py:48-51); CheckConstraints bounding grant remaining/refunds (wallets/models.py:148-152,192-200); state-machine partial uniques for open periods, pending top-ups, invoice-per-period, sandbox shape (ubb-platform/apps/billing/tenant_billing/models.py:47-51; topups/models.py:59-63; invoicing/models.py:106-107; tenants/models.py:84-90).

  > **In plain terms.** The most important rules of the money system — "the same credit/debit can never apply twice," "a credit grant can't go negative," "only one billing period open at a time" — are enforced by the *database itself*, not just by application code that could have a bug. The audit calls out that one such constraint *is* the exactly-once guarantee for credits and debits. This matters because database constraints are the last line of defense: even if a future code change is wrong, the database physically refuses to record an impossible state. Analogy: a safe whose mechanism won't let you set a balance below zero no matter what buttons you press. Business value: the financial invariants are protected at the deepest, most reliable layer — this is unusually disciplined for a company at this stage and is worth bragging about to a technical buyer or auditor.

- **Soft-delete done carefully:** `SoftDeleteMixin` with undelete-only policy (ubb-platform/core/soft_delete.py:12-61); the Customer unique key is soft-delete-aware (customers/models.py:39-43); `lock_for_billing` restores a soft-deleted wallet rather than violating the OneToOne (ubb-platform/apps/billing/locking.py:26-38).

  > **In plain terms.** When records like customers or wallets are "deleted," UBB doesn't actually erase them — it marks them inactive but keeps the data, and handles the tricky edge cases (you can restore a wallet rather than break the rule that each customer has exactly one). This matters for a billing system because financial records must be retainable for audit, dispute, and compliance reasons; truly deleting a wallet that holds money would be reckless. Analogy: moving a file to a recoverable archive rather than shredding it. Business value: data is recoverable and the audit trail survives accidental or routine deletions, while the careful handling prevents subtle corruption (like a "deleted" record blocking the creation of its replacement). Note this protects against *application-level* deletion — it is not a substitute for database backups, which the gaps show are entirely absent.

- **Index discipline incl. opclass awareness:** composite indexes on UsageEvent (usage/models.py:53-60); the tags GIN index was deliberately swapped from `jsonb_path_ops` to `jsonb_ops` (because `has_key` compiles to `?`) via CREATE/DROP INDEX CONCURRENTLY with `atomic=False`, a real reverse migration, and an INVALID-index recovery runbook (usage/migrations/0022_swap_tags_gin_to_jsonb_ops.py:1-87), with a test asserting the old index is gone (apps/metering/tests/test_analytics_scale.py:303-318).

  > **In plain terms.** Indexes are the database's search shortcuts. UBB's team showed real expertise here: they noticed that a particular index type couldn't actually accelerate the "does this tag exist?" queries the product runs, and surgically swapped it for the right one — using the advanced, no-downtime technique (CONCURRENTLY) and even writing a recovery runbook for if the swap is interrupted, plus a test proving the old index is gone. Analogy: realizing the library's card catalog was sorted the wrong way for how people actually search, and re-sorting it without ever closing the library. Business value: this is a strong signal that the engineers understand database performance at a senior level, not just ORM defaults. The caveat (a separate gap): that same care was *not* applied uniformly — most other indexes were added the plain, lock-taking way, and the table is arguably over-indexed.

- **Migration discipline:** 104 migrations across 14 apps; CI enforces `makemigrations --check --dry-run` (.github/workflows/ci.yml:88-90); data migrations ship RunPython backfills with explicit reverses (tenants/0012; wallets/0004; usage/0018, 0020).

  > **In plain terms.** "Migrations" are the versioned, replayable scripts that evolve the database structure over time. UBB has 104 of them across the system, the data-changing ones include explicit "undo" instructions, and the CI pipeline automatically checks that the code and the database schema never drift out of sync. This matters because uncontrolled schema changes are a classic way to corrupt or lose data in production. Analogy: every change to the building's blueprints is logged, reversible, and checked against the actual building before anyone signs off. Business value: the database can be rebuilt deterministically and changes can be rolled back — solid engineering hygiene that reduces deployment risk. The caveat surfaced in the gaps: a few of these migrations rewrite entire large tables in one shot or row-by-row, which will be slow and lock-prone once real data volume exists.

- **Operational-table lifecycle is beat-scheduled:** outbox prune 30d/90d with failed never auto-deleted (ubb-platform/apps/platform/events/tasks.py:136-152); Stripe webhook events 90d/180d via PK-range batched deletes to avoid long locks (ubb-platform/apps/billing/stripe/tasks.py:11-43); delivery attempts 30d/90d (events/tasks_webhook_cleanup.py:13-29).

  > **In plain terms.** UBB automatically cleans up its own "scratch" tables — the event outbox, Stripe webhook logs, delivery attempts — on a schedule, deleting old processed records after 30-90 days while deliberately *never* auto-deleting failed ones (so failures stay visible for investigation). The Stripe cleanup even deletes in small batches to avoid locking the table. This matters because operational tables that grow forever eventually slow everything down or fill the disk. Analogy: a self-emptying recycling bin that keeps anything flagged as a problem. Business value: the system maintains itself rather than requiring manual housekeeping, and the "never auto-delete failures" rule means money-related problems don't quietly disappear. The honest caveat (a gap): this good batching discipline is applied inconsistently — the outbox cleanup itself uses one big unbatched delete, the very pattern the Stripe cleanup correctly avoids.

- **UsageEvent is immutable by code** (save raises on update, delete raises — usage/models.py:66-72) and is never pruned; no PARTITION anywhere in the apps.

  > **In plain terms.** Once a usage event is recorded, the code physically refuses to update or delete it — any attempt throws an error. Events are append-only and never pruned. This is the correct, auditable design for the system's source-of-truth records: the history of what a customer actually consumed should never be silently rewritten. Analogy: a ledger written in permanent ink where you can add corrections but never erase a line. Business value: total trust in the usage history — what was billed can always be reconstructed and defended in a dispute. But this strength carries the section's biggest structural risk, flagged loudly in the gaps: because events are never deleted and the table is never partitioned, it grows without bound — billions of rows a year at scale — and partitioning, the standard fix, cannot easily be retrofitted later. Immutability is right; the missing growth-management story around it is the problem.

- **JSONB bounded where it was designed:** tags validated at the choke point (≤50 keys, key regex, string values ≤256 chars — usage_service.py:77-94); `pricing_provenance` is platform-constructed and bounded; `line_snapshot` is frozen at first push claim (invoicing/models.py:88-91).

  > **In plain terms.** UBB stores some flexible, schema-less data (tags, pricing provenance, invoice line snapshots) in JSON columns — and where the team *designed* these deliberately, they bounded them tightly (tags are capped at 50 keys with validated names and value lengths; provenance is system-generated; invoice snapshots are frozen once). This matters because unbounded flexible fields are a common way for a database to bloat and slow down. Analogy: a "notes" field that has a sensible character limit rather than letting anyone paste a whole book. Business value: where the engineers were paying attention, they got it right and showed they know the hazard. The pointed caveat — which becomes a gap — is the phrase "where it was designed": the `metadata` and `usage_metrics` JSON fields on the very same hot table were *not* bounded, so the discipline exists but wasn't applied everywhere it matters.

- **Backup/PITR: nothing in the repo** — repo-wide grep for pg_dump/PITR/pgbouncer/wal-g/pgbackrest/backup returns zero matches; the only deployment artifact is a docker-compose with one named volume (docker-compose.yml:12-13,22-23). **Read replicas/warehouse export: nothing** — no DATABASE_ROUTERS, no CDC, no S3/Snowflake/BigQuery sync.

  > **In plain terms.** This bullet is the alarming one, stated plainly by the audit: there is no database backup, no point-in-time recovery, no replication, and no export to a data warehouse anywhere in the codebase — the only deployment artifact is a Docker file with a single local storage volume. This matters more for UBB than for almost any other kind of software, because the wallet ledger *is* customer money. The clever self-healing jobs UBB has can repair drift against Stripe, but they cannot resurrect a PostgreSQL database that has been lost, corrupted, or deleted. Analogy: a bank with immaculate internal bookkeeping but no vault, no insurance, and no second copy of the records — one fire and the money's history is gone. Business value: this is an existential operational gap, not a feature gap; the audit escalates it to a blocker in the availability section. It must be closed before real customer money flows.


### Competitor evidence

- **Lago's *default* self-host stack ships partition-aware Postgres:** `db = getlago/postgres-partman:15.0-alpine` in the canonical docker-compose (https://github.com/getlago/lago/blob/main/docker-compose.yml), and its at-scale events engine is ClickHouse fed by Kafka, with Postgres retained for transactional data (https://github.com/getlago/lago/wiki/Using-Clickhouse-to-scale-an-events-engine).
- **OpenMeter** runs Postgres for billing plus ClickHouse for real-time usage aggregation and Kafka for event streaming as its baseline architecture (https://github.com/openmeterio/openmeter).
- **Orb** commits to never overwriting or permanently deleting ingested usage data — backfilled originals are archived and remain queryable (https://docs.withorb.com/guides/events-and-metrics/reporting-errors). Metronome's status page lists "Data Export" as a first-class product component (https://status.metronome.com).

The pattern: every competitor treats the event store as a partitioned/append-only, analytically exportable stream, separate in kind from the transactional store. UBB stores events in one monolithic, ever-growing OLTP heap.

**Gap analysis — plain-English, verified against code at HEAD.**

**Unbounded UsageEvent growth, no partitioning — major.** **Verified: Confirmed.** I read `usage/models.py`: the table has eight single-column indexed fields (lines 17-43), four composite indexes plus a GIN tags index, a unique constraint, and FK indexes, and the model physically blocks deletes/prunes (lines 66-72). A repo-wide search for `PARTITION`/`partman` found matches only in docs and review artifacts, never in the app — so there is genuinely no partitioning. **What's wrong:** The fastest-growing, never-pruned table lives as one giant monolithic heap. At 100 events/sec that's billions of rows and multiple terabytes per year, all in one structure. **Why we should care:** As the heap grows, inserts (the ingest hot path), index maintenance, and the analytics queries all degrade together, and routine maintenance (vacuum, reindex) gets slower and more disruptive. The killer detail: partitioning is very hard to retrofit onto a huge existing table — it effectively requires rewriting it — so the cost of *not* doing this rises every day data accrues. Every serious competitor treats the event store as a partitioned/append-only stream separate from transactional data; Lago's default open-source stack literally ships `postgres-partman` (03 https://github.com/getlago/lago/blob/main/docker-compose.yml). **Options:** Adopt monthly range partitioning on `ubb_usage_event` keyed on `effective_at` (declarative partitioning + pg_partman), and detach-and-archive closed partitions past the reconcile horizon to cheap object storage as Parquet — *before* meaningful data accrues. **Codebase effect:** *Structural and time-sensitive.* It changes the physical table, touches the ingest insert path, and interacts with the immutability rule and the analytics queries. Doing it now (near-empty table) is a manageable migration; doing it after the table is huge is a major outage-risk project. This is the clearest "do it while it's cheap" item in the audit.

**No backup / PITR posture — major (blocker in §12).** **Verified: Confirmed.** Repo-wide search for `pg_dump`/`PITR`/`pgbouncer`/`wal-g`/`pgbackrest`/`partman` returned zero matches in application or deployment code; `docker-compose.yml:12-13,22-23` defines exactly one local named volume and nothing else. **What's wrong:** There is no way to recover the database if it is lost or corrupted. The reconcile jobs repair drift versus Stripe but cannot rebuild a destroyed PostgreSQL. **Why we should care:** The wallet ledger is customer money and the usage events are the billing source of truth. Losing the database means losing the financial record — unrecoverable, legally and commercially catastrophic, and an instant deal-breaker the moment any real customer does diligence. This is why the audit rightly escalates it to a blocker in availability. Competitors treat durability as a product surface — Metronome publishes a "Data Export" component and Orb publishes per-component uptime and never-deletes-data guarantees (03 https://status.metronome.com, https://docs.withorb.com/guides/events-and-metrics/reporting-errors). **Options:** Run on managed PostgreSQL with point-in-time recovery (RDS/Cloud SQL/Aurora), or self-manage with wal-g/pgbackrest plus WAL archiving. Critically, write *and rehearse* a restore runbook whose final step re-runs `reconcile_topups_with_stripe`/`reconcile_usage_drawdowns` so a restored database re-converges with Stripe. **Codebase effect:** Mostly *additive at the infrastructure layer* — it's an ops/deployment change, not a code rewrite. The one code-adjacent piece is making the documented restore-then-reconcile procedure real and tested, which leans on machinery UBB already has. Low blast radius on the codebase, very high business priority.

**No warehouse export / read-replica routing — major.** **Verified: Confirmed.** I read `metering_endpoints.py:395-431` (the tag-grouping analytics aggregations — `Count`, `Sum`, group-by over the events table) and confirmed via search there are no `DATABASE_ROUTERS` and no CDC/S3/warehouse sync anywhere. So analytics runs on the same primary database that serves the ingest hot path. **What's wrong:** Heavy analytical scans and the latency-sensitive ingest writes compete for the same database. **Why we should care:** A customer running a big cost-breakdown query can slow down — or, combined with the no-throttle gaps, contend with — live usage ingestion, and there's no separate analytical store to absorb that load. As data grows this gets worse. It also means UBB can't easily offer customers their own data export, which competitors treat as a first-class product (Metronome's "Data Export" component; Orb keeps events queryable and exportable — 03 https://status.metronome.com, https://docs.withorb.com/guides/events-and-metrics/reporting-errors), and OpenMeter/Lago run a dedicated analytical engine (ClickHouse) for exactly this (03 https://github.com/openmeterio/openmeter). **Options:** Near-term, route the read-only analytics endpoints to a PostgreSQL read replica via a database router so they never touch the write primary. Longer-term, ship periodic Parquet export of events/ledger/invoices keyed by (tenant, day) as a v1 warehouse-sync feature, then CDC into a columnar store. **Codebase effect:** The read-replica router is *moderately additive* — a routing layer plus per-query hints, with care that it only catches genuinely read-only paths. The warehouse export is *additive* (a new background job and storage target). Neither rewrites the core flow, but the router must be tested to avoid accidentally sending a write to a replica.

**Unbounded metadata / usage_metrics on the hottest table — moderate.** **Verified: Confirmed.** Same finding as §1: `schemas.py:29` (`metadata: dict`, no size cap) and `:32` (`usage_metrics: dict[str,int]`, only a value-sign check, no key bound), versus the tight `validate_tags` cap at `usage_service.py:77-94`. These blobs are stored directly on `ubb_usage_event` (`models.py:20,33`). **What's wrong:** Two unbounded JSON fields sit on the table that grows fastest and is never pruned. **Why we should care:** This compounds the partitioning gap above — a misbehaving SDK storing megabytes per event accelerates the terabyte-scale growth, slows the ingest insert, and bloats the analytics scans, all on the most expensive table to fix later. It's a slow cost-and-performance leak rather than an outage, hence moderate. Orb avoids it entirely by forbidding non-primitive event properties (03 https://docs.withorb.com/quickstart/ingest). **Options:** Reuse the discipline already written for tags — cap serialized `metadata` size (~16 KB) and bound `usage_metrics` key count in the schema; optionally constrain value types as Orb does. **Codebase effect:** *Additive* — a few edge-validation lines, no schema migration, no hot-path restructuring. Caution only that a generous initial limit and customer notice are wise, since an existing sloppy integration could be sending oversized payloads today.

**SQLite silent fallback — moderate.** **Verified: Confirmed.** I read `settings.py:81-87`: `DATABASES` uses `dj_database_url.config(default="sqlite:///db.sqlite3")`, so an unset `DATABASE_URL` silently yields SQLite. The only fail-fast in that file is for `ALLOWED_HOSTS` (line 18-20), *not* for the database. And the GIN index migration is vendor-guarded (`if connection.vendor != "postgresql": return`, `0022:50`), so on SQLite it's a silent no-op. **What's wrong:** A misconfigured production deploy (forgotten env var) boots happily, writing billing data into a local throwaway file, with PostgreSQL-only indexes silently skipped — no crash, no warning. **Why we should care:** This is a quiet-data-loss trap. A worker that writes wallet transactions into an ephemeral SQLite file means money records vanish when the container restarts, and the operator gets no signal until the damage is done. It's gated on a config mistake (hence moderate, not blocker), but the failure mode is severe and invisible. Best practice is fail-loud configuration: refuse to start in a misconfigured state. **Options:** Raise `ImproperlyConfigured` when `DATABASE_URL` is unset and `DEBUG=False` — exactly mirroring the `ALLOWED_HOSTS` pattern already three lines away in the same file. **Codebase effect:** *Purely additive* — a handful of lines in settings, zero schema or hot-path impact. The only caution is ensuring local development and the test suite (which legitimately use SQLite or a local URL) still satisfy the guard, which is trivial since the guard only bites when `DEBUG=False`.

**Index over-provisioning on the insert hot path — moderate.** **Verified: Confirmed.** I read `usage/models.py:17-60`: `idempotency_key` carries `db_index=True` (line 18) *and* is the third column of the `uq_usage_event_idempotency_v2` unique constraint (lines 48-51) — so a standalone index plus the unique index both cover it. The single-column `db_index=True` on `product_id`/`service_id`/`agent_id` (lines 27-29) overlap the `idx_usage_attribution` composite (lines 56-57) that leads with those same columns. **What's wrong:** The busiest write table carries roughly 18 indexes, several provably redundant. **Why we should care:** Every index must be updated on every insert, so redundant indexes directly tax the ingest hot path (the exact path §1/§10 already flag as the throughput ceiling) and waste storage — and the waste scales with the unbounded row growth above. Moderate because it's a performance/cost drag, not a correctness bug. **Options:** After a load test to confirm which indexes actually get used (`idx_scan=0`), drop the redundant ones using the established no-downtime CONCURRENTLY pattern the team already wrote for migration 0022. Best practice is to keep the leading-column composite and drop the standalone single-column indexes it subsumes, and drop the standalone idempotency_key index the unique constraint already covers. **Codebase effect:** *Additive/subtractive at the schema layer via migrations* — dropping indexes is lower-risk than adding partitioning, but it must use CONCURRENTLY + `atomic=False` to avoid locking the live table, and should be validated against real query plans first so a still-needed index isn't removed.

**Unbatched large-table backfills — minor.** **Verified: Confirmed.** I read the cited migrations: `usage/0020` does one full-table `.update(billing_owner_id=F("customer_id"))` inside an atomic migration; `usage/0018` does two full-table `.update()` calls atomically; `wallets/0004` loops row-by-row with `.iterator()` + per-row `.save()`; and `events/tasks.py:142-152` (cleanup_outbox) issues one unbatched `.delete()` — while `stripe/tasks.py:36-42` correctly batches by PK range. **What's wrong:** Several data migrations and one cleanup job rewrite or delete across an entire large table in a single statement (or one row at a time), rather than in bounded batches. **Why we should care:** On an empty or small table this is harmless, which is why it's minor. But once `ubb_usage_event` is large, a full-table UPDATE in one transaction takes a long lock and bloats the write-ahead log, and a row-by-row loop is simply slow — either can stall ingestion during a deploy. The team clearly knows the right pattern (the Stripe cleanup proves it); the discipline is just inconsistent. **Options:** Batch all backfills and cleanups by primary-key range, reusing the `_batched_delete` helper pattern already in `stripe/tasks.py`. Make batched-by-PK the standing rule for any operation touching event-scale tables. **Codebase effect:** *Additive going forward* — new migrations/jobs adopt the pattern. The already-applied migrations (0018/0020/0004) are historical and shouldn't be rewritten; the live fix worth doing now is converting `cleanup_outbox` to batched deletes, which is a small, contained change to one task.

**CONCURRENTLY inconsistency — minor.** **Verified: Confirmed.** Only `usage/0022` sets `atomic = False` and uses `CREATE/DROP INDEX CONCURRENTLY` (lines 49-87); other index additions on the usage table are plain transactional `CREATE INDEX`, which take a lock that blocks writes for the duration of the build. **What's wrong:** The no-downtime index-build technique was applied once but isn't the standing convention for the event-scale table. **Why we should care:** A plain `CREATE INDEX` on a large `ubb_usage_event` locks the table against writes — i.e. blocks ingestion — for as long as the build runs, turning a routine index change into an ingestion outage. On today's small table it's invisible; at scale it's a self-inflicted incident. Minor only because no such blocking index add is pending right now. **Options:** Make `CONCURRENTLY` + `atomic = False` the mandatory rule for any index change on event-scale tables (a one-line code-review/checklist standard), exactly as the team already did once in 0022, which even includes a recovery runbook to copy. **Codebase effect:** *Purely a convention/process change* — no change to existing schema, applied only to future migrations. The only nuance is that CONCURRENTLY migrations can't run inside Django's default transaction wrapper, so each such migration needs `atomic = False` and the documented INVALID-index recovery step, both of which the existing 0022 already models.

### Actions

1. Adopt monthly range partitioning on ubb_usage_event (declarative + pg_partman, keyed on effective_at) **before meaningful data accrues**; detach-and-archive closed partitions past the reconcile horizon to object storage as Parquet.
2. Run on managed Postgres with PITR (or wal-g/pgbackrest + WAL archiving); write and rehearse a restore runbook that ends with re-running `reconcile_topups_with_stripe`/`reconcile_usage_drawdowns` to converge with Stripe.
3. Ship periodic Parquet export of UsageEvent (+ ledger + invoices) keyed by (tenant, day) as the v1 warehouse-sync feature; later CDC + a read-replica router for analytics endpoints.
4. Cap `metadata` serialized size (~16 KB) and bound `usage_metrics` keys in the schema, mirroring `validate_tags`.
5. Raise ImproperlyConfigured when `DATABASE_URL` is unset and `DEBUG=False`.
6. After a load test, drop `idx_scan=0` indexes via the established CONCURRENTLY pattern; make CONCURRENTLY + `atomic=False` the standing rule for event-scale tables; batch all backfills/cleanups by PK range.

## 3. Auth & permissions

**Relevance: critical. Verdict: adequate.**

### What exists

- **API key auth:** django-ninja HttpBearer (ubb-platform/core/auth.py:9-20); `TenantApiKey.verify_key` SHA-256-hashes the bearer token and does a per-request unique-indexed DB lookup with `select_related('tenant')`, filtered on key and tenant both active (ubb-platform/apps/platform/tenants/models.py:177-194). No caching ⇒ revocation/rotation effective on the very next request.

  > **In plain terms.** This is the front door lock on the whole product: every API request must present a bearer key, which we hash and look up in the database before anything happens. The brag-worthy part is what we deliberately did NOT do — we don't cache that lookup. Every single request re-checks the database, so the instant a customer revokes or rotates a key (say, an employee leaves or a key leaks to GitHub), the very next call is rejected. Many systems cache keys for speed and accept a window of minutes where a "revoked" key still works; we close that window entirely. The hidden caveat is the cost of that choice: one database read per request is a real load tax at high volume, so this is a correctness-over-raw-speed trade we'll want to watch as traffic grows.

- **Live/test prefixes are enforced, not cosmetic:** `ubb_live_`/`ubb_test_` + `secrets.token_urlsafe(32)` (256-bit entropy) at mint (models.py:166-167); verify re-asserts prefix mode == `tenant.is_sandbox` (models.py:192-193), so a crafted test key can never resolve onto a live tenant.

  > **In plain terms.** Keys come in two flavours — live (`ubb_live_`) and test (`ubb_test_`) — like a real credit card versus a practice card that spends fake money. The strength here is that the prefix isn't just a label for humans to read; at verification time the system re-checks that a test key is actually attached to a sandbox account and a live key to a live account. So even if someone hand-crafted a fake "test" key in the database, it could never reach into a live, real-money account. The keys themselves carry 256-bit randomness (think a password longer than anyone could ever guess by brute force). The business payoff: customers can hand test keys to contractors, demos, or CI pipelines without any path for those keys to touch production billing. The caveat is minor — this safety rests on the prefix being preserved in the stored key fragment, which the code guarantees.

- **Self-serve key lifecycle:** mint returns the raw key once; rotate is atomic mint-successor+deactivate-old; revoke refuses to kill the tenant's last active key (409) under `select_for_update` (ubb-platform/api/v1/tenant_endpoints.py:183-282).

  > **In plain terms.** Customers manage their own keys without filing a support ticket — they mint, rotate, and revoke through the API. Two details make this safe rather than dangerous. First, the raw key is shown exactly once at creation and never again (we only keep a hash), so a leaked screenshot of our database can't reveal anyone's key — this matches how Stripe and GitHub hand out secrets. Second, rotation is atomic (the new key is born and the old one retired in one step, no gap), and revoke refuses to kill a tenant's last active key, returning a 409 error instead. That last guard is the brag: it prevents a customer from accidentally locking themselves out of the API entirely, which would otherwise require us to manually rescue them. The caveat: self-serve rotation exists for API keys but, per the gaps below, not yet for the widget secret.

- **Widget auth:** HS256 JWT per-tenant secret (`token_urlsafe(48)`), 15-min default expiry, safe two-step decode, issuer pinning; `rotate_widget_secret` invalidates all outstanding tokens (ubb-platform/core/widget_auth.py; tenants/models.py:117,127-130).

  > **In plain terms.** The customer-facing widget (the embeddable usage/billing UI a tenant drops into their own app) uses short-lived signed tokens rather than the powerful API key. Each tenant has its own signing secret, and tokens expire in 15 minutes by default — so even if a browser token is stolen, it's worthless within minutes, like a hotel key card that deactivates at checkout. Tokens are also pinned to the issuing tenant, so one tenant's token can't be replayed against another. The strength: this keeps the all-powerful API key on the server and never ships it to the browser, which is exactly the separation a billing product needs. The hidden caveat is operational — the secret that signs these tokens can only be rotated through admin/database access today (see the gaps), so a tenant who suspects their widget secret leaked can't fix it themselves yet.

- **No unauthenticated business endpoint:** all 12 NinjaAPI routers instantiate with `auth=ApiKeyAuth()` (or WidgetJWTAuth); only Stripe webhooks and the Connect OAuth callback are key-less, gated by signature/state nonce (api/v1/webhooks.py:43-52; connect_endpoints.py:45-69).

  > **In plain terms.** There is no "open" business endpoint — every one of the 12 API surfaces demands either an API key or a widget token before it will respond. The only doors without a key are the Stripe webhook and the Stripe Connect OAuth callback, and those are not actually unprotected: the webhook verifies Stripe's cryptographic signature, and the OAuth callback checks a one-time state nonce we issued. In plain business terms, this means a stranger on the internet cannot read a customer's usage, move money, or even enumerate what exists without first holding a valid credential — and the two "public" endpoints can only be driven by Stripe itself, not by an attacker. This is the kind of blanket default-deny posture an auditor or enterprise security review wants to see. The caveat is the flip side of the scopes gap below: authenticated does not yet mean least-privilege.

- `last_used_at` buffered in Redis and flushed by beat every 5 min (core/auth.py:19; core/tasks.py:12-31).

  > **In plain terms.** We track when each API key was last used (useful for spotting dormant or compromised keys), but we don't write that timestamp to the database on every single request — that would mean a database write on every call just to record "you were here." Instead we stash it in Redis (fast in-memory store) and flush it to the database every five minutes via a background job. The business value is a clean audit signal — "this key hasn't been touched in 90 days, probably safe to revoke" — bought cheaply. The hidden caveat: the last-used time can lag reality by up to five minutes, and if Redis is wiped between flushes a few minutes of "last used" data evaporates. That's an acceptable trade for a convenience signal, but it means last_used_at should never be treated as a precise security-forensics timestamp.


### Competitor evidence

|---|---|

**Gap analysis — plain-English, verified against code at HEAD.**

**No authorization granularity below a tenant-wide key — major.** **Verified:** Confirmed. `TenantApiKey` (tenants/models.py:133-141) carries only `tenant`, `key_prefix`, `key_hash`, `label`, `is_active`, `last_used_at` — there is no scope, role, or read-only field. `ApiKeyAuth.authenticate` (core/auth.py:9-20) sets `request.tenant` and a sandbox flag and stops; it never consults a permission. `ProductAccess` (core/auth.py:23-34) checks `self.required_product in request.tenant.products` — that is the *tenant's* product entitlement, not the key's authority, so it cannot make one key weaker than another. The doc's claim is accurate, not overstated. **What's wrong:** Every key a tenant holds is a master key. There is no way to mint a key that can only ingest events, or only read, or that is barred from moving money. **Why we should care:** A single leaked key — pasted in CI logs, a laptop, a third-party tool — grants withdraw, refund, grant-credit, AND key rotation. For a billing product that literally moves customer funds, the blast radius of one leak is "everything," which is exactly what enterprise security reviews probe and what raises the cost of incident response. **Options:** Add a `scopes` field (or a `RestrictedKey` model) enforced inside `ApiKeyAuth`, shipping at least an ingest-only and a read-only class, and binding withdraw/refund/grant to an explicit write-money scope. Metronome already scopes tokens by access level (read-only), environment, and endpoint — though scope changes there require contacting a rep rather than self-serve (https://docs.metronome.com/api-reference/authentication). Orb and Lago document no fine-grained scopes (https://docs.withorb.com/api-reference; https://getlago.com/docs/api-reference/intro), so shipping self-serve scopes would put UBB ahead of two of three named competitors and at parity with Metronome on capability while beating it on self-serve. **Codebase effect:** Largely additive — a nullable scopes column plus an enforcement branch in one auth class; existing keys default to full scope so nothing breaks. The structural care item is choosing the scope vocabulary once (renaming scopes later is a migration on live credentials), and threading the scope check into every money-moving endpoint so the gate is uniform rather than per-handler.

**Unsalted single-round SHA-256 key hashing — minor.** **Verified:** Confirmed and correctly characterised. `create_key` computes `hashlib.sha256(raw_key.encode()).hexdigest()` (models.py:168) and `verify_key` recomputes the identical bare digest for lookup (models.py:180). There is no salt, no pepper, no HMAC, no key-stretching. The doc's nuance is right: because the key itself carries 256-bit entropy from `secrets.token_urlsafe(32)`, plain SHA-256 is genuinely fine against brute force — you cannot guess a 256-bit secret no matter how fast the hash. **What's wrong:** The missing property is a *pepper boundary*, not stretching. The hash is computed purely from data sitting in the database, so anyone who steals a database dump can hash a separately-leaked candidate key offline and confirm it matches a stored row — the database alone is enough to validate a key. **Why we should care:** It chiefly weakens our position in a compound-breach scenario (DB dump plus a key surfacing elsewhere) and is the kind of finding a SOC 2 or pen-test reviewer flags even when low-risk. The day-to-day risk is low because of the entropy; the reputational/audit risk is the real cost for a billing vendor. **Options:** Switch to `HMAC-SHA256` keyed by a `SECRET_KEY`-derived pepper held in config/secret manager, outside the database — OWASP's password-storage guidance recommends exactly this post-hash HMAC-with-server-side-secret pattern, with the pepper "stored separately from the password database" (https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html, fetched 2026-06-16). With the pepper absent from a stolen DB, offline validation becomes impossible. **Codebase effect:** Structural in one narrow way despite being a small diff — the hash function changes, so *every existing key_hash is invalidated* unless you migrate. The clean path is to namespace the new scheme (store an algorithm tag) and verify against both during a rotation window, or force a one-time rotate. Be cautious: a naive swap silently 401s every live key on deploy. No UX change once migrated.

**No API to fetch or rotate widget_secret — minor.** **Verified:** Confirmed. `rotate_widget_secret` exists only as an ORM method on the Tenant model (tenants/models.py:127-130), and the secret is auto-minted on first save (models.py:116-117). A grep of `ubb-platform/api` for `widget_secret` returns only test files exercising the JWT helper — no endpoint reads, returns, or rotates it. By contrast, API-key mint/rotate/revoke are fully exposed (tenant_endpoints.py:183-282), so the gap is specifically the widget secret being a second-class citizen. **What's wrong:** A tenant cannot retrieve their widget secret to configure their integration, nor rotate it themselves if it leaks — both require us to act via admin/shell. **Why we should care:** The widget secret signs the tokens that drive the embedded customer UI; if it leaks (it lives in the tenant's backend), the only remediation today is a support ticket to us. That is slow exactly when speed matters — during a live leak — and it quietly contradicts the otherwise strong self-serve key story above, which is a credibility ding in a security review. **Options:** Add an authenticated endpoint to fetch-once and to rotate the widget secret, mirroring the existing API-key lifecycle endpoints (the pattern, transaction shape, and event-emit are already in tenant_endpoints.py). No competitor fact in doc 03 speaks to widget-secret self-rotation specifically; best practice is that any signing secret a customer can leak should be self-rotatable without vendor intervention, the same principle UBB already applied to API keys. **Codebase effect:** Purely additive — one new router handler reusing `rotate_widget_secret`, plus an event for the audit trail; no schema change and no impact on existing tokens beyond the intended invalidation on rotate. The only care item is treating the fetch path like the API-key mint (authenticated, rate-limited) since it returns a live signing secret.

### Actions

1. Add a scopes field (or RestrictedKey model) enforced in ApiKeyAuth; ship ingest-only and read-only key classes at minimum; bind withdraw/refund/grant to an explicit write-money scope.
2. Switch key hashing to HMAC-SHA256 with a SECRET_KEY-derived pepper (no UX change).
3. Add an authenticated fetch-once/rotate endpoint for the widget secret.

## 5. Cloud & compute

**Relevance: high. Verdict: weak.**

### What exists

- **The process topology is fully derivable from code:** 1 web (WSGI) + ≥1 Celery worker consuming 8 explicitly declared queues (ubb-platform/config/settings.py:134-143) + exactly one beat driving 25 schedule entries (settings.py:147-254). Every task decorator pins `queue=` explicitly (verified by grep); a bare `celery worker` consumes all 8 because CELERY_TASK_QUEUES is set.

  > **In plain terms.** This is praise, not a feature list. The way the product runs in production — one web server, at least one background "worker" handling eight named job lines, and exactly one "beat" clock firing 25 scheduled jobs — isn't documented in a runbook anywhere, but an engineer can reconstruct all of it just by reading the code, because every job declares which line it belongs to. Think of it as a factory with no posted floor plan, but every machine is clearly labelled so you could draw the map yourself. That is genuinely good discipline: nothing is hidden or ambiguous. The caveat is the flip side — "derivable from code" means there is no operator-facing document, so the knowledge lives in one developer's head and in source files, not in a procedure anyone on call could follow at 3am.

- **Celery safety defaults are configured, not just claimed:** acks_late, reject_on_worker_lost, prefetch 1, 600s/300s time limits, max-tasks-per-child 1000 (settings.py:124-130) — a worker crash means redelivery, not loss.

  > **In plain terms.** The background-job system is configured so that if a worker machine dies mid-task, the job is redelivered and re-run rather than silently lost. "Acks late" means a job is only marked done after it actually finishes; "reject on worker lost" hands a crashed job back to the queue; processing one job at a time avoids hoarding; and hard time limits kill anything that hangs. For a billing system this matters a lot: these are the jobs that move money, send invoices, and reconcile against Stripe — losing one could mean a customer is never charged or never credited. The strength is that these protections are actually set in the config, not merely assumed. The honest caveat: redelivery guarantees the job runs again, so every such job must be safe to run twice — which the wider system does handle via database-level idempotency, but it is a standing requirement.

- **Health endpoints do real checks:** GET /api/v1/health liveness stub; GET /api/v1/ready runs `connection.ensure_connection()` + raw Redis ping, 503 with per-check detail on failure (ubb-platform/api/v1/endpoints.py:11-38, mounted at config/urls.py:40).

  > **In plain terms.** The app exposes two status URLs. "Health" is a simple "am I alive" heartbeat. "Ready" is the meaningful one: it actually opens a database connection and pings Redis, and returns a failure code (503) with a per-check breakdown if either is down. This is exactly what a load balancer or container platform needs to decide whether to send traffic to an instance or pull it out of rotation — the difference between "the process is running" and "the process can actually do work." The strength is that it tests real dependencies, not a hardcoded "ok." The caveat, picked up in the gaps: it checks the database and Redis but not whether the background workers or the beat clock are alive, so a fully dead job-processing tier would still report "ready" — the request path looks healthy while the money-moving machinery has silently stopped.

- **The web tier is stateless:** no FileField/MEDIA, JSON-to-console logging only, DB sessions used only by admin, contextvar request context (core/middleware.py:19-36), Celery correlation-id propagation via signals (config/celery.py:14-31).

  > **In plain terms.** "Stateless" means the web servers keep nothing important in their own local memory or disk between requests — no uploaded files, no in-process sessions, no per-server scratch data. Everything durable lives in the shared database and Redis. The business value is that you can run many identical copies of the web server behind a load balancer and it doesn't matter which one handles your request; you can add or kill instances freely, which is the foundation of both scaling and zero-downtime deploys. They've also wired request-tracing IDs through to the background jobs, so a single customer action can be followed across the whole system in the logs. The caveat is that being safe to replicate is not the same as actually replicating — as the deployment sections note, there is currently nothing set up to run more than one instance.

- One Redis serves cache + broker + result backend off one URL (settings.py:107-118); conn_max_age=600 with no pooler anywhere.

  > **In plain terms.** A single Redis server is doing three jobs at once: it's the fast-access cache, the queue that hands work to background workers (the "broker"), and the store for job results. This is simple and cheap to run, which is fine at small scale. But it concentrates risk: that one Redis is now load-bearing for caching, job dispatch, and rate limiting all together, and the three uses actually want conflicting settings (a cache should evict old data when full; a job queue must never throw work away). The note that there's "no pooler" means database connections aren't pooled by a dedicated layer, so connection limits become an early bottleneck when scaling. The honest read: a reasonable starting choice that becomes a single point of failure and a tuning conflict as the system grows, and is flagged again in the caching and availability sections.

## 7. Security & RLS

**Relevance: critical. Verdict: adequate.**

### What exists

- **Tenant isolation is systematic application-level FK scoping:** every business query filters on `request.auth.tenant` (e.g. metering_endpoints.py:50; tenant_endpoints.py:49,89); me_api scopes to the widget customer; pooled-seat leakage is explicitly blocked (me_endpoints.py:334-339,420-427). No tenant-crossing raw SQL in production paths (cursor()/raw() appear only in tests/scripts).

  > **In plain terms.** Every customer (tenant) on the platform shares the same database, so the single most important safety rule is that one tenant can never see another's money or usage data. UBB enforces this the disciplined way: every query that touches business data is filtered by "which tenant is this API key for?" before it runs — like a bank teller who physically cannot pull up an account that isn't yours. The code even goes a step further and blocks a subtle leak where pooled "seats" (sub-accounts) under one business could otherwise peek at siblings. The strength: it is consistent and applied everywhere, with no tenant-crossing raw SQL in production. The caveat: this is enforced purely in application code — there is no database-level backstop, so the discipline has to hold in every future endpoint (the gap section returns to this).

- **Inbound Stripe webhooks:** `stripe.Webhook.construct_event` signature verification with the built-in 300s tolerance (api/v1/webhooks.py:71; subscriptions endpoints.py:198), plus DB-row dedup with CAS-guarded retry (webhooks.py:81-126). Live/test isolation: separate test endpoint requiring its own secret, `reject_for_mode()`, and `livemode_filter()` binding event.account lookups to `is_sandbox` (invoice_routing.py:22-54).

  > **In plain terms.** When Stripe tells UBB "this customer just paid," UBB has to be certain the message genuinely came from Stripe and isn't a forgery or a stale replay — because these messages move money. UBB checks Stripe's cryptographic signature on every incoming message and rejects anything older than five minutes, the way a signed, dated courier receipt proves both authenticity and freshness. It also records each message so the same one can't be processed twice (a "we already handled this" ledger). On top of that, it keeps live (real-money) and test (sandbox) traffic strictly separated, so a test event can never accidentally touch a real customer's account. This is exactly the posture a billing vendor should have. The one nuance worth noting: this rigor applies to messages coming in from Stripe; UBB's own outgoing messages have a separate, weaker legacy path covered in the gaps.

- **Outbound webhook SSRF defense is real and the IP pinning actually pins:** validate_webhook_url rejects non-https/localhost/private/loopback/link-local/reserved and returns the first validated IP; `_PinnedIPBackend.connect_tcp` swaps the hostname for that IP while TLS SNI/cert verification keeps the original hostname; URL re-validated at delivery time, closing the DNS-rebinding window (core/url_validation.py; events/webhooks.py:19-68,171-181).

  > **In plain terms.** When UBB sends a notification to a customer's server, an attacker could try to trick UBB into instead calling an internal address — the cloud provider's secret-credentials endpoint, say — by pointing the destination at a private address. This class of attack (SSRF) is a top way cloud systems get breached. UBB defends properly: it refuses non-HTTPS and any private/internal address, then "pins" the connection to the exact safe address it already verified, so the destination can't quietly switch to an internal target in the split second between the check and the call (the classic "DNS rebinding" trick). Think of it as confirming the courier's address, then driving them there yourself rather than letting them re-read the address en route. Genuinely strong. The caveat in the gaps: one shared-address range (carrier-grade NAT, 100.64/10) slips through the current filter.

- **Outbound signing has a replay-bounded v2:** `compute_signature_v2` over `{ts}.{body}` as X-UBB-Signature-V2 (webhooks.py:87-102,150-159); the SDK enforces a 300s tolerance and supports rotation candidates (ubb-sdk/ubb/webhooks.py:39-94).

  > **In plain terms.** UBB also signs the notifications it sends out, so a customer's server can verify "this really came from UBB and hasn't been tampered with." The newer "v2" scheme bakes a timestamp into the signature, which means a captured message stops being valid after five minutes — an eavesdropper who records one delivery can't replay it endlessly to, for example, re-trigger an action. The SDK that customers install enforces that five-minute window and even supports key rotation (accepting two signing secrets briefly during a changeover) so customers can rotate secrets without downtime. This is the modern, correct design, matching how Stripe and others sign their own webhooks. The important caveat, picked up in the gaps: the old timestamp-less "v1" signature is still sent alongside v2 on every delivery, so any receiver still trusting v1 inherits the indefinite-replay weakness.

- **Prod Django hardening correct** (see §4 anchors); bearer APIs correctly CSRF-exempt; webhooks signature-verified.

  > **In plain terms.** These are the standard "lock the doors and windows" web settings, and UBB has them set correctly for production: forcing encrypted HTTPS connections, telling browsers to refuse the unencrypted version for a year, marking session cookies secure, and blocking the page from being embedded in a hostile site (clickjacking). One subtlety that is correct here: the money-moving API endpoints deliberately switch off CSRF protection — that protection is designed for browser-form apps, and for a token-authenticated API it would actively get in the way; the real protection there is the bearer key plus, for webhooks, the signature check. The takeaway for a non-engineer: the conventional web-security checklist is genuinely ticked, which is reassuring because these are exactly the items auditors and security questionnaires ask about first.

- **No committed secrets** (git grep for sk_live_/whsec_/AKIA/PRIVATE KEY: nothing); **logs redact PII/secrets** via RedactingFilter (core/logging.py:19-25).

  > **In plain terms.** Two everyday-hygiene wins. First, no live secrets — Stripe keys, signing secrets, AWS credentials — are accidentally checked into the source code history. This matters because leaked secrets in a code repo are one of the most common and embarrassing breach causes; once committed they effectively live forever in the history and must be rotated. A scan finds none here. Second, the logs automatically scrub out personal data and secrets before writing them down, so an engineer reading logs (or an attacker who gets log access) doesn't see customer PII or credentials in plain text. Think of it as a shredder wired into the logging pipe. Both are signs of a team that has internalized security basics. The honest framing: these are table-stakes done right, not differentiators — but getting the basics wrong is precisely what sinks early-stage vendors, so credit is due.

**Gap analysis — plain-English, verified against code at HEAD.**

**No Postgres RLS and no automated tenant-scoping test — moderate.** **Verified:** Confirmed. `apps/platform/tests/test_product_boundaries.py` is an AST walker that enforces *import* boundaries between the four products (its rule names are literally `imports-api-layer`, `platform-imports-product`, `core-imports-product`, `unsanctioned-cross-product`) — it never asserts that a query carries `.filter(tenant=...)`. So the doc's claim is accurate, not overstated. **What's wrong:** Tenant isolation lives entirely in hand-written application filters. There is no database Row-Level Security as a second wall, and no test that systematically proves every endpoint scopes to the caller's tenant. **Why we should care:** This is the highest-consequence failure mode a multi-tenant billing vendor has: a single forgotten `.filter(tenant=...)` in some future endpoint silently exposes one customer's financial data to another, with nothing to catch it. It is the kind of bug that ends a vendor's credibility in one incident, and the blast radius is "all tenants." **Options:** The gold-standard fix is Postgres RLS keyed on a per-request session variable set from `request.auth.tenant` in middleware — then even a buggy query physically cannot return another tenant's rows (defense in depth the way money systems are built). A cheaper interim is a systematic test that walks every endpoint/queryset and asserts tenant scoping. **Legacy body-only webhook signature still emitted — moderate.** **Verified:** Confirmed. `apps/platform/events/webhooks.py:147,158` computes and sends the timestamp-less `X-UBB-Signature` (`compute_signature`, body-only HMAC) on *every* delivery, right beside the replay-bounded `X-UBB-Signature-V2` (line 150,159). The v1 function's own docstring admits "a captured delivery verifies forever." **What's wrong:** Any receiver still validating the old header gets no replay protection — a captured, valid delivery stays valid indefinitely. **Why we should care:** Webhooks here announce real events (payments, credits, suspensions). An attacker who records one delivery could replay it against a v1-trusting receiver later to re-trigger downstream actions; the freshness guarantee UBB built into v2 is only as strong as the weakest header a customer accepts, and shipping both keeps the weak door open. The exposure is bounded (it requires a customer who still verifies v1, plus a captured payload), hence moderate not major. **Options:** Set a published sunset date for v1; make legacy emission opt-in; default all new webhook configs to v2-only. This mirrors how Stripe handles signature-scheme migration — a deprecation window, not perpetual dual-emission. Best practice is to stop sending the weak credential by default the moment the strong one ships. **Codebase effect:** Additive and small — a per-config flag gating the v1 header plus a default of off for new configs; no schema rewrite, and existing receivers can be migrated on a timeline rather than in a breaking change.

**SSRF validation misses CGNAT 100.64.0.0/10 — moderate.** **Verified:** Confirmed. `core/url_validation.py:45` rejects only `ip.is_private or ip.is_loopback or ip.is_link_local or ip.is_reserved`. The carrier-grade-NAT range 100.64.0.0/10 is none of those in Python's `ipaddress` module, so it passes; a repo-wide grep confirms `is_global` is used nowhere. The doc's "verified" tag is right. **What's wrong:** The SSRF allow/deny list is built as a blocklist of specific bad ranges, and that list is incomplete — CGNAT (used by cloud NATs, mobile carriers, and some internal fabrics) slips through, as can other non-global shared ranges. **Why we should care:** The whole point of the otherwise-excellent SSRF defense (IP pinning, re-validation) is to stop UBB being tricked into calling internal infrastructure. A customer-controlled webhook URL resolving into 100.64/10 could reach an internal service in some network topologies — a narrow but real hole in an otherwise strong control, which is why it rates moderate. **Options:** Invert the logic — require `ip.is_global` to be true and reject everything else. This is strictly safer because it is an allowlist: it automatically subsumes private, loopback, link-local, reserved, CGNAT, and any future special range without anyone maintaining a list. 03 notes competitors don't publish SSRF internals, so the bar here is best practice, and "require is_global" is the canonical best-practice formulation. **Codebase effect:** Additive and tiny — a one-line predicate change in `validate_webhook_url`, plus tests for the newly-rejected ranges; the IP-pinning machinery around it is untouched.

**No dependency-security automation in CI — moderate.** **Verified:** Confirmed (cross-referenced with §6, which establishes there is no `pip-audit`/`bandit`/CodeQL job and no `dependabot.yml`, and that CI has in fact never executed). **What's wrong:** Nothing automatically watches the third-party libraries UBB depends on for newly-disclosed vulnerabilities, and nothing scans UBB's own code for common security anti-patterns. **Why we should care:** Modern breaches frequently arrive through a dependency, not the app's own code — a popular library ships a fix for a serious flaw and everyone who hasn't upgraded is exposed on a public timer. For a vendor that holds the keys to customer money movement, "we found out from the news" is not an acceptable detection path. The cost of the gap compounds: the longer dependencies drift, the larger and riskier the eventual catch-up. **Options:** Add `pip-audit` (known-CVE scan of the locked dependencies) and a CodeQL workflow (static analysis of UBB's code), plus Dependabot configured weekly and grouped so it proposes upgrade PRs automatically. These are free on GitHub and standard for any product handling money; 03 shows every named competitor leads with formal security programs (SOC 2, annual pentests), of which automated dependency scanning is the entry-level floor. **Codebase effect:** Purely additive — new CI workflow files and a `dependabot.yml`; no application code changes. The real prerequisite is the §6 finding that CI must actually run first.

**OpenAPI/Swagger served unauthenticated — minor.** **Verified:** Confirmed. All 12 NinjaAPI routers are instantiated without `docs_url=None` (grep shows `NinjaAPI(auth=...)` for billing, connect, metering, platform, sandbox, tenant, webhooks, margin, subscriptions, referrals, me, and the main api — none disable docs). django-ninja therefore serves interactive Swagger UI and the machine-readable `/openapi.json` for each, with no auth on the docs route. **What's wrong:** The complete shape of the private API — every endpoint, parameter, and money-moving operation — is publicly enumerable in production. **Why we should care:** This doesn't grant access (the endpoints still require a valid bearer key), so it's correctly rated minor. But it hands an attacker a free, accurate map of the attack surface — including the withdraw/refund/grant operations — turning reconnaissance from guesswork into reading the manual. It also leaks product internals to anyone curious. **Options:** Pass `docs_url=None` in production (or gate the docs route behind a staff-only decorator), keeping it on in dev/sandbox. Best practice for a private API is that the schema is not anonymously reachable in production; vendors typically publish a curated public doc site instead of exposing the live introspection endpoint. **Codebase effect:** Additive and trivial — a settings-driven argument on each `NinjaAPI(...)` constructor (ideally via one shared factory so all 12 are consistent); no behavior change to the endpoints themselves.

### Actions

1. Either add RLS keyed on a session GUC set from `request.auth.tenant` in middleware (higher assurance), or at minimum a systematic test walking every endpoint/queryset asserting tenant scoping.
2. Set a sunset date for the legacy webhook signature; make legacy emission opt-in; default new configs v2-only.
3. Invert the SSRF check to require `ip.is_global` (subsumes private/CGNAT/reserved).
4. Pass `docs_url=None` (or a staff-only decorator) in production.
5. Start the compliance runway (SOC 2 readiness, pentest, trust page) — every competitor leads with this.

## 8. Rate limiting

**Relevance: high. Verdict: weak.**

### What exists

- **One real limiter:** per-customer fixed-window RPM in `RiskService.check` — key `ratelimit:{customer.id}:rpm`, threshold `RiskConfig.max_requests_per_minute` (default 60), 60s window, TTL-preserving incr (ubb-platform/apps/billing/gating/services/risk_service.py:21-33). It degrades gracefully (try/except skips limiting on Redis failure).

  > **In plain terms.** UBB has exactly one working speed limit: it counts how many requests a single customer makes per minute and blocks them past a threshold (default 60). Think of it as a turnstile that lets 60 people through per minute, resetting each minute. It's built to "fail open" — if the Redis counter it relies on goes down, it stops limiting rather than blocking everyone, so an infrastructure blip doesn't take the product offline. That fail-open choice is defensible for a courtesy limit. The crucial caveat, which the gaps expand on: this turnstile is installed on a side door (an optional "pre-check" call) that customers can simply walk past, while the actual high-traffic entrances — the metering ingestion endpoints — have no turnstile at all. So the protection exists but is in the wrong place to stop abuse or runaway cost.

- **Coarse input caps:** usage batch 1–100 items (schemas.py:57), rate-card batch ≤100, analytics dimensions ≤6, list limits clamped ≤100.

  > **In plain terms.** Separate from rate limiting, UBB does cap the *size* of individual requests: a usage batch can carry at most 100 events, analytics queries at most six grouping dimensions, and list endpoints return at most 100 rows at a time. These are sensible guardrails — they stop a single request from being absurdly large and hammering the database, the way a "max 20 items" sign at a checkout keeps any one customer from clogging the line. But it's important not to confuse this with rate limiting: size caps limit how big one request is, not how many requests arrive. A caller can still fire unlimited 100-item batches back-to-back. So these caps blunt the worst single-request abuse but do nothing to throttle sustained volume or cost — which is exactly the protection the gaps show is missing at the ingestion path.


### Competitor evidence

Every competitor publishes (or enforces) limits at the **ingest** chokepoint:

- **Lago:** default **500 requests/sec per organization** on POST /events and /events/batch, with 429s carrying `x-ratelimit-remaining`/`x-ratelimit-reset` headers (https://getlago.com/docs/guide/events/ingesting-usage).
- **Stripe meters:** 1,000 calls/sec per account live mode on v1 meter_events; the v2 stream does 10k events/sec standard, 200k via sales; sandbox calls count toward the sandbox global rate limit — 25 req/s (https://docs.stripe.com/billing/subscriptions/usage-based/recording-usage-api; limit figure per https://docs.stripe.com/rate-limits).
- **Orb:** test mode hard-limited to 2,000 events/min and 10 req/s — explicitly "to ensure test mode workloads do not affect live mode availability"; production advisory threshold of 10,000 events/min before contacting the team (https://docs.withorb.com/events-and-metrics/event-ingestion; quoted test-mode rationale from https://docs.withorb.com/quickstart/ingest).
- **Metronome:** documents 429 + exponential-backoff behavior on ingest, with no public numeric cap (https://docs.metronome.com/connect-metronome/send-usage-events/).

UBB, by contrast, rate-limits a path callers can simply not call, and its sandbox tenants share the same (non-existent) ingest limits as live.

### Gaps

| Gap | Severity |
|---|---|
| **The RPM limiter is only invoked on POST /billing/pre-check (billing_endpoints.py:259) — an optional advisory call.** The actual chokepoints POST /metering/usage and /usage/batch do not call RiskService at all (confirmed: no RiskService/cache.incr in apps/metering). A client that skips pre-check has zero rate limiting on ingestion — the DoS and runaway-cost vector on a public billing API. | major |
| **No global, per-API-key, or per-IP throttle anywhere.** No NinjaAPI passes `throttle=`; no NINJA_DEFAULT_THROTTLE_RATES. Invalid-key probing (each = an unthrottled DB lookup), webhook-endpoint flooding, and whole-tenant abuse across customers are all unthrottled. | major |

**Gap analysis — plain-English, verified against code at HEAD.**

**RPM limiter only guards the optional /pre-check, not the ingest chokepoints — major.** **Verified:** Confirmed. The only rate-limit code is in `RiskService.check` (`apps/billing/gating/services/risk_service.py:22-33`), and grep shows `RiskService.check` is invoked from exactly one place: `api/v1/billing_endpoints.py:259`, the `POST /billing/pre-check` handler. A search across `apps/metering` for `RiskService`, `cache.incr`, `ratelimit`, or `throttle` returns nothing — so `POST /metering/usage` and `/usage/batch` call no limiter at all. **What's wrong:** The one turnstile is on an advisory call a client can skip; the real front doors (event ingestion) are wide open. **Why we should care:** On a public billing API this is both a denial-of-service vector and, worse, a runaway-cost vector — every ingested event triggers ~9-13 DB round trips and synchronous pricing (per §10), so an unthrottled flood can saturate the shared database for *all* tenants and, because UBB prices in-line, can also run up real spend fast. **No global, per-API-key, or per-IP throttle anywhere — major.** **Verified:** Confirmed. No NinjaAPI router passes a `throttle=` argument (the grep of all 12 constructors shows only `auth=...`), and there is no `NINJA_DEFAULT_THROTTLE_RATES` setting. **What's wrong:** There is no ceiling on aggregate traffic from a single key, a single IP, or the platform as a whole. **Why we should care:** Three concrete abuses are unthrottled. (1) Invalid-key probing — every attempt is an unthrottled hashed-key DB lookup, so credential-stuffing or scanning hits the database at full speed. (2) The unauthenticated webhook endpoints can be flooded. (3) One tenant, spread across many of its own customers, can saturate shared capacity for everyone (the noisy-neighbor problem, which §10 also flags). Without a per-key or per-IP cap, UBB has no lever to contain a single bad actor short of pulling them off the platform manually. **Options:** Add a global per-API-key throttle and a per-IP throttle — via django-ninja's `throttle=` or, better, at an edge/proxy layer that also protects the key-less webhook routes — and return standard rate-limit headers as Lago does (https://getlago.com/docs/guide/events/ingesting-usage). Orb's pattern of giving sandbox deliberately lower limits than live (https://docs.withorb.com/quickstart/ingest) is worth copying. **Codebase effect:** Additive if done in-app (throttle classes + settings); cleaner long-term as an edge concern (gateway/WAF) that also covers the unauthenticated endpoints the app can't easily self-throttle.

**Limiter and gate default fail-open — moderate.** **Verified:** Confirmed. `RiskConfig.gate_fail_closed` defaults to `False` and `BudgetConfig.fail_closed` defaults to `False` (`apps/billing/gating/models.py:9,34`); `budget_service.py:79` reads `fail_closed = bool(rc and rc.gate_fail_closed)`, and `risk_service.py:32-33` swallows any cache exception with `pass`. **What's wrong:** If Redis is degraded, the rate limit and the budget cap both quietly stop enforcing rather than blocking. **Why we should care:** This is the right call for a *courtesy* rate limit (you don't want a cache blip to 500 the whole API — §9 makes this argument well), but it is the wrong call for a *hard money cap*: an attacker who can degrade Redis silently removes both the throttle and the spend ceiling at once, with no alarm. The danger is that the same fail-open switch governs both a soft and a hard control, so the operator gets no signal that enforcement has evaporated. Rated moderate because it requires an attacker to first degrade Redis. **Options:** Split the policy by what's being protected — keep rate limiting fail-open but alert loudly the moment it degrades, and offer (or default) fail-closed for hard money caps so a budget can't be bypassed by knocking over the cache. Best practice for spend-control gates in money systems is fail-closed with explicit, monitored exceptions. **Codebase effect:** Additive — the `fail_closed` flags already exist; this is changing defaults for money caps and wiring a degradation alert, not new machinery.

**max_concurrent_requests defined but never enforced — moderate.** **Verified:** Confirmed. `RiskConfig.max_concurrent_requests` (default 10) is declared at `models.py:8` and shown in the admin list and `__str__` (`admin.py:7`, `models.py:15`), but grep finds no read of it in any service — `risk_service.py` enforces only `max_requests_per_minute`; the only other hits are a test fixture and migrations. **What's wrong:** It's dead configuration — a knob in the admin that does nothing. **Why we should care:** A concurrency cap (how many requests a customer may run *at once*, distinct from per-minute rate) is a real lever for protecting the database from a burst of simultaneous expensive operations. Presenting an operator a "max concurrent requests = 10" field that is silently ignored is worse than absent: it manufactures false assurance, so an operator may believe they're protected against a concurrency burst when they are not. Rated moderate on that false-confidence basis. **Options:** Either implement it (a concurrency limiter is harder than a counter — typically a short-lived Redis lease/semaphore per customer, acquired at request entry and released at exit) or delete the field so the config tells the truth. Stripe documents a concurrency limit of one in-flight meter call per customer per meter (https://docs.stripe.com/billing/subscriptions/usage-based/recording-usage-api), confirming concurrency control is a real ingest concern — but a half-built knob is the worst of both worlds. **Codebase effect:** Implementing is structural-ish (new lease logic on the hot path, with its own fail-open story); deleting is a trivial additive cleanup (drop the field via migration + admin). The honest near-term move is usually delete-now, implement-when-needed.

**RiskConfig admin-only, no tenant API — minor.** **Verified:** Confirmed. `RiskConfig` is reachable only through Django admin (`admin.py`) and the ORM; no endpoint exposes it, so every tenant runs the hard-coded default of 60 rpm. **What's wrong:** Tenants can't see or adjust their own rate/risk limits self-serve. **Why we should care:** It's minor because it's an operability/product-polish gap, not a safety hole — but it means limits are one-size-fits-all and any change requires UBB staff to edit the admin by hand, which doesn't scale and gives no plan-based differentiation. **Options:** Expose RiskConfig through a tenant-facing API with plan-based defaults, and deliberately give sandbox tenants lower limits than live — Orb's documented pattern, where test mode is capped specifically so it can't affect live availability (https://docs.withorb.com/quickstart/ingest). This pairs naturally with fixing the first two gaps, since you'd want per-tenant limits to be settable once real ingest throttling exists. **Codebase effect:** Additive — a CRUD endpoint over the existing model plus plan-default seeding; no change to the gating mechanics themselves.

### Actions

1. Enforce limiting at record_usage and /usage/batch themselves (batch counts *items*, not requests).
2. Add a global per-API-key throttle and a per-IP throttle (django-ninja `throttle=` or edge layer), including the unauthenticated webhook endpoints; return rate-limit headers like Lago's (https://getlago.com/docs/guide/events/ingesting-usage).
3. Alert loudly when the limiter degrades; consider fail-closed defaults for hard money caps.
4. Implement or delete `max_concurrent_requests`.
5. Expose RiskConfig via tenant API with plan-based defaults; give sandbox tenants deliberately lower limits (Orb's pattern).

## 9. Caching & CDN

**Relevance: high. Verdict: adequate.**

### What exists

- **CDN: correctly absent** — JSON API backend, no static/media pipeline beyond admin (settings.py:102).

  > **In plain terms.** A CDN (content delivery network) is the global cache giants use to serve images, video, and web pages from a server near each user. We don't have one, and that's the right call, not a gap. UBB is a pure data API: it receives "this customer used X" and returns a price decision. There's nothing static to cache near users — every request needs a fresh, exact answer from the database. Buying a CDN would add cost and complexity for zero benefit. The one place static files exist is the internal admin screen, which only our own staff use. Think of it like a bank's transaction system versus its marketing website: the website wants a CDN, the ledger does not. The reviewer flagged this as "correctly absent" precisely to pre-empt anyone asking "where's your CDN?" — the honest answer is that it would be the wrong tool here.

- **One Redis, four roles:** Django cache, Celery broker, Celery results, /ready probe (settings.py:110-118; api/v1/endpoints.py:24-32).

  > **In plain terms.** Redis is a fast in-memory scratchpad sitting beside the main database. We run a single Redis box doing four jobs at once: (1) a cache for quick lookups, (2) the to-do list that hands background work to our task workers, (3) a place those workers post their results, and (4) a heartbeat that tells our load balancer "I'm healthy." Using one Redis for everything is cheap and simple for launch volume — but it's also a strength worth questioning, because those four jobs have conflicting needs and a single Redis is a single point of failure. The caveat the reviewer raises later is real: piling unrelated workloads onto one Redis means an operational mistake on one job (say, clearing the cache) can disrupt another (the work queue). At our current scale it's a reasonable, money-saving choice; at scale it becomes a thing to split apart.

- **Complete cache usage map:** budget spend counters with 62-day TTL and rebuild-on-miss that deliberately avoids double-count (apps/billing/gating/services/budget_service.py:10,41-65); per-seat fixed-window rate limit (risk_service.py:22-31); tenant-products dispatch gate with 300s TTL + write-through invalidation (events/dispatch.py:15-34; tenants/models.py:125); API-key last-used write-behind buffer flushed by beat (core/auth.py:19; core/tasks.py:12-31).

  > **In plain terms.** This is the reviewer cataloguing every single thing we cache, which is a sign of a well-understood system rather than cache sprinkled around hopefully. Four things use the cache: a running tally of each customer's monthly spend (kept ~2 months so we don't recount from scratch and, importantly, rebuilt carefully so we never double-count); a per-customer "requests per minute" speed limit; a 5-minute memory of which products each tenant has bought (refreshed instantly when it changes); and a buffer of "API key last used" timestamps that get written to the database in batches rather than on every call. The strength here is discipline — each cache entry has a clear purpose, a clear lifetime, and a clear refresh rule. Nothing is cached "just in case." The caveat is that this map also reveals what is NOT cached: the per-request config and balance lookups the reviewer flags later as uncached hot reads.

- **Redis-death behavior is explicitly engineered for the gating paths:** rate limiting fails open; budget check fails open by default with explicit fail-closed opt-in returning `budget_unavailable`; the post-drawdown counter increment is fully fail-open with an hourly Postgres rebuild (risk_service.py:32-33; budget_service.py:72-83,126-153). Tests exercise Redis-failure paths (gating/tests/test_risk_service.py:102).

  > **In plain terms.** The key question for any cache is: what happens when it dies? Here the team has deliberately designed each gate to "fail open" — if Redis goes dark, the speed limiter simply stops limiting and lets traffic through, and the budget check defaults to allowing usage (with an explicit opt-in to instead block, for customers who'd rather be safe than sorry). The spending counter, when Redis returns, is rebuilt from the real database every hour. The strength: a Redis outage degrades gracefully into "slightly less protected" rather than "completely down," and there are tests that literally simulate Redis failing. The caveat — and this is the crux of the whole section — is that this careful design has one hole the reviewer found: the login/auth step does an unguarded cache write, so a Redis blip there crashes the request instead of degrading. The intent is sound; one line breaks the pattern.

- **Counters are caches, not ledgers:** durable budget truth is the Postgres UsageEvent aggregate; idempotency is Postgres constraints; all mutual exclusion is `select_for_update` with a documented canonical lock order — zero Redis locks (apps/billing/locking.py:1-11).

  > **In plain terms.** This is the most important reassurance in the section. The money truth never lives in Redis. The cache holds fast approximations — running totals, speed-limit counts — but the authoritative record of what every customer owes lives in the main database (Postgres), protected by database-level locks and uniqueness rules. Redis is the speedometer; Postgres is the odometer that the invoice is built from. Practically, this means even a total Redis wipe cannot lose or corrupt a penny of billing data — at worst a counter is briefly wrong until the hourly rebuild corrects it. There are deliberately zero locks held in Redis; all the "only one writer at a time" safety is done in the database with a documented, fixed lock ordering to prevent deadlocks. This is exactly the architecture you want for a billing system and the reviewer credits it explicitly: the cache failures are inconveniences, not data-integrity risks.

- **Celery-on-Redis death story:** domain events live in a Postgres transactional outbox swept every minute, so delivery resumes from durable state (events/models.py:15-39; settings.py:180-183).

  > **In plain terms.** "Celery" is our background-work system and it uses Redis as its to-do list. The worry would be: if Redis loses that list, do we drop work — like failing to charge a wallet or send a webhook? The answer is no, because of a pattern called a "transactional outbox." When something important happens, we write the to-do item into the durable database in the very same step that records the event, so the two can't disagree. A scheduled job sweeps that database table every minute and re-hands the work to the queue. So even if Redis loses everything, a sweep resumes the pending work from the database. The strength: no business event is silently lost when the queue's memory is wiped. The caveat is mostly about timing — recovery happens on the next sweep (within a minute), not instantly — but nothing is dropped, which is what matters for billing.

- Test hygiene acknowledges the shared-DB hazard: tests run on Redis DB 15 so `cache.clear()` can't flush broker data (ubb-platform/conftest.py:1-42).

  > **In plain terms.** Because one Redis does four jobs, there's a real risk a test that clears the cache could accidentally wipe the work queue too — the same hazard the reviewer flags as a production gap. The team noticed this and routed the test suite to a separate, isolated Redis slot (database number 15) so a cache-clearing test can never reach into the real queue's data. The strength: this is evidence the team already understands the shared-Redis danger — it's not an oversight, it's a known trade-off they've fenced off in testing. The caveat, and the reason the reviewer keeps pointing at it, is that the same protective separation they applied in tests has NOT been applied in production, where cache and queue still share one slot. So the test comment is simultaneously reassuring (they get it) and an indictment (they haven't fixed it where it counts).


### Competitor evidence / best-practice framing

No competitor publishes cache internals; this category is judged on best practice. The "Redis is a cache, Postgres is the ledger" discipline here is exactly right and worth crediting — the failure is a single inconsistency, not the design.

**Result backend configured but never read — minor.** **Verified:** Confirmed. `settings.py:118` sets `CELERY_RESULT_BACKEND = REDIS_URL`, but a repo-wide search finds zero `AsyncResult` usages and no Celery `.get()` on a task handle (the only `.get()` hits are Django QuerySet/ContextVar calls in tests), and there is no `CELERY_TASK_IGNORE_RESULT`/`task_ignore_result` setting anywhere. It's pure write churn and memory pressure on the broker instance — made worse by the shared-DB gap above, since that churn lands in the same Redis the queue depends on. **Why we should care:** Low severity by itself — it wastes a little memory and I/O. The reason to fix it is that it compounds the shared-Redis risk: needless result writes consume the same memory budget the broker needs to hold real tasks. **Options:** Best practice (standard Celery guidance) is to set `CELERY_TASK_IGNORE_RESULT = True` globally when nothing consumes results, and opt specific tasks back in only if a result is ever needed. No competitor evidence applies (internal plumbing). **Codebase effect:** Additive one-liner in settings; no behavior change because nothing reads results today.

**Non-atomic fixed-window rate limiter — minor.** **Verified:** Confirmed at `risk_service.py:24-31`: it does `cache.get(cache_key, 0)`, compares to the limit, then `cache.incr` — a read-then-increment that is not atomic, in a fixed per-minute window keyed `ratelimit:{customer.id}:rpm`. The whole block is wrapped to fail open (line 32-33). **What's wrong:** Two concurrent requests can both read the same count before either increments, so the limit can be overshot under race; and because the window resets on a fixed clock boundary, a client can fire a full window's worth just before the boundary and again just after — roughly 2x the nominal limit across the seam. **Why we should care:** Today the limiter is an advisory, fail-open guard, so approximate enforcement is acceptable — the doc rates it minor for exactly that reason. It only becomes a real problem if the limit ever becomes a contractual promise to a customer or a hard money cap, where "we might let through double" is not good enough. **Options:** Lago returns precise rate-limit headers (`x-ratelimit-remaining`, `x-ratelimit-reset`) on a documented 500 req/s/org limit (03, https://getlago.com/docs/guide/events/ingesting-usage, fetched 2026-06-12), implying exact accounting. Best practice for exactness is INCR-first (increment then check, atomic by Redis's single-threaded nature) or a sliding-window/token-bucket algorithm. Otherwise, simply document it as best-effort. **Codebase effect:** Additive and localized — swap the get-then-incr for an atomic INCR-first pattern in one function; no schema or API change unless headers are added.

**Uncached hot per-request config reads — minor.** **Verified:** Confirmed. On the gating path, `risk_service.py:18` reads `customer.tenant.risk_config`, `:39-43` reads the `Wallet` row and `get_customer_min_balance`, and `budget_service.py:32-35` reads `BudgetConfig` — all live primary-DB queries with no cache layer. Only tenant products get the 5-minute cache (`dispatch.py:15-34`). Accurate. **What's wrong:** Each authenticated request that hits the gate re-queries the database for config that changes rarely (risk limits, budget caps) plus a balance read, instead of caching the slow-changing pieces. **Why we should care:** It's fine at launch volume — these are indexed single-row lookups. It matters only when request rate climbs and these reads start contributing to p99 latency and database connection pressure (which ties into the §10 connection-pooler and throughput gaps). **Options:** The codebase already has the exact pattern to copy — the tenant-products 300s cache with write-through invalidation in `dispatch.py` and `tenants/models.py:125`. Best practice is a short 30-60s TTL cache for the slow-changing config (RiskConfig/BudgetConfig), leaving the wallet balance uncached since it must be exact. No competitor cache-internals evidence exists (03). **Codebase effect:** Additive — mirror the existing cache+invalidate pattern for two config models; no structural change, and balance reads stay live.

### Actions

1. try/except the auth `cache.set` and Tenant.save `cache.delete` (matching the pattern three lines away in the gating services); regression-test with a raising cache.
2. Split cache vs broker into separate logical DBs (separate env vars), document `noeviction` for the broker, preferably two instances.
3. Set `CELERY_TASK_IGNORE_RESULT = True` until something consumes results.
4. If the limit ever becomes contractual, switch to INCR-first or sliding window; otherwise document best-effort.
5. When p99 gate latency matters, add 30–60s cached reads for Risk/BudgetConfig mirroring the tenant_products pattern.

## 10. Load balancing & scaling

**Relevance: high. Verdict: weak.**

### What exists

- **The hot path is fully characterized:** a typical single-metric tiered record_usage costs ~9–13 sequential DB round trips in one transaction — key lookup, customer 404, idempotency pre-check, billing-owner resolution, rate-card resolve per metric per side, tier ladder advance (exists + savepoint create + SELECT FOR UPDATE + UPDATE), UsageEvent INSERT, OutboxEvent INSERT — plus 2 Redis ops (anchors: tenants/models.py:182; metering_endpoints.py:50; usage_service.py:122-132; pricing_service.py:69-72,98-100,144-146; tier_counter_service.py:52-65; usage_service.py:176; outbox.py:26; core/auth.py:19).

  > **In plain terms.** The reviewer has traced exactly what happens, step by step, when one usage event comes in: recording a single tiered metric costs roughly 9-13 back-and-forth trips to the database, all inside one transaction, plus two quick Redis touches. Those trips include checking the API key, finding the customer, making sure it's not a duplicate, working out who pays, looking up the price, advancing the running tier total under a lock, and writing both the event and its outbox to-do. The strength is that this is fully mapped — the team knows precisely where the time goes, which is the prerequisite for any scaling work. The caveat is the count itself: 9-13 sequential round trips per event is a lot of chatter for a system that competitors run at hundreds of thousands of events per second. It's correct and safe, but it's the heavy, careful approach, not the high-throughput one.

- **Serialization points verified:** the per-(tenant, customer, lineage, calendar-month) `PricingPeriodCounter` row lock is held from tier_counter_service.py:62 until the **outer** transaction commits — the event insert and outbox insert execute under it (usage_service.py:156-220); multi-metric deadlock prevented by sorted lock order (pricing_service.py:143-144); wallet money movement is deliberately OFF the hot path, serialized per billing owner asynchronously in the outbox handler (apps/billing/handlers.py:49-50 via locking.py:26-39).

  > **In plain terms.** A "serialization point" is a place where requests must queue up single-file instead of running in parallel. The reviewer verified the main one: for a given customer's tiered pricing in a given month, only one event at a time can hold the lock that advances the running total, and that lock is held until the whole transaction commits. That's deliberate — it's what guarantees tiered pricing is counted exactly right, never double-counted or skipped. Importantly, the actual money movement (charging the wallet) is pushed OFF this hot path and done afterwards in the background, serialized per payer, so it doesn't widen the queue. Multi-metric events sort their locks in a fixed order to avoid deadlocks. The strength: correctness is airtight. The caveat: this single-file lock per customer-per-month is precisely the ceiling the gaps section flags — it caps how fast one busy customer's events can flow, no matter how many servers you add.

- **Reasoned capacity class:** order of 100–300 record_usage rps for one web instance against one Postgres with traffic spread across customers; a single hot (customer, lineage, month) tier ladder is hard-capped at one lock-holder at a time — order of 100–200 events/s for that customer/metric regardless of replica count.

  > **In plain terms.** The reviewer gives an honest, reasoned estimate of how fast UBB can go today: roughly 100-300 events per second on one web server against one database when traffic is spread across many customers, and only about 100-200 per second for any single hot customer because of that one-at-a-time tier lock. This is a "reasoned" figure, not a measured one — nobody has load-tested it — so treat it as an informed engineering judgment, not a benchmark. The strength is the intellectual honesty: rather than claiming a big number, the reviewer derived a defensible range from the actual code path. The caveat, made explicit in the competitor framing, is that this is two to three orders of magnitude below Metronome's headline 100,000 events/sec and Orb's 250,000+ — though those are streaming pipelines we don't need on day one. The real near-term bar is Orb's standard API at "tens of millions of events/day," and we sit at or just below its bottom edge.

- **Web-tier horizontal scaling is safe by construction:** stateless requests, DB-keyed idempotency makes cross-replica retries return the original event, and the batch endpoint deliberately runs items as independent transactions (metering_endpoints.py:184-205).

  > **In plain terms.** "Horizontal scaling" means adding more identical web servers to handle more traffic. The good news here is that the code is built so this is safe: each request carries no memory of previous ones (stateless), and duplicate-protection lives in the database, so if the same event hits two different servers, both correctly resolve to the one original record rather than billing twice. The batch endpoint even runs its items as independent transactions so they spread cleanly. The strength is genuine and worth crediting — the hard correctness work for running multiple servers is already done. The crucial caveat, spelled out in the very next bullet, is that "safe to run two servers" is not the same as "set up to run two servers." Nothing in the codebase actually launches a second replica — there's no load balancer, no process manager, not even the web server (gunicorn) installed. The foundation is poured; the building isn't there.

- **There is no LB, autoscale, replica, or capacity configuration anywhere in the repo** — gunicorn isn't even installed.

  > **In plain terms.** This is the blunt bottom line: despite the code being scale-ready, there is zero operational machinery to actually scale it. No load balancer config, no autoscaling rules, no replica definitions, no sizing guidance, and — strikingly — gunicorn, the standard production web server for Python, isn't even installed. The docker-compose file ships only a database and a Redis, with no web service at all. The strength of stating it this plainly is that it leaves no false impression of readiness. The caveat is that this is the headline weakness of the whole section: today UBB can run exactly one server, by hand, and there is no documented path to a second. For a billing platform pitching reliability, "we've only ever run one instance and have no config to run more" is a real gap — not because the code can't, but because nobody has wired up the means to.

## 11. Error tracking & logs

**Relevance: critical. Verdict: weak.**

### What exists

- **A real structured-logging stack, stronger than typical for this stage:** correlation-ID contextvar + CorrelationIdFilter + RedactingFilter + JsonFormatter emitting timestamp/level/logger/correlation_id/message/data/exception (ubb-platform/core/logging.py:15,59-64,67-95,98-126). The `extra={"data": {...}}` convention is documented and followed (113 of 124 logging calls in non-test source).

  > **In plain terms.** Every line our software writes to its log is already shaped like a clean database row, not a wall of free text — it carries a timestamp, a severity, which component spoke, a tracing ID, and a structured data blob. Think of it as the difference between a shoebox of handwritten receipts and a proper accounting ledger: the ledger can be searched, filtered, and totalled by machine. The discipline is real and unusually mature for our stage — 113 of 124 log calls in real (non-test) code follow the documented convention, which is the hard, human part that most teams skip. The strength: the day we plug in monitoring, this data is instantly usable. The caveat: producing tidy logs and *watching* them are different things — today these lines stream to a screen nobody is looking at (see the gaps).

- **PII redaction exists and is tested:** REDACT_KEYS covers email/phone/card/ip/etc., substring matching covers secret/token/api_key/password, an email regex scrubs messages and exception traces (logging.py:19-26,32,55-56,92,111-114); tests pin that stripe keys and emails redact while amounts and Stripe customer IDs are deliberately preserved (core/tests/test_logging.py:33-59).

  > **In plain terms.** Before any log line is written, sensitive customer data — email addresses, phone numbers, card details, API keys, passwords, tokens — is automatically scrubbed and replaced with a placeholder. Crucially this is *tested*: there are automated checks proving that a Stripe secret and a customer email get masked, while the genuinely useful business facts (dollar amounts, Stripe customer IDs) are deliberately kept so we can still investigate problems. The analogy is a bank that records "customer paid $4,000" but never writes down the card number. Why this matters commercially: leaking PII into logs is one of the most common, most embarrassing, and most regulated mistakes a billing company can make, and it directly affects SOC 2 / GDPR posture. We have addressed the common case by design. The caveat is a narrow blind spot for unusually-formatted data, covered in the gaps.

- **Correlation IDs are end-to-end:** middleware validates/accepts/echoes X-Correlation-ID (core/middleware.py:19-36); Celery propagation via signals (config/celery.py:14-31); outbox events persist it and the dead-letter alert carries it (outbox.py:21-22; events/tasks.py:40).

  > **In plain terms.** Every request that enters the system gets a tracking number, and that same number is stamped onto every log line, every background job, and every event it triggers — even as the work hops between the web server and the asynchronous workers that move money. It is the courier-parcel tracking number for a single customer action: when something goes wrong, you can follow that one ID across the whole journey instead of guessing which of thousands of overlapping operations was the culprit. For a billing platform this is the foundation of dispute forensics — "show me everything that happened to this charge." The strength is that it is wired *end-to-end*, including across the worker boundary where most systems lose the thread. The caveat: it is only as useful as our ability to search the logs by it, which today (no aggregation) we cannot do after the terminal closes.

- **A consistent dotted event-name taxonomy forms a ready-made alert table:** CRITICAL `outbox.dead_letter`; ERROR including `billing.usage_push_failed_permanent`, `postpaid.snapshot_divergence`, `ledger.usage_deduction_amount_mismatch`, `wallet.drawdown_repair_spike` (threshold-gated), `pricing.tier_rerate_drift`, Stripe top-up reconciliation mismatches; WARNING including webhook delivery failures and `wallet.drawdown_repaired` (anchors throughout apps/billing and apps/platform — e.g. events/tasks.py:31; postpaid_service.py:94,156,254; handlers.py:57; wallets/tasks.py:146,170-171).

  > **In plain terms.** The code already names its alarm conditions with a consistent, machine-readable vocabulary — `outbox.dead_letter`, `postpaid.snapshot_divergence`, `wallet.drawdown_repair_spike` and so on — and grades each by severity (critical, error, warning). In effect the engineering team has pre-written the list of "things that should wake someone up," which is normally the slowest and most contested part of standing up monitoring. The analogy: the smoke detectors are installed and labelled by room; we just have not connected them to a fire alarm yet. The commercial value is that adopting an alerting tool becomes a wiring job of hours, not a design project of weeks, because the meaningful signals are already defined and emitted at the right moments. The caveat, made explicit in the gaps: a name is only an alert if something is counting and watching it — right now nothing is.

- **Runbook-shaped management commands with guard rails:** `reprocess_webhook` refuses non-failed events; `repush_usage_invoice` documents --rebill-void and refuses statuses where repush would double-bill (apps/billing/stripe/management/commands/reprocess_webhook.py:6-33; invoicing/management/commands/repush_usage_invoice.py:7-36).

  > **In plain terms.** For the two scariest manual recovery operations — replaying a failed webhook and re-pushing a usage invoice — there are purpose-built admin tools with safety catches: the webhook tool refuses to touch anything that is not already in a failed state, and the invoice tool refuses to run in situations where it would double-bill a customer. Think of it as a defibrillator with an interlock that won't fire if it would do harm. This matters because the moments you reach for these tools are exactly the high-stress, 2am, real-money moments where a fat-fingered command can turn one incident into two. Having the guard rails baked into the tool, rather than into the operator's memory, is genuine operational maturity. The caveat: the tools exist but, as the gaps note, there is no runbook telling a tired engineer *when* and *how* to use them.

- **Correction to one source audit:** the claim that no /health or /ready endpoints exist is **wrong** — both exist with real DB+Redis checks (ubb-platform/api/v1/endpoints.py:11-38) and are mounted at config/urls.py:40. The valid residue of that finding is the absence of *metrics* and a *beat heartbeat*, covered below.

The honesty cuts both ways: the audit corrects the false negative but preserves the *valid* residue of the original finding — we still lack metrics and a heartbeat that proves the background scheduler is alive. ## 12. Availability & recovery

**Relevance: critical. Verdict: weak.**

### What exists

The recovery *design* is the strongest part of this codebase and deserves explicit credit:

- **Transactional outbox with belt-and-suspenders delivery:** events written in the same transaction as the domain change (events/outbox.py:26-35), dispatched on_commit AND re-dispatched by a minutely sweep that reclaims stuck rows after 5 min (events/tasks.py:84-121), bounded retries (backoff 30s→2h, ~2h43m horizon) and dead-letter logging (tasks.py:20,29-43). Broker loss is survivable because Postgres rows, not broker messages, are the source of truth.

  > **In plain terms.** When the system changes something that matters (say, a wallet is debited), it writes the "now tell everyone else" message into the *same database transaction* as the change itself. So the fact and its follow-up notification are saved together or not at all — they can never disagree. That message is then delivered two independent ways: immediately, and again by a sweeper that runs every minute and re-sends anything that got stuck. The analogy is mailing yourself a carbon copy of every instruction *and* having an assistant re-check the outbox tray each minute. Why a CEO should value this: the usual nightmare of event-driven money systems is "the charge happened but the downstream effect was lost." Here, because the durable Postgres row — not a fragile in-flight broker message — is the source of truth, losing the message broker entirely is survivable. The system is built to not lose work, only to occasionally delay it.

- **A real, self-healing reconcile lattice** (each verified in source, all beat-scheduled in settings.py:147-254): `reconcile_usage_drawdowns` repairs dead-lettered wallet debits exactly-once via the `usage_deduction:{event_id}` key + anti-join column, with GRACE=6h deliberately exceeding the retry horizon and a repair-spike alert (wallets/tasks.py:99-171); `reconcile_topups_with_stripe` re-credits lost PaymentIntent successes from Stripe with a 4-day lookback exceeding Stripe's ~3-day webhook retry (connectors/stripe/tasks.py:117-174); `reconcile_invoice_payment_status` repairs missed invoice webhooks along the same legal-transition table as the fast path (invoicing/tasks.py:146-207); `reconcile_postpaid_usage` bounded to terminal failed_permanent inside Stripe's 24h idempotency window (postpaid_service.py:126-134); `reconcile_wallet_balances` is a full ledger-vs-balance drift detector (wallets/tasks.py:10-96). **Stripe-down degrades and converges rather than losing money state.**

  > **In plain terms.** This is the strongest asset in the whole platform, and it is worth understanding why. For every kind of money movement — usage debits, wallet top-ups, invoice payment status, postpaid usage, ledger balances — there is a second, independent background job that periodically asks "did this actually complete, and does it match the truth?" and *repairs* it if not, without ever double-charging. Verified in code: the usage-debit repair waits a deliberate 6-hour grace (longer than the live retry window) and uses a unique key per event so it can only ever apply a missing debit once; the top-up and invoice repairs re-read Stripe with a 4-day lookback that outlasts Stripe's own ~3-day retry window. The business meaning: if Stripe has an outage, our money state does not get corrupted or lost — it *degrades and then converges* back to correct on its own once Stripe returns. This is precisely the "accuracy guarantee" enterprise billing vendors sell as their premium differentiator (Orb advertises enterprise SLAs "for accuracy", https://www.withorb.com/pricing); we have it demonstrably in code. The one caveat: the repair jobs only run if the scheduler is alive, which is why the missing heartbeat (gaps) matters so much.

- **Redis-down graceful degradation exists in the spend-control layer as designed** (risk_service.py:32-33; budget_service.py:73-83,126-153) — defeated at the auth layer, see gaps.

  > **In plain terms.** The spend-control layer (budget caps and risk checks) was explicitly designed to keep working when the fast in-memory store, Redis, is unavailable: if it can't read the live spend counter it falls back to a configured policy — fail-open (let the request through) or fail-closed (block it) — rather than crashing. The analogy is a tollbooth whose computer is down choosing, by prior policy, to either wave cars through or lower the barrier, instead of simply seizing up. This is good defensive engineering and shows the team thought about partial outages. The important caveat, flagged honestly in the gaps and verified in code: this careful design is *defeated one layer up* at authentication, where a single unguarded Redis write means a Redis outage rejects all API traffic before the graceful-degradation code is ever reached. So the capability exists but is currently short-circuited — a small fix unlocks a feature already largely built.

- **Inbound webhooks crash-safe** (signature + DB dedup + CAS retry + 500-for-retryable so Stripe redelivers ~3 days; hourly pollers backstop beyond that — api/v1/webhooks.py:66-176).

  > **In plain terms.** When Stripe calls us to report an event (a payment succeeded, an invoice was paid), the handling is built to survive crashes and duplicates: it verifies the message is genuinely from Stripe, records it once in the database so the same event can't be processed twice, and if something transient fails it deliberately returns an error so Stripe will redeliver it — Stripe keeps retrying for about three days, and our own hourly pollers catch anything beyond that. The analogy is a signed-for delivery that the courier keeps re-attempting, with a backstop check in case the courier eventually gives up. Why this matters: webhooks are how the outside world tells our billing system that money moved, and dropped or double-processed webhooks are a classic source of wrong invoices. This design closes both failure modes. The strength here reinforces the reconcile lattice above — the fast path and the slow repair path use the same legal-transition rules, so they cannot disagree.

- **Celery loss semantics defensive:** acks_late + reject_on_worker_lost; money tasks bound Stripe retries and never hold DB locks across Stripe calls (two-phase claim, tenant_billing/tasks.py:122-182); stale pending top-ups expire after 30 min (topups/tasks.py:10-29).

  > **In plain terms.** The background-job system is configured so that if a worker dies mid-task, the job is not silently lost — it is re-queued and retried, and the money-touching jobs are written to never hold a database lock open while waiting on Stripe (a two-phase "claim then call" pattern) and to bound how many times they retry against Stripe. Stale half-started top-ups are also expired after 30 minutes so they can't linger. The analogy is a kitchen where, if a cook collapses, the ticket goes back on the rail rather than the order vanishing — and no cook ever blocks the whole pass while waiting on a slow supplier. Why a CEO should value this: worker crashes are routine in any deployment, and the difference between "job retried automatically" and "charge silently dropped" is the difference between a non-event and a customer dispute. Combined with the outbox and the reconcile lattice, the platform's posture is consistently "delay, don't lose."

- Accurate readiness reporting (api/v1/endpoints.py:16-38); honest migration posture (0022 CONCURRENTLY runbook; 0006 quiesce note).

  > **In plain terms.** The readiness endpoint doesn't just say "I'm up" — it actually tests the database and Redis connections and reports honestly, returning a not-ready status when a dependency is broken; and the team has documented the awkward database migrations truthfully (which ones need special handling or a quiet window) rather than papering over them. The analogy is a pre-flight checklist that genuinely checks the instruments instead of a pilot just saying "looks fine." Why this matters commercially: honest readiness reporting is what lets an orchestrator stop sending customer traffic to a broken instance, and honest migration notes are what prevent a routine deploy from corrupting live billing data. The strength is the candor — the system tells the truth about its own state. The caveat, developed in the gaps, is that knowing a migration needs a quiet window is not the same as having the deployment machinery to enforce one.

**Gap analysis — plain-English, verified against code at HEAD.**

**Backups, PITR, DR, RPO/RTO — blocker.** **Verified:** Confirmed by reading `docker-compose.yml:1-23`: a single Postgres service with one named volume (`ubb_postgres_data`, lines 12-13) is the only persistence, and there is no backup, snapshot, point-in-time-recovery, or standby anywhere in the repo, nor any document stating a target RPO/RTO. **What's wrong:** The usage-event stream and the wallet ledger — the records of what customers consumed and what they owe — exist in exactly one database with no copy. The §12 narrative correctly notes Stripe can reconstruct top-up and invoice state (the reconcile lattice proves this), but usage and the wallet ledger have no external source of truth, so if that volume is lost the effective Recovery Point Objective is "everything." **Why we should care:** For a billing company, the ledger *is* the product; losing it is not a degraded service, it is an extinction event and an un-survivable customer-trust failure. This is the single largest distance-to-best-practice item in the entire audit. Competitors treat durability as table stakes and sell on it — Lago advertises "99.9% historical uptime" with enterprise SLAs (https://www.getlago.com/security) and Orb publishes per-component uptime including Invoicing at 100.0% (https://status.withorb.com); none of those numbers are even expressible without backups. **Options:** Move to managed Postgres (RDS/Cloud SQL/equivalent) with automated backups plus PITR targeting RPO under five minutes, a multi-AZ standby, and — critically — a *rehearsed* restore that ends by letting the reconcile lattice re-converge state. Best practice is to test the restore, not just configure the backup. **Codebase effect:** Primarily infrastructural and additive (managed-DB connection string, ops procedure); the code already tolerates the convergence step, so no structural domain change is required.

**Three SPOFs with no HA story — major.** **Verified:** Confirmed. `docker-compose.yml` runs one Postgres and one Redis; `settings.py:107-118` shows that single Redis serves all three roles at once — Django cache, Celery broker, and Celery result backend (all pointed at `REDIS_URL`). Beat is a single scheduler with no redundancy, and the whole beat schedule lives at `settings.py:147-254`. **What's wrong:** Three components each have exactly one instance and no failover, and one of them (Redis) is triple-loaded, so a single Redis incident takes out caching, job queuing, and job results simultaneously. The most dangerous is beat: if it stops, the minutely outbox sweep and every hourly reconcile job stop with it, and — verified above — nothing detects the silence. **Why we should care:** The self-healing design that is this platform's crown jewel *assumes its repair jobs actually run*; a dead beat quietly disables the entire safety net while everything still looks superficially alive. Competitors run multi-component, presumably-redundant infrastructure behind their public status pages (Orb, https://status.withorb.com). **Options:** Best practice is a Redis replica with Sentinel (or splitting cache from broker onto separate instances), a database standby (overlaps the blocker), and RedBeat plus a staleness heartbeat so a stalled scheduler pages someone. **Codebase effect:** Mostly additive/infrastructural; RedBeat is a config-level swap of the beat scheduler, and the cache/broker split is a settings change — no domain logic.

**Fail-open architecture defeated by one unguarded Redis write — major.** **Verified:** Confirmed precisely at `core/auth.py:19`: after `TenantApiKey.verify_key(token)` has already succeeded, the code calls `cache.set(f"apikey_used:{key_obj.pk}", ...)` with no try/except. If Redis is down this line raises and the request 500s — even though authentication itself succeeded and the spend layer's verified fail-open code (`budget_service.py:73-83`) was designed to keep serving. Secondary site confirmed at `events/dispatch.py:18,30,33`: unguarded `cache.get`/`cache.set` in product gating push handlers onto the retry/dead-letter path during a Redis outage. **What's wrong:** A best-effort, non-critical write (buffering "last used" timestamps) is on the hard path of every authenticated request, so it converts a cache outage into a total API outage — the opposite of the intended graceful degradation. **Why we should care:** It silently negates a resilience feature we already built and tested; during a Redis blip every customer integration would see 500s at the door. **Options:** Best practice is to wrap both best-effort cache sites in try/except (the timestamp is reconstructable; the gating cache miss should fall back to a DB read), then chaos-test `record_usage` with Redis stopped to prove the documented degradation end-to-end. **Codebase effect:** Additive and surgical — a try/except at two or three call sites; no structural change, and it activates behavior the rest of the system already expects.

**All failure alerting terminates in unwatched logs — major.** **Verified:** Confirmed; this overlaps §11's blocker. The dead-letter, drift, and repair-spike conditions are emitted (e.g. `wallet.drawdown_repair_spike` at `wallets/tasks.py:171`, `outbox.dead_letter` at `events/tasks.py:31`) but, with no aggregator or pager wired, they land in stdout nobody reads. **What's wrong:** The reconcile lattice's carefully tuned grace and lookback windows (6h, 4 days) only bound how long repair takes *given that a human keeps the system running and notices a spike*; if no one is paged, a sustained failure or a stopped beat is invisible until customers complain. **Why we should care:** The strongest part of the platform is load-bearing on an assumption — "someone is watching" — that is currently false, which downgrades a genuine self-healing guarantee to a self-healing *capability*. **Options:** Best practice is to wire the existing CRITICAL/ERROR events to a pager (PagerDuty/Opsgenie) and add a daily "the lattice ran" heartbeat; the dotted event taxonomy makes the alert rules almost write themselves. **Codebase effect:** Additive — alert routing and one heartbeat task; resolved jointly with the §11 observability work, no domain change.

**Broker non-durability — moderate.** **Verified:** Confirmed at `docker-compose.yml:15-20`: the `redis:7` service has no volume and no AOF/persistence flags (grep for appendonly/aof/--save/redis.conf returns nothing). So queued Celery tasks live only in memory and vanish on a Redis restart. **What's wrong:** On a broker restart, in-flight queued jobs are simply lost. The outbox-and-sweep design reduces this from "lost forever" to "delayed until the next sweep" for swept paths — but that safety is partial (only outbox-backed effects are re-driven) and, importantly, the durability posture here is *accidental*: it's a dev compose default, not a deliberate production choice. **Why we should care:** Relying on an unstated accident for money-job durability is the kind of thing that works until the one restart that drops a non-swept task. The risk is bounded by the lattice but not eliminated. **Options:** Best practice is to enable AOF (append-only file) plus a persistent volume on any non-dev Redis, and to keep the standing engineering rule that "every async effect must have a reconcile twin," documenting which tasks are sweep-covered so the gaps are known rather than assumed. **Codebase effect:** Additive/infrastructural — a Redis volume and config line, plus documentation; no code change.

**No rolling-deploy machinery for migration constraints — moderate.** **Verified:** Confirmed. The audit credits the team for honest migration notes (the 0022 CONCURRENTLY runbook and the 0006 quiesce requirement), but there is no release-pipeline enforcement point anywhere in the repo that actually honors them — the constraint is documented in a comment, not encoded in tooling. **What's wrong:** A migration that the code itself says needs a worker-drain or a quiet window can be run by a routine deploy that doesn't know to provide one. **Why we should care:** For a live billing system, running a data migration while workers are mutating the same tables risks corruption or downtime during exactly the high-stakes moment of a deploy; "we wrote it in a comment" is not a control. **Options:** Best practice is to encode the migration policy in the release pipeline — automatically use CONCURRENTLY for hot-table indexes and insert a worker-drain step for data migrations — so the documented requirement becomes an enforced gate. **Codebase effect:** Additive at the pipeline/deploy layer (CI steps, deploy scripts); the migrations themselves are already written correctly, so no schema rework — this is structural only in the sense that it introduces a deploy-process stage that does not exist today.
