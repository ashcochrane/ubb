# Wave 4 — Multi-Axis Subscription Orchestration (J2) — Design

**Date:** 2026-06-09 · **Method:** 5-agent research-grounded analysis (map current code + the margin-corruption bug → research Stripe CURRENT billing API + competitor multi-axis orchestration → synthesize → adversarial critique). **Effort: L/XL.**

**Goal:** UBB orchestrates the *whole* J2 model — flat access fee + per-seat + usage — as **one coherent Stripe bill** per business on the tenant's own connected account. UBB owns rating; Stripe owns the document. This is the real J2 feature (the D+ → competitive jump).

## North Star architecture (current-Stripe-aligned)
**One Stripe Subscription per business** carries:
- **access fee** → a **licensed `SubscriptionItem`** (recurring Price, `usage_type=licensed`, qty 1)
- **per-seat** → a **licensed `SubscriptionItem`** (recurring Price, qty = seat count)
- **usage** → a **`subscription=`-pinned pending `InvoiceItem`** carrying UBB's already-computed `billed_cost_micros` (cost+margin).

**Decision 1 (LOCKED): usage = pinned InvoiceItem, NOT a Stripe Meter.** UBB's rate-card engine already prices usage; a Meter would force Stripe to re-price (per-customer Price churn / unit-hacks). The InvoiceItem path pushes the exact final dollar amount and pins it to the subscription's renewal invoice via `subscription=<sub_id>` → all three axes land on **one invoice, one PaymentIntent, deterministically** (today's sweep is *accidental*). This is how Metronome coexists with Stripe and sidesteps every deprecated surface.

**Read the CURRENT API everywhere — never `subscription.plan`.** stripe==15.2.0 (Basil-capable); target API version **2025-03-31.basil**. Iterate `subscription.items.data[]` (each item has `price` + `quantity`); period fields are **item-level** (`items.data[i].current_period_*`) — the top-level `current_period_*` are REMOVED in Basil.

## 1. Models (minimal, YAGNI) — `apps/subscriptions/plans/models.py`
**`TenantBillingPlan`** (the definition, one row per tenant plan; `db_table=ubb_billing_plan`): `tenant` FK; `key` (CharField64, `unique_together(tenant,key)`); `name`; `access_fee_micros` (BigInt, 0 allowed); `per_seat_micros` (BigInt, 0 allowed); `interval` (`month`|`year`); `usage_mode` (`invoice_item` default | `none` for prepaid/wallet); lazy Stripe binding `stripe_access_product_id`/`stripe_access_price_id`/`stripe_seat_product_id`/`stripe_seat_price_id` (blank) + `provisioned_at` (null). Prices live on the plan (immutable + reusable) — one access Price + one seat Price per (tenant, plan), shared by every subscriber (no per-customer Price proliferation).

**`CustomerSubscriptionItem`** (the per-line instance binding — the missing multi-line representation + seat-quantity handle; `db_table=ubb_customer_sub_item`): `tenant`/`customer` FK; `stripe_subscription` FK→`StripeSubscription`; `stripe_subscription_item_id` (unique — the `si_…` handle for `SubscriptionItem.modify`); `axis` (`access`|`seat`); `stripe_price_id`; `unit_amount_micros` (snapshot at attach); `quantity` (access→1, seat→N); `plan` FK (null). One row per Stripe SubscriptionItem we create. Keep `StripeSubscription` as the read-mirror.

## 2. Orchestrator — `apps/subscriptions/orchestration/service.py` (`SubscriptionOrchestrator`)
Every Stripe write goes through `stripe_call()` (retry + error-map) with `stripe_account=connected` + a deterministic idempotency key, gated by `_require_charge_ready(tenant)` (`stripe_connected_account_id` set AND `charges_enabled` — the existing postpaid gate).

- **`ensure_plan_provisioned(plan)`** — lazy, idempotent. Create `Product` + recurring `Price` for access (if `access_fee_micros`) and seat (if `per_seat_micros`), keys `plan-prod-access-{plan.id}` / `plan-price-access-{plan.id}` (+ seat). **MUST-FIX (critique Minor): save each `product_id`/`price_id` to the row IMMEDIATELY after its create** (partial save), and skip an axis whose price id is already set — so a mid-way failure is retry-safe past Stripe's 24h idempotency window (no orphan Products). Set `provisioned_at` only when all required axes exist.
- **`subscribe(customer, plan, seats)`** — `owner = customer.resolve_billing_owner()` (seats never subscribed directly — the sub lives on the **business**, per Stage E). Build `items=[{price: access, quantity:1}] (+ {price: seat, quantity: seats} if per_seat)`. `stripe.Subscription.create(customer=owner.stripe_customer_id, items=..., collection_method="charge_automatically", billing_cycle_anchor=<unix first-of-next-month>, proration_behavior="create_prorations", stripe_account=connected, idempotency_key=f"sub-create-{owner.id}-{plan.id}")`. **MUST-FIX (critique Critical 3): pin `billing_cycle_anchor` to the 1st of next month** so the Stripe renewal aligns with UBB's calendar-month `close_postpaid_usage_periods` (default anchor=now → mixed-period invoices for mid-month subscribers). Persist the `StripeSubscription` mirror + one `CustomerSubscriptionItem` per `sub["items"]["data"]` (NEVER `sub.plan`).
- **`set_seats(business, plan, new_seats)`** — `stripe.SubscriptionItem.modify(id=<si_seat>, quantity=new_seats, proration_behavior="create_prorations", stripe_account=connected, idempotency_key=f"seat-qty-{si_id}-{seat_change_event_id}")`. **Decision 2 (LOCKED): `create_prorations`** (accrue credit+debit to the next monthly invoice — one document, no mid-cycle surprise; never `none` for live changes). **MUST-FIX (critique Important): idempotency key = a monotonic seat-change EVENT id (OutboxEvent/audit id), NOT the target quantity** — else a 24h-expired replay of "set to 12" can stale-overwrite a later "set to 11". `customer.subscription.updated` refreshes quantity authoritatively.
- **Seat→quantity hooks. MUST-FIX (critique Important): explicit hooks at every roster transition** — seat add (`POST /api/v1/platform/customers` create) AND suspend/close/soft_delete — each calls `set_seats(business, plan, seat_count(business))` where `seat_count` = active seats under the business; dispatched `on_commit` in the same transaction as the roster change so a remove can't leave Stripe billing ghost seats. (Pooled businesses only; allocated/non-pooled deferred.)

## 3. Usage↔subscription coherence — `apps/billing/invoicing/services/postpaid_service.py`
Replace the `has_sub` standalone-vs-sweep branch (~L135-148). For each aggregated line: look up the business's active subscription; if present, `stripe.InvoiceItem.create(subscription=sub.stripe_subscription_id, customer=owner.stripe_customer_id, amount=cents, period={start,end}, stripe_account=connected, idempotency_key=f"usage-item-{rec.id}-{i}")` — the `subscription=` pin guarantees the same invoice as access+seat. Else (true no-subscription postpaid) create+finalize a standalone invoice.
- **MUST-FIX (critique Important): status filter = `status__in=["active","trialing","past_due","unpaid"]`** (matches revenue.py; today's `active`-only wrongly standalone-invoices a trialing/past_due/unpaid customer who already has an open Stripe invoice → two collection events).
- **MUST-FIX (critique Important): late-push double-bill guard.** After `InvoiceItem.create`, if the subscription's `latest_invoice` is already `paid`/`void` (the cycle closed before the item swept — the ~1h Stripe close is *inside* the outbox retry horizon), the design's "never double" claim is FALSE. Instead: detect it, route the item to a standalone invoice for that period, and emit a loud alert. Never leave a pending item to silently accumulate into the wrong period.
- **Prepaid (`usage_mode="none"`):** usage settles in real time against the wallet (Stage D drawdown) — NO usage InvoiceItem. The subscription still carries access+seat licensed lines; usage never touches Stripe. Clean split.

## 4. The sync FIX + margin correctness (MUST ship in Wave 4 — the live bug)
`subscription_nominal_for_window` (revenue.py ~L68) does `sub.amount_micros * sub.quantity` over a mirror fed by the deprecated `stripe_sub.plan` — a 2-item sub collapses to one item → revenue understated → false "unprofitable" alerts from day one.
- **Shared helper `apps/subscriptions/stripe/items.py` `_sum_items(stripe_sub)`** → `(amount_micros, seat_qty, interval)`: `amount_micros = Σ licensed item.unit_amount × quantity` (access + seat); metered items contribute 0. Item-level period helpers (`items.data[0].current_period_*`). **MUST-FIX (critique Minor): document the single-interval assumption + warn if items disagree.**
- **`sync.py`**: `expand=["data.items.data.price.product"]` (NOT `data.plan.product`); iterate items via `_sum_items`; write item-level periods; capture `stripe_product_name` from the licensed item's `price.product.name`.
- **`api/webhooks.py`** `handle_subscription_created`/`updated`: replace `stripe_sub.plan.*`/`getattr(sub,"quantity",1)` with `_sum_items` + item-level periods; refresh `CustomerSubscriptionItem.quantity` per item (the seat-change confirm path).
- **revenue.py one-liner**: since `amount_micros` now holds Σ(access + seat×qty), change L68 to `per_interval = sub.amount_micros` (drop `* sub.quantity`). **MUST-FIX (critique Critical 2): the sync write + this revenue read are ONE atomic change** — gate both behind a single deploy unit / feature flag; split = a window of seat-count-multiple (e.g. 10×) overstatement persisted into `CustomerEconomics`.
- **MUST-FIX (critique Critical 1): `MarginService.compute_business` (`economics/services.py`).** It sums subscription revenue *per seat*, but the Wave-4 subscription lives on the **business** → every seat's `subscription_nominal_for_window(seat.id)` returns 0 → business rollup shows `subscription_revenue=0` regardless of the sync fix. Fix: compute the subscription axis ONCE against the business owner (`accrued_subscription_revenue(tenant, business.id, …)` / a new `subscription_nominal_for_owner` that resolves the billing owner), and sum per-seat only for usage/cost. Without this the margin dashboard + unprofitable alerts are wrong for every J2 business.

## 5. Connector hardening (build FIRST)
- **MUST-FIX (critique Critical 4): subscriptions webhook dedup FIRST.** The subscriptions webhook endpoint (`apps/subscriptions/api/…`) has NO `StripeWebhookEvent` dedup + bare 500-on-exception, yet `customer.subscription.updated` is the authoritative seat-quantity confirm path. Add `StripeWebhookEvent` get_or_create dedup + error classification (the `api/v1/webhooks.py` pattern) **before** any orchestration write ships.
- **MUST-FIX (critique Important): bump `requirements.txt` `stripe>=8.0` → `stripe>=15.0,<16.0`** + a startup assertion `stripe.VERSION >= "15.0"` (a fresh install could pull a pre-Basil 8.x and silently run the old shapes).
- **API version pin — handle with care (critique Important).** A global `stripe.api_version="2025-03-31"` mutation affects ALL existing paths (auto-topup, checkout, billing webhooks built/tested on the account-default version). **Before** a global pin: run the FULL suite against `stripe==15.2.0` with the Basil version; if any existing path regresses, pin per-call on the orchestration `stripe_call`s instead of globally. Decide during build based on the suite result.

## 6. Plan + subscribe API + SDK (Decision 3: LOCKED — ship now)
`POST /api/v1/platform/plans` (`key, name, access_fee_micros, per_seat_micros, interval, usage_mode`) → create `TenantBillingPlan`; `POST /api/v1/platform/customers/{external_id}/subscribe` (`plan_key, seats`) → `subscribe(...)`; `POST .../seats` (`seats`) → `set_seats(...)`. SDK: `create_plan(...)`, `subscribe_customer(external_id, plan_key, seats)`, `set_seats(external_id, seats)`, `get_subscription(external_id)`. Provisioning stays lazy on first subscribe.

## 8. Capstone (end-to-end, postpaid pooled business)
Via SDK: tenant (charges_enabled) `create_plan("pro", access $50/mo, seat $8/mo)` → lazy-provisions 2 Products + 2 Prices on the connected account; `subscribe_customer(biz, "pro", seats=10)` → `Subscription.create(items=[access×1, seat×10])`, assert **2 SubscriptionItems** via `sub.items.data[]` + 2 `CustomerSubscriptionItem` rows + mirror `amount_micros == 130_000_000`; `set_seats(biz, 12)` → prorated credit+debit, mirror quantity→12 via webhook; record usage (rate-card→billed) → close pushes ONE `subscription=`-pinned InvoiceItem → **ONE** finalized Stripe invoice (access + seat×12-prorated + usage, one PaymentIntent, no standalone); `MarginService.compute_business` → `subscription_revenue` = $50 + 12×$8 + usage, margin **not understated** (the regression proof the sync fix worked).

## 9. Explicitly Wave 5 (kept out)
`invoice.paid`/`payment_failed`/`voided` full per-axis reconcile of the consolidated invoice; `hosted_invoice_url` + end-customer `/me` usage invoices; `always_invoice` per-tenant override + SubscriptionSchedule downgrades; allocated (non-pooled) business subscriptions; mixed-interval plans; converting existing subscriptions read calls onto `stripe_call`.

## Where the design deliberately stays minimal
No Stripe Meter (UBB rates usage); no per-customer Prices (one Price/axis/plan, reused); access+seat axes only; pooled businesses only; the read-mirror stays blended (only the accrual *nominal* is made correct — which is what margin consumes). Everything else is YAGNI until a tenant demands it.
