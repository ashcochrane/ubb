# Docs Consolidation — Matt Pocock Skills Layout for UBB

- **Status:** ACCEPTED (2026-07-13) — building.
- **Scope:** `ubb/` only (`ubb-platform` + `ubb-sdk` + `ubb-ui`). `localscouta-api` /
  `localscouta-ui` are out of scope; the same pattern can be replicated later.
- **Toolkit:** aligning durable docs to **mattpocock/skills** (github.com/mattpocock/skills),
  replacing the `superpowers` design→implementation doc habit.

## Problem

UBB has ~71 dated plan docs plus reviews, generated by the `superpowers` workflow. The knowledge
is high quality but: (1) **nothing is auto-loaded** — no `CLAUDE.md`/`AGENTS.md` anywhere; (2)
everything sits in one undifferentiated pile with no glossary and no conventions layer; (3)
**nothing closes the loop** — a superpowers cycle emits `-design.md` + `-implementation.md` and
moves on, so the pile only grows.

## Model: three layers by relationship to time

| Layer | What | Edited? | Loads |
|---|---|---|---|
| **0 · Always-on** | `CLAUDE.md` — orientation, golden rules, pointers | rarely, in place | auto, every session |
| **1 · Living docs** | `CONTEXT-MAP.md` + per-product `CONTEXT.md`, `docs/conventions/` — *current truth* | continuously, in place | on demand |
| **2 · History** | `docs/adr/`, `docs/architecture/`, `docs/plans/*`, `docs/reviews/*` — *decisions in time* | frozen | archaeology |

`ADR-001` is the aspirational form: dated, yet stays true because
`apps/platform/tests/test_product_boundaries.py` enforces it in CI. **Pin durable claims to an
executable check wherever possible** — prose-only rules re-erode (the ADR documents 14 boundary
violations creeping back in ~100 commits).

## The Matt Pocock file set (the standing `.md` files)

Matt's toolkit is **issue-tracker-centric**: new specs/plans become GitHub issues (`to-spec` →
`to-tickets` → `wayfinder`) that close when done, not `docs/plans/*.md`. So the standing durable
files are a small fixed set:

| File | Layer | Purpose | Produced by |
|---|---|---|---|
| `CLAUDE.md` (root) w/ `## Agent skills` | 0 | orientation + pointers to the config files | this effort (replicating setup skill) |
| `docs/agents/issue-tracker.md` | config | GitHub via `gh`; how skills create/read issues | setup skill seed |
| `docs/agents/domain.md` | config | domain-doc **consumer rules** — the discovery wiring | setup skill seed |
| `CONTEXT-MAP.md` (root) | 1 | the 5 product contexts + how they relate | this effort, then `/domain-modeling` |
| `ubb-platform/apps/<product>/CONTEXT.md` ×5 | 1 | per-context glossary (glossary-only) | `/domain-modeling`, lazily |
| `docs/adr/NNNN-slug.md` | 2 | new tiny sequential decision records | `/domain-modeling` |
| `docs/conventions/{coding,testing,django}.md` | 1 | coding standards — **Matt has no home for these; our addition** | this effort |

`docs/agents/triage-labels.md` is **omitted** — only written when the `triage` skill is installed
(it isn't).

## Multi-context decision (accepted)

UBB is a textbook multiple-bounded-context codebase — metering, billing, subscriptions, referrals
on the platform kernel — and those boundaries are already formalised in `ADR-001`. So: **multi-
context** layout (`CONTEXT-MAP.md` → per-product `CONTEXT.md`). Note the setup skill only auto-
detects *JS* monorepos, so this is a deliberate choice, not an inferred one, and the per-context
files live at the non-standard `ubb-platform/apps/<product>/CONTEXT.md` — fine, because
`CONTEXT-MAP.md`'s job is to record where each context actually lives; skills follow the map.

## Discovery wiring — why the setup files are load-bearing

`CONTEXT-MAP.md` is only discoverable to the **whole** toolkit through:

```
CLAUDE.md  (auto-loaded; ## Agent skills → ### Domain docs)
   └─► docs/agents/domain.md   ("multi-context: read CONTEXT-MAP.md, then per-context CONTEXT.md")
          └─► CONTEXT-MAP.md  ─►  ubb-platform/apps/<product>/CONTEXT.md
```

- `domain-modeling` / `grill-with-docs` hardcode the `CONTEXT-MAP.md` indirection — they find it
  regardless.
- Everyone else (`tdd`, `diagnosing-bugs`, `improve-codebase-architecture`, `codebase-design`,
  `triage`, `code-review`, `to-spec`…) learns it only from `docs/agents/domain.md` + the
  `CLAUDE.md` block. Without those, they look for a root `CONTEXT.md`, don't find one, and
  "proceed silently" — glossaries sit unread. **Hence the setup files are built first.**

## The ratchet (long-term win)

Matt's workflow *builds in* the loop-closing that superpowers lacked: `/domain-modeling` updates
`CONTEXT.md` the moment a term crystallises; hard decisions become ADRs; plans become closable
issues. `CLAUDE.md` states the rule so every agent applies it: **when a plan/issue lands, fold its
lasting outcomes into the living docs; prefer backing hard rules with a test.**

## Build order (you = user, me = agent)

1. **[me] Setup wiring + scaffold** — `docs/agents/issue-tracker.md`, `docs/agents/domain.md`,
   root `CLAUDE.md` (+ `## Agent skills`), `CONTEXT-MAP.md`. *(this commit)*
2. **[me] Seed the 5 per-product `CONTEXT.md` glossaries** from `ADR-001` + `positioning.md` +
   code — first drafts for you to sharpen.
3. **[me] `docs/conventions/{coding,testing,django}.md`** from what the code already does.
4. **[you] Install the plugin** to *use* the downstream skills going forward:
   `npx skills add mattpocock/skills` (+ `--skill=triage` etc. as wanted). Optionally run
   `/setup-matt-pocock-skills` to reconcile the config files with the official templates.
5. **[me, discrete + verified] Migrate `ADR-001` → `docs/adr/0001-product-boundaries.md`** — this
   also edits `test_product_boundaries.py:31` (`ADR = "docs/architecture/…"`, used in CI failure
   messages) and must be run against the test. Deferred out of step 1 for exactly this reason.
6. **[ongoing] `STATUS:` headers** on the most-referenced `docs/plans/*` (`landed → see <x>` /
   `historical`).

## Non-Goals

- Per-app `CLAUDE.md` files (the `CONTEXT.md` files cover per-product context).
- Rewriting/deleting existing plans/reviews — frozen history.
- `localscouta-*` repos.
- Automated ratchet enforcement (hook/CI) — start with the convention.

## Success criteria

1. A fresh agent session in `ubb/` auto-loads `CLAUDE.md`, finds the golden rules, the test
   commands, and (via `docs/agents/domain.md`) the multi-context glossary — without reading a plan.
2. `CONTEXT-MAP.md` + per-product `CONTEXT.md` reflect real, current vocabulary; each claim
   verified against code or a test.
3. Historical plans are distinguishable at a glance; new work lands as issues, not plan markdown.
